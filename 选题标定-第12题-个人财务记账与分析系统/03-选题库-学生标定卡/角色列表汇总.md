## §4 备注

### 4.1 第 38 题角色列表的中英文选择

第 38 题原 JWT 角色行同时给了英文角色名 + 中文括注翻译(`owner(业主)/ staff(物业人员)/ admin(管理员)`)。本汇总按教师演示统一惯例(参考 `memory/project_审计完成基线.md` + plan §2.D2)使用**中文形式**:`业主 / 物业人员 / 管理员 / 物业主管`。教师如需用英文形式,可改为 `owner / staff / admin / property_supervisor`(或类似 P1 物业主管对应英文)。

### 4.2 case C 变体的全量化修正(2026-05-14 Phase 1 漏审补做)

第 29 / 33 / 41 题原 JWT 角色行写为 `单一用户角色(管理员/admin 通过 user.role 字段区分)`。这是 **case C 变体**(P0 单角色 + 括注暗示 P1 有管理员/admin),实质跟 Phase 1 修复的第 01/02/03/04 题(`单一用户角色（P1 起加管理员）` 等)相同 — 都应按"全量化原则"改 case D 形式。Phase 1 的 grep `单一用户角色（P1 起` 没匹配到这 3 题的字面变体("管理员通过 user.role 字段区分" / "admin 通过 user.role 区分" / "管理员通过 user.role 字段"),属漏审。

**2026-05-14 已补做**(Phase 1 修法 #1 扩展):
- 第 29 题 → `用户 / 管理员（user.role 字段区分 · P0 阶段以用户单角色起步）`
- 第 33 题 → `读者 / 管理员（user.role 字段区分 · P0 阶段以读者单角色起步）`(图书馆语境用"读者")
- 第 41 题 → `用户 / 管理员（user.role 字段区分 · P0 阶段以用户单角色起步）`
- 母版 3 段同步(行 1190/1392/1543)

**真正可保留 `单一用户角色` 填值的判定**:括注必须是"纯字段实现提示"(如 `owner_id 字段体现归属` / `作者` 同义词)或"业务层内角色"(如第 11 `行程内角色 由 carpool_member.role 字段区分` / 第 45 `项目内角色 由 project_member.role 区分` · **不是 JWT user 表的角色**)。若括注提到的是 user.role 字段区分的另一类用户(如"管理员/admin"),则应该改 case D 全量化。

### 4.3 第 06 题"班长/学委"角色名解读

第 06 题 `班长/学委 / 普通成员` 中 `班长/学委`(无空格的斜杠)是**同一角色的两种叫法合并**(对齐班级管理实务 · 班长和学委职责重叠),非两个独立角色。`{{角色列表}}` 照抄原写法即可,不拆开为 3 角色。

### 4.4 全量化原则的 P0+P1 处理

第 05 / 09 / 38 题 JWT 角色行含"P1 起再加/加/拆出 X"字样,本汇总按 **§7.2 例 1 范例**(`学生 / 维修员 / 管理员(P1 起再加宿管员)` → `学生 / 维修员 / 管理员 / 宿管员`)把 X 并入全集。这跟"全量设计 + 阶段化实现"原则一致 — CLAUDE.md `{{角色列表}}` 一次填全 P0+P1 所有角色,Phase 4 按 P0/P1/P2 编号选实现,占位符永不回改。

### 4.5 JWT 层全局身份 vs 业务层动态身份(第 11/45 题特别说明)

**核心区分**(决定一个角色该不该进 `{{角色列表}}`):

| 维度 | JWT 层全局身份 | 业务层动态身份 |
|---|---|---|
| 字段所在表 | `user` 表的 `role` 字段 | 业务关系表的 `role` 字段(如 `project_member.role` / `carpool_member.role`) |
| 含义 | 用户登录后的**全局**身份标签(普通用户/管理员/admin)| 用户在**某个业务实体**内的身份(项目内/行程内/团队内) |
| 写进 JWT payload? | ✅ 写入 · 解析 JWT 即知 · 全局一份 | ❌ 不写入 · JWT 颁发时未必知道用户要操作哪个业务实体 |
| 鉴权方式 | 接口拦截 `@PreAuthorize("hasRole('ADMIN')")` / `Authentication.getAuthorities()` | Service 层动态查询(如 `projectMemberMapper.selectByUserAndProject(uid, pid)` 拿 role 再判定) |
| 一个用户可有多种身份? | 全局只有 1 个 role | 可同时存在多个(A 项目 owner + B 项目 member) |
| 进 `{{角色列表}}`? | ✅ 全量化进 | ❌ 不进 · 业务层实现 |

**判定规则**:
- 看 JWT 角色行括注里提到的字段是 `user.role` 还是 `XXX_member.role` / `XXX_relation.role`
  - `user.role` → 全局身份 · 全量化进 `{{角色列表}}`(case D 形式 · 见 §4.2)
  - 业务关系表.role → 业务层动态身份 · 保持 `{{角色列表}}` 为 `单一用户角色` · Phase 4 业务层实现

**典型 case**:
- **第 11 题** `单一用户角色（行程内角色:发起人 / 乘客 由 carpool_member.role 字段区分）` → `carpool_member.role` 是关系表字段 · `{{角色列表}}` 填 `单一用户角色`
- **第 45 题** `单一用户角色（项目内角色:owner/member 由 project_member.role 区分）` → `project_member.role` 是关系表字段 · `{{角色列表}}` 填 `单一用户角色`

**学生 Phase 4 实现提醒**(第 11/45 题专属):
虽然 `{{角色列表}}` = `单一用户角色`,**但 Phase 4 写 Service 层时必须实现业务层权限判定**:
- Service 方法在执行"删除项目 / 修改项目 / 邀请成员"等涉及项目权限的操作前,**先查 `project_member.role`** 判定该用户在该项目内是 owner 还是 member,只有 owner 能做敏感操作
- 同理第 11 题的"取消行程 / 修改起止点"应判 `carpool_member.role` 是发起人才能做
- AI 在 Phase 1 SRS 生成时从 §三 P0 功能清单的"前置条件 / 谁能做什么"业务描述中,会推断出业务层权限要求并写入 PRD §3 各功能的"前置条件"列;Phase 4 feature-coder 读 PRD §3 时按此实现

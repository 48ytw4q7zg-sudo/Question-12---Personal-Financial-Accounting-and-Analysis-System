一、第1章 Spring 框架（3题）
-------------------

**1.** 同一个接口有多个实现类（比如 UserService 有两个实现类），用 @Autowired 注入会出什么问题？口头说出至少两种解决方案。

参考答案：会抛 NoUniqueBeanDefinitionException，因为找到多个候选 Bean。  
解决方案：① @Autowired 配 @Qualifier("beanName") 指定具体 Bean；② 改用 @Resource(name="beanName") 按名称注入；③ 在某个实现类上加 @Primary 设为主选；④ 字段名直接写成目标 Bean 的名字（@Autowired 类型不唯一时会按名称匹配）。

**2.** 口头解释 AOP 中"切面、切入点、连接点、通知"这四个核心概念的含义，并举一个日志切面的例子说明它们怎么配合。

参考答案：

·       **连接点（****JoinPoint****）**：程序执行中的一个点，比如方法调用；

·       **切入点（****Pointcut****）**：一组连接点的匹配规则，比如「Service 包下所有方法」；

·       **通知（****Advice****）**：在切入点要执行的动作（前置/后置/环绕等）；

·       **切面（****Aspect****）**：切入点 + 通知的封装类。

日志切面例子：定义一个切面类，切入点匹配 com.xx.service.* 的所有方法，用环绕通知在方法执行前打印「方法名+参数」、执行后打印「返回值+耗时」。

**3.** @Transactional 加在 private 方法上能生效吗？同一个类里 A 方法直接调用 B 方法（B 上有 @Transactional），事务能生效吗？分别说明原因。

参考答案：

·       加在 private 方法上**不生效**。Spring AOP 基于代理（JDK 动态代理或 CGLIB），private 方法无法被代理覆盖。

·       同类内部 this.B() 调用**也不生效**，因为没经过代理对象。

·       解决方法：把 B 方法拆到另一个类，或通过 AopContext.currentProxy() 拿到代理对象再调。

二、第2章 Spring MVC（3题）
--------------------

**4.** 口头说明 @RequestParam、@PathVariable、@RequestBody 三种参数接收方式分别用于什么场景，前端发请求的形式分别是什么样。

参考答案：

·       **@RequestParam**：接收 URL 查询参数（?key=value）或表单参数。前端形如 /users?id=1 或 form-data 表单。

·       **@PathVariable**：接收 URL 路径中的变量。前端形如 /users/1，对应注解 @GetMapping("/users/{id}")。

·       **@RequestBody**：接收请求体里的 JSON。前端 POST/PUT 请求 + Content-Type: application/json + body 写 JSON。

**5.** 用 RESTful 风格设计一个用户增删改查接口，口头说出 5 个接口的「HTTP 方法 + URL 路径」。

参考答案：

·       列表查询：GET /users

·       详情查询：GET /users/{id}

·       新增：POST /users

·       修改：PUT /users/{id}

·       删除：DELETE /users/{id}

关键点：URL 用名词复数；用 HTTP 方法语义化；不写成 /getUserById 这种动词形式。

**6.** 口头描述 Spring MVC 处理一个请求的完整流程（DispatcherServlet 的工作过程，至少 5 步）。

参考答案：① 请求到达 DispatcherServlet；② HandlerMapping 根据 URL 找到对应的 Handler（Controller 方法）；③ HandlerAdapter 调用 Handler，执行业务方法；④ Handler 返回 ModelAndView 或数据；⑤ ViewResolver 解析视图（或 @ResponseBody 直接序列化为 JSON）；⑥ 渲染响应返回客户端。

三、第3章 MyBatis（3题）
-----------------

**7.** resultType 和 resultMap 有什么区别？什么场景下必须用 resultMap？

参考答案：

·       **resultType**：简单映射，要求查询字段名与 JavaBean 属性名完全一致（或下划线转驼峰开关开启）；

·       **resultMap**：手动映射，可以自定义"字段名 → 属性名"对应关系，处理复杂场景。

必须用 resultMap 的场景：① 字段名和属性名不一致（如 user_name ↔ userName 且未开自动驼峰转换）；② 一对一/一对多关联查询（用 <association>、<collection>）；③ 主键字段需特殊处理。

**8.** 口头描述：要根据「姓名、年龄、状态」三个可选条件查询用户列表（任一条件可不传），动态 SQL 的整体思路是什么？会用到哪些标签？

参考答案：  
整体思路：在 SELECT *FROM user 之后用 <where> 标签包住三段 <if> 判断；

·       每个 <if> 用 test 表达式判空（如 name !=null and name != ''），成立才拼接对应的 AND ... 条件；

·       用 <where> 而不是直接写 WHERE，是因为它能自动去掉首个多余的 AND，且当所有 <if> 都不成立时不输出 WHERE 关键字。  
涉及标签：<if>、<where>。

**9.** <foreach> 标签实现 IN(1,2,3) 批量查询时，要配置哪几个属性？分别什么含义？

参考答案：

·       collection：要遍历的集合参数名；

·       item：每次遍历的当前元素变量名（用于 #{item} 引用）；

·       open：遍历前拼接的字符（如 (）；

·       close：遍历后拼接的字符（如 )）；

·       separator：元素之间的分隔符（如 ,）；

·       index（可选）：当前下标。

四、第5+6+7章 Spring Boot（3题）
-------------------------

**10.** @Value 与 @ConfigurationProperties 都能读配置文件，主要区别是什么？什么场景该用哪个？

参考答案：

·       **@Value("${key}")**：注入单个属性值，简洁但配置多了重复；

·       **@ConfigurationProperties(prefix="xxx")**：批量绑定一组前缀相同的配置到 JavaBean 字段，结构化、可复用、支持松散绑定（如 user-name 自动映射到 userName）。

选用：

·       单个零散属性 → @Value；

·       一组相关配置（如数据源、自定义业务配置）→ @ConfigurationProperties。

**11.** Spring Boot 的"约定优于配置"思想体现在哪些方面？至少说三点。

参考答案：① **自动配置**：根据 classpath 里引入的 Starter 自动装配 Bean（如引入 web starter 就自动配好 Tomcat、SpringMVC）；② **默认配置**：约定的默认端口 8080、约定的配置文件名 application.properties/yml、约定的资源目录 static/templates；③ **Starter** **依赖**：一个依赖打包一组相关库，省掉手动管理；④ **内嵌服务器**：直接 java-jar 启动，不用部署到外部 Tomcat。

**12.** 全局异常处理是怎么实现的？口头说说要用到哪些注解、放在哪个类型的类上、典型会处理哪些异常。

参考答案：

·       在一个普通类上加 @RestControllerAdvice（或 @ControllerAdvice + @ResponseBody），让它能拦截所有 Controller 抛出的异常；

·       类里写多个方法，每个方法上加 @ExceptionHandler(异常类型.class) 指定处理哪种异常；

·       方法返回统一的 Result 对象。

典型处理的异常：① 参数校验异常 MethodArgumentNotValidException（@Valid 失败时）；② 自定义业务异常；③ 兜底的 Exception.class，避免堆栈信息泄露给前端。

五、第8章 数据访问（2题）
--------------

**13.** Redis 在项目里典型有哪些用途？至少说三类。

参考答案：① **缓存**：热点数据（用户信息、商品列表）缓存，减少数据库压力；② **会话****/Token** **存储**：分布式 Session、JWT 黑名单；③ **排行榜**：用 ZSet 按分数排序；④ **计数器**：用 INCR 做点赞数、访问量；⑤ **分布式锁**：用 SETNX 实现；⑥ **简单消息队列**：用 List 的 LPUSH/RPOP。

**14.** @Cacheable、@CachePut、@CacheEvict 三个缓存注解分别在什么场景使用？

参考答案：

·       **@Cacheable**：放在查询方法上，先查缓存，命中就直接返回；没命中才执行方法，并把结果存入缓存。

·       **@CachePut**：放在更新方法上，每次都执行方法，并把结果**写入****/****更新**缓存。

·       **@CacheEvict**：放在删除方法上，执行方法后**清除**缓存（可指定 key，也可 allEntries=true 清空全部）。

六、第12章 Vue 基础（5题）
-----------------

**15.** Vue 组件的生命周期完整流程是怎样的？至少说出 4 个常用钩子的名字和作用。

参考答案：  
完整流程：创建前 beforeCreate → 创建后 created → 挂载前 beforeMount → 挂载后 mounted → 更新前 beforeUpdate → 更新后 updated → 卸载前 beforeUnmount → 卸载后 unmounted。

常用钩子：

·       mounted：DOM 渲染完成，常用于发起 AJAX 请求；

·       created：数据初始化已完成（但 DOM 还没渲染）；

·       updated：数据更新且 DOM 重新渲染后；

·       unmounted：组件销毁，常用于清理定时器、移除事件监听。

Composition API 中对应叫 onMounted、onUpdated、onUnmounted。

**16.** computed 和 watch 都能响应数据变化，使用场景上有什么区别？

参考答案：

·       **computed**：计算并返回一个**派生值**，依赖其他响应式数据；适合「一个值由多个数据计算得到」（如购物车总价）；有缓存。

·       **watch**：监听某个数据变化时**执行副作用**（如发请求、写日志、操作 DOM）；适合「值变化时要做某件事」；无返回值。

简单记：**需要新值用** **computed****，需要触发动作用** **watch**。

**17.** 父子组件通信中，父传子用什么？子传父用什么？口头描述完整流程。

参考答案：

·       **父传子**：父组件在使用子组件标签时通过属性绑定传值（如 :msg="parentMsg"），子组件用 defineProps 声明接收。

·       **子传父**：子组件用 defineEmits 声明可触发的事件，需要时调用 emit('事件名', 参数)；父组件在子组件标签上用 @事件名="处理函数" 监听。

流程示例：父→子传一个商品对象，子组件展示并提供"加入购物车"按钮，按钮点击 emit 一个 add 事件带上商品 id，父组件收到后更新购物车。

**18.** ref 和 reactive 都能创建响应式数据，怎么选？

参考答案：

·       **ref**：包装基本类型（number/string/boolean），也可包装对象；JS 中访问/修改要 .value，模板中自动解包不用写 .value。

·       **reactive**：包装对象/数组，返回 Proxy 代理；不需要 .value，但解构后会丢失响应性（要用 toRefs）。

实践推荐：基本类型用 ref；复杂对象/数组也常用 ref（统一加 .value 心智一致），或者用 reactive 但避免直接解构。

**19.** 什么是回调地狱？Promise 怎么解决它？async/await 又比 Promise 好在哪？

参考答案：

·       **回调地狱**：多个异步操作嵌套（一个回调里再写一个回调），代码向右无限缩进，难读难维护，错误处理也麻烦。

·       **Promise**：用 .then().then().then() 链式调用，把嵌套展平为线性，错误用 .catch 统一捕获。

·       **async/await**：让异步代码看起来像同步代码（constdata = await api.get()），无 .then 链；用 try/catch 直接捕获异常，调试时也能像同步代码一样看堆栈。

七、第13章 Vue 进阶（4题）
-----------------

**20.** Vue Router 中编程式导航和声明式导航各举一个例子，并说什么场景用哪种。

参考答案：

·       **声明式**：<router-link to="/login">登录</router-link>，模板中固定的跳转链接（侧边栏菜单、导航栏）。

·       **编程式**：router.push('/login') 或 router.push({name: 'Detail', query: { id: 1 } })，用于 JS 逻辑里跳转（登录成功后、按钮点击带条件判断后）。

**21.** 口头说说怎么用 beforeEach 实现"未登录用户访问受保护页面跳转到登录页"的整体思路。

参考答案：① 在路由配置中给受保护页面加上 meta: {requireAuth: true }；② 在 router.beforeEach((to,from, next) => {...}) 里读取 localStorage 里的 token；③ 判断「目标路由需要鉴权 且 token 不存在」时调用 next('/login') 跳到登录页；④ 否则调用 next() 放行。

⚠️ 关键：next() 必须调用，否则导航卡住；不要在守卫里写死循环跳转。

**22.** defineProps 和 defineEmits 在 <scriptsetup> 中是怎么工作的？需要 import 吗？为什么？

参考答案：**不需要** **import**。它们是 Vue 提供的**编译器宏**（compiler macro），由 Vue 编译时识别并直接转换成对应代码。

·       defineProps 用来声明接收的 props，可以传数组或对象（带类型校验）；

·       defineEmits 用来声明可以触发的事件，调用它返回一个 emit 函数。  
这两个宏只能在 <scriptsetup> 里用，且不能从其他文件 import。

**23.** Element Plus 全局引入和按需引入有什么区别？哪种推荐？

参考答案：

·       **全局引入**：一次性 app.use(ElementPlus)，所有组件都能用，简单但打包体积大；

·       **按需引入**：通过 unplugin-vue-components 插件自动按需导入用到的组件，打包体积小，性能好。

实际项目推荐**按需引入**；课程练习/Demo 可用**全局引入**简化配置。

**24.** 口头描述：要查询「年龄大于 18、姓名以"张"开头、按创建时间倒序」的用户列表，用 LambdaQueryWrapper 怎么实现？涉及哪些方法？

参考答案：  
思路：newLambdaQueryWrapper<User>() 创建包装器，链式调用：

·       .gt(User::getAge, 18) → 大于 18；

·       .likeRight(User::getName, "张") → 姓名以"张"开头（likeRight 表示右模糊，等价 LIKE '张%'）；

·       .orderByDesc(User::getCreateTime) → 按创建时间倒序。

最后调 userMapper.selectList(wrapper) 拿结果。

关键优势：用 User::getXxx 方法引用，**编译期类型安全**，字段改名编译不过。

**25.** MyBatis-Plus 的分页插件怎么生效？口头说出配置步骤和调用步骤。

参考答案：  
配置步骤：① 写一个配置类，注入 MybatisPlusInterceptor Bean；② 给它添加内层拦截器 PaginationInnerInterceptor(DbType.MYSQL)（指定数据库方言）。

调用步骤：① 创建 Page<User> 对象，构造参数为「当前页号、每页大小」，比如 newPage<>(1, 10)；② 调用 userMapper.selectPage(page,queryWrapper)；③ 通过 page.getRecords() 拿数据列表，page.getTotal() 拿总记录数。

常见坑：忘了配拦截器，分页参数会被忽略，相当于查全量。

**26.** Git 的工作区、暂存区、本地仓库、远程仓库分别是什么？常用命令在它们之间怎么流转？

参考答案：

·       **工作区**：你正在改的代码文件夹；

·       **暂存区**：git add 后进入的临时区域，准备提交的文件；

·       **本地仓库**：gitcommit 后形成的本地版本历史；

·       **远程仓库**：GitHub/Gitee 上的服务器仓库。

流转路径：  
工作区 ──git add──→ 暂存区 ──git commit──→ 本地仓库 ──git push──→ 远程仓库；  
反向拉取用 git pull（远程 → 本地仓库 → 工作区）。

九、第16章 综合项目（4题）
---------------

**27.** 浏览器的同源策略限制了什么？解决跨域的常见方式有哪些？项目里 Spring Boot 是怎么配 CORS 的？

参考答案：

·       **同源策略**：协议、域名、端口三者必须完全一致才算同源，否则浏览器限制 AJAX 请求、Cookie、DOM 访问。

·       **跨域常见方案**：  
① **CORS**（推荐）：后端在响应头加 Access-Control-Allow-Origin 等告诉浏览器允许跨域；  
② **JSONP**：只支持 GET，已淘汰；③ **Nginx** **反向代理**：把前端域和后端域统一到同一域。

·       **Spring Boot** **配** **CORS**：实现 WebMvcConfigurer 接口，重写 addCorsMappings，配置允许的源、方法、请求头；或注册一个 CorsFilter Bean。

**28.** 口头描述 JWT 的认证完整流程。为什么用 JWT 而不用传统 Session？

参考答案：  
流程：① 用户提交账号密码到后端登录接口；② 后端校验通过后，用密钥对用户信息（id、角色等）签名生成 JWT，返回给前端；③ 前端把 Token 存到 localStorage，后续请求在 Authorization 头里携带（Bearer xxx）；④ 后端拦截器拦下请求，解析 Token，校验签名 + 过期时间，提取用户信息放入上下文供 Controller 使用；⑤ 校验失败返回 401 未授权。

JWT vs Session 优势：① **无状态**：服务器不保存会话，天然支持分布式/集群；② **跨域友好**：前后端分离、移动端、小程序都通用；③ **可携带载荷**：用户 id、角色直接编码到 Token 里，省一次查库。

JWT 缺点：颁发后无法主动失效（需配合黑名单），载荷不能放敏感信息（只是 base64 编码不是加密）。

**29.** Axios 怎么封装一个统一的请求实例？请求拦截器和响应拦截器分别在里面做什么事？

参考答案：  
整体思路：① 用 axios.create({baseURL, timeout }) 创建实例，统一配置后端地址和超时；② 在实例上注册请求拦截器和响应拦截器；③ 把实例 export 出去，业务代码统一引用。

**请求拦截器**做的事：

·       从 localStorage 读取 token，自动加到请求头 Authorization；

·       统一加 loading 提示；

·       加请求时间戳防缓存。

**响应拦截器**做的事：

·       统一处理后端返回的 Result 结构（如 code !=200 弹错误提示）；

·       Token 失效（401）自动跳转登录页 + 清除本地 token；

·       网络异常统一兜底提示；

·       关闭 loading。

**30.** 设计一个最简电商项目的核心数据库表，至少说出 4 张表的表名和关键字段，并说说表之间的关联关系。

参考答案：

| 表名                  | 关键字段                                                |
| ------------------- | --------------------------------------------------- |
| user（用户表）           | id、username、password（加密存储）、phone、create_time        |
| goods_type（商品类型表）   | id、type_name、parent_id（自关联，支持多级分类）                  |
| goods（商品表）          | id、name、price、stock、type_id（外键 → goods_type）        |
| cart（购物车表）          | id、user_id（外键 → user）、goods_id（外键 → goods）、quantity |
| order（订单表）          | id、user_id、total_amount、status、create_time          |
| order_detail（订单明细表） | id、order_id（外键 → order）、goods_id、price、quantity     |

关联关系：

·       一个用户可以有多个购物车记录、多个订单（一对多）；

·       一个订单包含多条订单明细（一对多）；

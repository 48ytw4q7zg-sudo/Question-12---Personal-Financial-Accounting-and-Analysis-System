## S000 设计模式使用清单

| 设计模式 | 位置 | 为什么用 / 解决了什么 |
|---|---|:---|
| **模板方法模式** | `GlobalExceptionHandler` extends `ResponseEntityExceptionHandler` | 覆盖 Spring 父类的 12 种异常处理方法，统一为 Result<T> 格式，避免每个 Controller 写 try-catch |
| **策略模式** | `ErrorCode` enum (28 个错误码) | 集中管理 6 个模块的错误码与消息，Service 层直接引用枚举常量而非硬编码字符串，码值统一可维护 |
| **单例模式** | `LoginRateLimiter` (static ConcurrentHashMap + static methods) | 全局唯一限流器，所有登录请求共享同一 RateLimiter 实例 |
| **工厂方法模式** | `Result.success() / Result.error()` 静态工厂 | 统一创建 Result 对象，隐藏构造细节，前端 axios 拦截器据此解析 |
| **代理模式** | Spring AOP `@Transactional` + `BudgetScheduler` | 声明式事务管理（JDK 动态代理），REQUIRES_NEW 独立事务隔离单用户预警失败 |
| **拦截器链模式** | `LoginInterceptor` -> `AdminInterceptor` (WebMvcConfig 注册) | 责任链处理 HTTP 请求：JWT 认证 -> 管理员鉴权 -> Controller，每个拦截器关注单一职责 |
| **DTO 模式** | `entity/dto/` 26 个 DTO 类 | 解耦 Entity 与 API 响应，防止 @JsonIgnore 字段泄漏、防止循环引用、数据库字段变更不直接暴露给前端 |
| **依赖注入** | `@RequiredArgsConstructor` (全项目) | Lombok + Spring IoC 构造器注入，消除 @Autowired 字段注入的紧耦合，便于单元测试 mock |
| **门面模式** | `EntityValidator.java` (3 个 Service 共享) | 提取 AccountServiceImpl/TransactionServiceImpl/RecurringBillServiceImpl 中重复的 validateAccount/validateCategory，一处修改三处生效 |
| **缓存/懒加载** | `echarts-lazy.js` (单例 Promise) | 首次加载 ECharts 核心后缓存，后续调用零延迟，且失败后清除缓存可重试 |
| **建造者/组合模式** | `LambdaQueryWrapper<User>().eq(...)` | MyBatis-Plus 的链式 API 构建 SQL 条件，类型安全 + IDE 补全 |
| **观察者/调度模式** | `@Scheduled` (BudgetScheduler) | Spring 定时任务，每日凌晨 2:00 自动检查预算预警，解耦定时触发与业务处理 |
| **路由守卫模式** | `router.beforeEach` (Vue Router) | 前置钩子集中鉴权：Token 过期预检 -> 需登录拦截 -> 管理员拦截 -> Chunk 加载失败容错，6 步一个入口 |

### Q2: 前后端怎么通信的？

**Axios + RESTful API + JWT**。

1. Axios 实例 (`frontend/src/api/request.js:15-18`): `baseURL='/api/v1'`, `timeout=10000`
2. 请求拦截器: 自动注入 `Authorization: Bearer <token>` (第 60-62 行)
3. Vite proxy: `/api` -> `localhost:8080` (配置在 `vite.config.js`)
4. 后端统一返回 `Result<T>`: `{ code, message, data }` (定义在 `common/Result.java`)
5. 响应拦截器: code=200->返回 data, code=401->跳/login, 其他->ElMessage 提示 (第 82-125 行)

**完整调用链** (以记一笔为例):
```
TransactionListPage.vue:424 handleSubmit()
  -> api/transaction.js:89 createTransaction(data)
    -> request.js:60 注入 Authorization Header
      -> POST /api/v1/transaction
        -> LoginInterceptor 验证 JWT
          -> TransactionController.create(@Valid)
            -> TransactionServiceImpl.create()
              -> EntityValidator.validateAccount/Category
                -> transactionMapper.insert()
                  <- Result<TransactionDTO>
    <- request.js:87 code=200 -> return data
  <- ElMessage.success('记账成功')
```

### Q3: 怎么处理异常？

**三层全局异常处理**:

1. **Service 层**: `throw new BusinessException(code, msg)` (定义在 `common/BusinessException.java`)
2. **GlobalExceptionHandler**: `@RestControllerAdvice` 继承 `ResponseEntityExceptionHandler`，覆盖 12 种异常处理，统一转为 `Result.error(code, msg)` (定义在 `common/GlobalExceptionHandler.java`)
3. **前端**: axios 响应拦截器统一处理 `code !== 200`，ElMessage.error 弹窗

**body-code-first 约定**: HTTP 状态码始终 200，业务状态码放在 JSON body 的 code 字段。前端拦截器检查 `res.data.code`，与 LoginInterceptor 保持一致。

### Q4: 数据库怎么设计的？有哪些表？

**7 张表** (按外键依赖顺序):

| 表 | 用途 | 核心约束 |
|---|:---|---|
| `user` | 用户 | username UNIQUE, password BCrypt, role 0/1 |

| `recurring_bill` | 周期账单 | period daily/weekly/monthly/yearly, next_due_date |
| `budget_alert` | 预算预警 | alert_level OVERSPENT/MONTHLY_WARN/DAILY_WARN/NORMAL |

#### 数据库设计原则

1. **金额精度**: 全系统统一 `DECIMAL(12,2)`，禁止 FLOAT/DOUBLE 防精度丢失
2. **时间字段**: 强制 `DATETIME`，Java 侧映射 `LocalDateTime`，禁止 `Date`
3. **外键约束**: 7 张表间 3 组 FK (user->account, user->transaction, account/category->transaction)
4. **软删除**: account 表 `status` 字段实现逻辑删除，保留历史交易可追溯
5. **唯一索引**: budget 表 `(user_id, category_id, month)` 联合唯一索引，防重复设置
6. **时间戳**: 每表强制 `create_time` + `update_time`

### Q6: JWT token 里有什么？为什么能防篡改？

**Payload 包含**: `sub`(userId) + `username` + `role` + `exp`(7天) + `iat`(签发时间)

**防篡改原理**: JWT 使用 **HMAC-SHA256 (HS256)** 签名。`JwtUtils.java:79-89 generateToken()` -> JJWT 库用 256-bit 密钥对 header+payload 做 HMAC 签名。任何人修改 payload 后签名不再匹配 -> 后端 `parseTokenPayload()` 验签失败 -> 返回 null -> LoginInterceptor 返回 401。

**性能优化**: `parseTokenPayload()` (第 98-117 行) **一次 HMAC 验签**同时提取 userId + username + role，消除旧实现 `parseToken()` + `parseRole()` **两次验签**的 50% 冗余密码学运算。

### Q9: 转账怎么实现？怎么防止并发死锁？

**核心流程** (`TransactionServiceImpl.java:302-400 transfer()`):
1. 校验: 金额>0, fromAccount!=toAccount
2. **死锁预防**: 按账户 ID **升序**加悲观锁 (第 329-334 行)
3. 余额校验: `initialBalance + 收入 - 支出 >= amount`
4. 查询"其他"支出分类 -> 生成 UUID transferId
5. `@Transactional` 内: INSERT 转出(支出) + INSERT 转入(收入)
6. 返回 `TransferDTO(transferId, outRecord, inRecord)`

**死锁预防原理**: 线程1转账 A->B、线程2转账 B->A：
- **旧实现** (有死锁): 线程1锁A等B, 线程2锁B等A -> 循环等待 -> 死锁
- **新实现** (零死锁): 都先锁 min(A,B) 再锁 max(A,B) -> 同样顺序 -> 无循环等待
- 这是操作系统课本"破坏死锁必要条件"的经典解法

**转账保护**:
- transfer_id 非空的记录**禁止修改金额** (error 3006)
- transfer_id 非空的记录**禁止单独删除** (error 3007, 保持配对完整性)

### Q12: 转账死锁预防的完整原理是什么？代码怎么实现的？

**问题建模**: A 和 B 是两个账户，线程1从 A 转账到 B (锁A后锁B)，线程2从 B 转账到 A (锁B后锁A)。

**死锁条件分析** (操作系统四大必要条件):
1. 互斥: `SELECT ... FOR UPDATE` 获取行级排他锁 -- 满足
2. 持有并等待: 线程1持有A的锁等待B，线程2持有B的锁等待A -- 满足
3. 不可剥夺: InnoDB 行锁不可被其他事务强制释放 -- 满足
4. 循环等待: A->B 与 B->A 形成闭环 -- 满足

**修复策略**: 破坏第 4 条"循环等待"。无论 from/to 哪个 ID 更小，所有线程都先锁 `min(A,B)` 再锁 `max(A,B)`，全局统一的加锁顺序。

**代码实现** (`TransactionServiceImpl.java:328-335`):
```java
Long firstId = Math.min(fromAccountId, toAccountId);   // 较小ID先锁
Long secondId = Math.max(fromAccountId, toAccountId);  // 较大ID后锁
Account firstAccount = accountMapper.selectByIdForUpdate(firstId);
Account secondAccount = accountMapper.selectByIdForUpdate(secondId);
// 还原 from/to 引用（锁顺序与业务方向解耦）
Account fromAccount = (fromAccountId.equals(firstId)) ? firstAccount : secondAccount;
Account toAccount = (fromAccountId.equals(firstId)) ? firstAccount : secondAccount;
```

**为什么用悲观锁不用乐观锁**: 转账是低频但强一致性场景。乐观锁(version字段)在并发转账时会反复重试，用户体验差且浪费DB资源。悲观锁一次锁定、一次成功。

**为什么不用分布式锁**: 本项目是单实例 SpringBoot + 单 MySQL，无需引入 Redis/Redisson 等额外组件。行级 SELECT FOR UPDATE 已足够。

### Q12.5: 为什么用 MyBatis-Plus 的 LambdaQueryWrapper 而不是写原生 SQL？具体举例说明类型安全和防注入优势

**三层理由**:

**第 1 层: 安全 -- SQL 注入防护天然内置**

LambdaQueryWrapper 底层强制使用 JDBC `PreparedStatement` 参数化查询, 即使用户输入 `' OR '1'='1` 也会被当作字面字符串值, 不会被解释为 SQL 片段。对比:

```java
// 用 LambdaQueryWrapper (安全): 参数化查询, 用户名中的 SQL 关键字被转义
new LambdaQueryWrapper<User>()
    .eq(User::getUsername, request.getUsername())  // 输入 "admin' OR '1'='1" -> 安全, 作为字面值

// 如果写原生 SQL (危险): 字符串拼接, 恶意输入直接注入
String sql = "SELECT * FROM user WHERE username = '" + request.getUsername() + "'";
// 输入 "admin' OR '1'='1" -> SELECT * FROM user WHERE username = 'admin' OR '1'='1'
// 结果: 返回所有用户, 绕过认证
```

**第 2 层: 类型安全 -- 编译期就能发现错误**

```java
// LambdaQueryWrapper (编译期检查): IDE 自动补全, 重构字段名时自动更新引用
new LambdaQueryWrapper<User>()
    .eq(User::getUsername, username)   // getUsername 是 Java 方法引用, 编译器验证存在
    .eq(User::getRole, role);

// 原生 MyBatis XML (运行时才能发现): 字段名写错编译通过, 运行时抛异常
// <if test="userName != null"> AND username = #{userName} </if>
// 属性名 userName 写错 (entity 中是 username) -> 编译通过, 运行时报错
```

具体在本项目中的类型安全收益: `AccountServiceImpl.java` 中有 `.eq(Account::getUserId, userId)` -- 如果某天重构将 `userId` 字段改名, IDE 一键重命名, Lambda 方法引用自动更新, XML 中的 `user_id` 字符串不会自动更新, 必须手动搜索替换, 容易遗漏。

**第 3 层: 开发效率 -- 链式 API 可读性高, 条件组合灵活**

```java
// 实际项目代码 (AccountServiceImpl.java:78-84): 4 个条件一行链式, 比 XML 的 <where><if> 更紧凑
accountMapper.selectList(
    new LambdaQueryWrapper<Account>()
        .eq(Account::getUserId, userId)       // 用户过滤
        .eq(Account::getStatus, 1)            // 仅活跃账户
        .orderByDesc(Account::getCreateTime)  // 最新优先
);
```

同样逻辑如果用 XML 写, 需要 `<select>` + `<where>` + 3 个 `<if>` 标签, 再加 Mapper 接口方法声明, 代码量是 Lambda 的 3-4 倍。

**什么时候用 XML?** 本项目遵循 CLAUDE.md S二.四 的约定: 简单 CRUD 全用 LambdaQueryWrapper, 只在多表 JOIN + GROUP BY + 动态 ORDER BY 这类 Lambda 做不到的场景才写 XML (如 `TransactionMapper.xml` 的 `selectTransactionList`)。这样既有 Lambda 的简洁安全, 又有 XML 的复杂查询表达能力。

#### 代码证据表

| 对比维度 | LambdaQueryWrapper | 原生 SQL (假设) |
|---|:---|:---|
| SQL 注入风险 | 零风险 (PreparedStatement) | 高风险 (拼接, 需手动防护) |
| 字段名错误 | 编译期报错 (方法引用) | 运行时报错 (XML/Ognl 字符串) |
| IDE 支持 | 自动补全 + 重构同步 | 无 (XML 字符串) |
| 代码量 | 1 个 Lambda 链 4-5 行 | Mapper 接口 + XML 共 20+ 行 |
| 本项目出现频率 | ~95% 查询用 Lambda | ~5% (TransactionMapper.xml 复杂查询) |

### Q12.6: Spring AOP 自调用失效问题你是怎么发现和解决的？结合 BudgetScheduler 的具体例子说明

**问题背景**: `BudgetScheduler` 是一个 Spring 定时任务类, 使用 `@Scheduled` 每天凌晨 2:00 自动检查所有用户的预算并生成预警。

**发现过程**: Q-CR 第三轮审计 (Phase 7) 审查 `BudgetScheduler.processAllUsers()` 时, 注意到方法内部直接调用了 `this.processUserBudgetAlerts(userId)`, 而该方法标注了 `@Transactional(propagation = Propagation.REQUIRES_NEW)`。

**问题原理 (AOP 自调用穿透)**:

```
BudgetScheduler (Spring Bean, 被 AOP 代理包裹)
  │
  ├─ processAllUsers()      -- @Transactional (默认 REQUIRED)
  │    │                       事务 T1 开始
  │    │
  │    └─ this.processUserBudgetAlerts(userId)  -- self-invocation!
  │         │                  直接调用 this 对象的方法, 绕过了 Spring AOP 代理
  │         │                  @Transactional(REQUIRES_NEW) 完全不生效!
  │         │                  方法仍然在事务 T1 中执行, 而非新事务
  │         │
  │         └─ 如果 A 用户的预警处理抛异常 -> T1 回滚 -> B/C/D 用户全部丢失
```

Spring AOP 默认使用 JDK 动态代理或 CGLIB 代理。当外部调用 `BudgetScheduler.processAllUsers()` 时, 调用经过代理对象 -> 代理执行事务切面逻辑 -> 再反射调用目标方法。但当目标方法内部 `this.processUserBudgetAlerts()` 时, `this` 指向目标对象本身而非代理, 直接跳过了 AOP 切面。

**修复方案 (提取独立 Service)**:

```java
// 修复前 (有 Bug): BudgetScheduler 内部自调用
@Service
public class BudgetScheduler {
    @Transactional
    public void processAllUsers() {
        for (Long userId : allUserIds) {
            this.processUserBudgetAlerts(userId);  // BUG: AOP 失效
        }
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void processUserBudgetAlerts(Long userId) {
        // 单用户预警处理逻辑
    }
}

// 修复后: 提取 BudgetAlertProcessorServiceImpl, 通过注入的代理调用
@Service
public class BudgetScheduler {
    private final BudgetAlertProcessorServiceImpl alertProcessor;  // 注入 -> Spring 代理

    @Transactional
    public void processAllUsers() {
        for (Long userId : allUserIds) {
            alertProcessor.processUserBudgetAlerts(userId);  // 经过代理, REQUIRES_NEW 生效
        }
    }
}

@Service
public class BudgetAlertProcessorServiceImpl {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void processUserBudgetAlerts(Long userId) {
        // 单用户预警处理逻辑 (独立事务, 失败不影响其他用户)
    }
}
```

**修复效果**: A 用户预警失败 -> A 的 REQUIRES_NEW 事务回滚 -> B/C/D 用户各自在独立事务中正常执行 -> 最多丢失 1 个用户的预警, 而非全部丢失。

**替代方案 (为什么不选用)**:

| 方案 | 优点 | 缺点 | 本项目选择 |
|---|---|---|---|
| 提取独立 Service | 职责清晰, 代码自解释 | 多一个类 | **选用** (3 个类变为 4 个, 职责更清晰) |
| `AopContext.currentProxy()` | 不改类结构 | 需要 `@EnableAspectJAutoProxy(exposeProxy=true)`, 代码中混杂 AOP 基础设施 | 不选 (框架耦合太重) |
| `@Autowired` 自注入 | 不改类结构 | 循环依赖风险, 代码异味 | 不选 |
| 事务改为手动编程 | 完全控制 | `TransactionTemplate` 侵入业务代码 | 不选 (声明式事务的风格一致性) |

### Q12.7: 前端路由懒加载是怎么配置的？Vite 分包策略具体怎么做的？

**路由懒加载配置** (`frontend/src/router/index.js:53-99`):

11 个页面全部使用动态 import 语法:

```javascript
// 静态导入 (不推荐): 首屏加载所有页面 chunk, 即使当前不访问
// import DashboardPage from '../views/DashboardPage.vue'

// 动态导入 (本项目): Vite 自动将每个 import() 拆为独立 JS chunk
const routes = [
  {
    path: '/',
    component: () => import('../views/DashboardPage.vue'),    // 单独 chunk
  },
  {
    path: '/account',
    component: () => import('../views/AccountPage.vue'),       // 单独 chunk
  },
  {
    path: '/transaction',
    component: () => import('../views/TransactionListPage.vue'), // 单独 chunk
  },
  // ... 其余 8 个页面全部使用 () => import(...)
];
```

**工作原理**: Vite/Rolldown 在构建阶段扫描所有 `import()` 调用, 为每个动态导入生成独立的 JS chunk 文件。用户访问 `/login` 时只加载 `LoginPage-[hash].js` (约 15KB), 不加载其他 10 个页面, 首屏体积从 ~300KB 降低到 ~15KB (仅页面代码)。

**Vite 分包策略** (`frontend/vite.config.js:24-47`):

```javascript
build: {
  chunkSizeWarningLimit: 1000,  // 1000KB 警告阈值 (Element Plus 包体较大)
  rollupOptions: {
    output: {
      manualChunks: {
        // 第 1 组: Element Plus UI 库 (最大的单独依赖, ~945KB)
        'vendor-element': ['element-plus'],

        // 第 2 组: Element Plus 图标 (~200KB)
        'vendor-icons': ['@element-plus/icons-vue'],

        // 第 3 组: Vue 生态 (Router + Pinia, ~30KB)
        'vendor-router': ['vue-router', 'pinia'],

        // 第 4 组: HTTP 库 (Axios, ~20KB)
        'vendor-axios': ['axios'],

        // 第 5 组: ECharts 按需拆为 3 个独立 chunk
        'echarts-core': ['echarts/core'],
        'echarts-charts': ['echarts/charts'],
        'echarts-components': ['echarts/components'],
        'echarts-renderers': ['echarts/renderers'],
      }
    }
  }
}
```

**分包的核心收益**:

| 维度 | 不分包 (单一 vendor) | 分包后 (本方案) |
|---|---|:---|
| 首屏加载文件数 | 1 个 ~1.3MB chunk | 4 个 vendor (~1.2MB, 6 连接并行下载) |
| 缓存粒度 | 改一行代码 -> 整个 vendor 重新下载 | 改一行代码 -> 仅当前页面 chunk 失效 |
| ECharts 加载 | 首屏强制加载 (即使 Dashboard 不显示图表) | Dashboard/Analytics 访问时才加载 (~350KB 按需) |
| 浏览器并行连接利用 | 1 个连接加载单文件 | 6 个连接同时下载 4 个 vendor + 2 个 echarts |

**为什么不是动态 import 全自动?** Vite 默认会把 `import()` 拆出来, 但对于 `node_modules` 中的第三方库, 不做 `manualChunks` 的话, Vite 会把所有第三方依赖打进一个巨大的 vendor chunk。如果这个 chunk 超过 1MB, 浏览器下载慢, 改一处就要重新下载 1MB。手动拆分后, 改业务代码时 vendor chunks 哈希不变, 浏览器直接走 304 缓存。

**Chunk 加载容错** (`router/index.js:177-186`): 如果 CDN 故障或网络丢包导致某个 chunk 加载失败:

```javascript
router.onError((error) => {
  // 检测 "Failed to fetch dynamically imported module" 错误
  if (/dynamically imported module/.test(error.message)) {
    ElMessage.error('页面加载失败，请刷新浏览器重试');
  }
});
```

### Q14: 怎么防止 SQL 注入？

**MyBatis-Plus LambdaQueryWrapper** (全项目首选):
```java
// 正确: Lambda 构建类型安全条件, 底层参数化查询
new LambdaQueryWrapper<User>().eq(User::getUsername, username);

// 错误: 字符串拼接 (本项目严禁)
"SELECT * FROM user WHERE username = '" + username + "'";
```

**XML 参数化** (复杂查询例外):
```xml
<!-- 使用 #{keyword} 参数占位, MyBatis 自动参数化 -->
WHERE t.note LIKE CONCAT('%', #{keyword}, '%') ESCAPE '\'
<!-- 使用 #{id} 代替 ${id} (${} 是直接拼接, #{} 是参数化) -->
AND t.account_id IN <foreach>#{id}</foreach>
```

**LIKE 注入防护**: 用户输入 `%` `_` `/` 在 Service 层转义为 `\%` `\_` `\\`，防止被 MySQL LIKE 解释为通配符而非字面字符。

### Q21: ECharts 怎么做按需加载的？

### Q21.6: 项目中用到了哪些设计模式？每个模式解决了什么具体问题？

**在 S000 设计模式使用清单的基础上补充 5 个模式的实际问题场景**:

| 设计模式 | 具体问题 | 不用这个模式的后果 | 代码证据 |
|---|:---|:---|---|
| 模板方法 | 12 种异常类型各自在 Controller 写 try-catch -> 重复代码, 容易遗漏 | Controller 臃肿, 每种异常处理不一致, 改一处需改 N 处 | `GlobalExceptionHandler.java extends ResponseEntityExceptionHandler` |
| 策略模式 | 28 个错误码分散在各 Service 硬编码字符串, 码值冲突、改消息找不到地方 | 错误码管理混乱, 不同模块可能用同一个码值, 改文案需全局搜索 | `ErrorCode.java` enum 28 个常量 |
| 单例模式 | 多个 RateLimiter 实例导致每个实例计数独立, 限流失效 | 攻击者可以分散请求到不同实例的限流窗口, 实际 TPS 远超限制 | `LoginRateLimiter.java` static Map + static methods |
| 工厂方法 | 各处 `new Result(code, msg, data)` 构造器不统一, 字段赋值遗漏 | code/message 字段容易传反, 构造不统一导致前端解析失败 | `Result.success() / Result.error()` 静态工厂 |
| 代理模式-事务 | 转账需要两条 INSERT 原子: 要么全成功要么全失败 | 第一条成功第二条失败 -> 转出钱了没到账 -> 资金丢失 | `@Transactional` on `transfer()`, Spring AOP 代理 |
| 代理模式-时钟 | BudgetScheduler 内部 `this.processUserBudgetAlerts()` 绕过 Spring AOP 代理, REQUIRES_NEW 失效 | 定时任务中 A 用户预警失败导致整个批次回滚, B/C/D 用户的预警全部丢失 | 提取独立 `BudgetAlertProcessorServiceImpl` 注入调用 |
| 拦截器链 | JWT 校验 + 管理员鉴权写在同一个方法里 -> 职责不清, 改一处需测全部 | 新增鉴权维度要动认证代码, 符合性差 | `LoginInterceptor` + `AdminInterceptor` 独立注册 |
| DTO 模式 | Entity 直接返给前端 -> @JsonIgnore 字段(密码)丢失, 数据库字段变更直接暴露 | 密码字段注册时被忽略无法写入; 改数据库字段名前端报错 | `entity/dto/` 26 个 DTO 类 |
| 依赖注入 | `@Autowired` 字段注入 -> 紧耦合, 单测时无法 mock, 需启动 Spring 上下文 | Service 单测变集成测试, 175 个测试执行从 25s 变 5 分钟+ | `@RequiredArgsConstructor` (Lombok 构造器注入) |
| 门面模式 | Account/Transaction/RecurringBill 三个 Service 各自写 `validateAccount(userId, accountId)` 校验逻辑 | 校验规则不一致 (A 校验归属, B 不校验), 改规则需改三处 | `EntityValidator.java` 统一校验入口 |
| 建造者/组合模式 | 拼接动态查询条件用 if-else 堆砌字符串 SQL -> 可读性差, 易 SQL 注入 | 字符串拼接 SQL 注入风险, 条件组合逻辑复杂难以维护 | `LambdaQueryWrapper<Transaction>().eq(...).orderByDesc(...)` |
| 观察者/调度模式 | 需要手动写 while(true) + sleep 轮询检查预算 -> CPU 浪费 + 不精确 | 轮询间隔难选 (太长预警延迟, 太短 CPU 浪费), 没有 Spring 调度框架的容错 | `@Scheduled(cron="0 0 2 * * ?")` 凌晨 2 点触发 |
| 路由守卫模式 | 每个页面自己检查是否登录 -> 漏一个页面就暴露未授权访问 | 未登录用户直接访问 URL 可看到页面, 安全性为零 | `router.beforeEach` 集中鉴权, 6 步检查 |

**补充: 第 13 个模式 -- 模板方法在路由守卫中的应用**:

`router.beforeEach` 集中定义了 6 步检查流程 (Token 过期预检 -> 需登录拦截 -> 管理员拦截 -> 已登录访问 /login 重定向 -> Chunk 加载失败容错 -> 正常放行), 每个路由变更都经过相同的鉴权模板。如果把这 6 步散落在 11 个页面的 `onMounted` 里, 新增页面时极易遗漏其中某一步。

**设计模式不是目的**: 以上每个模式都是"先遇到具体问题, 然后找到合适的模式解决", 而不是"为了用模式而用模式"。模板方法是继承 Spring 框架后自然获得的; 策略模式是错误码太多后自然重构出来的; 门面模式是发现三处重复校验后提取的。

## S7 高频对比问题

### Q22: MyBatis-Plus vs 原生 MyBatis? 为什么选 LambdaQueryWrapper?

**MyBatis-Plus 优势**:
1. `BaseMapper` 内置 CRUD (insert/selectById/updateById/deleteById), 减少重复代码
2. `LambdaQueryWrapper` 编译期类型安全, 字段引用 `User::getUsername` 而非字符串 `"username"`, IDE 重构字段名时自动同步
3. 分页插件 `PaginationInnerInterceptor` 一行配置即可

**为什么不全部用 XML**: 项目遵循 CLAUDE.md S二.四 规范--简单查询用 LambdaQueryWrapper，只有多表 JOIN / GROUP BY / 动态 ORDER BY 等复杂场景才写 XML。

**用法示例** (来自 `AccountServiceImpl.java:78-84`):
```java
accountMapper.selectList(
    new LambdaQueryWrapper<Account>()
        .eq(Account::getUserId, userId)       // 编译期校验字段存在
        .eq(Account::getStatus, 1)            // 活跃账户
        .orderByDesc(Account::getCreateTime)  // 倒序
);
```

### Q23: Vue 3 Composition API 比 Options API 好在哪？

1. **逻辑复用**: 可提取独立的 composable 函数 (如 `logger('DashboardPage')`)
2. **类型推导**: `<script setup>` + TypeScript 配合更好
3. **代码组织**: 相关逻辑自然聚合 (不像 Options API 按 data/methods/mounted 分散)
4. **Tree-shaking**: 未使用的 API 在构建时被 Vite 移除

**本项目全用 `<script setup>` 写法** (对齐 CLAUDE.md S三.二)。

### Q25: 事务管理怎么处理的？隔离级别是什么？

**Spring Boot 声明式事务**: `@Transactional` (由 MybatisPlusConfig @EnableTransactionManagement 启用)

**使用场景**:

| 场景 | 事务类型 | 隔离级别 | 代码位置 |
|---|:---:|---|---|
| 创建交易 (记一笔) | @Transactional | InnoDB 默认 REPEATABLE READ | `TransactionServiceImpl.java:176` |
| 更新交易 | @Transactional | REPEATABLE READ | `TransactionServiceImpl.java:219` |
| 删除交易 | @Transactional | REPEATABLE READ | `TransactionServiceImpl.java:279` |
| 转账 (双 INSERT) | @Transactional | REPEATABLE READ | `TransactionServiceImpl.java:310` |
| CSV 导入 | @Transactional | REPEATABLE READ | `TransactionServiceImpl.java:430` |
| 预算保存 (并发安全) | @Transactional | REPEATABLE READ | `BudgetServiceImpl.java:129` |
| 交易列表查询 | @Transactional(readOnly=true) | REPEATABLE READ | `TransactionServiceImpl.java:124` |
| 预算进度查询 | @Transactional(readOnly=true) | REPEATABLE READ | `BudgetServiceImpl.java:241` |
| 预警处理 (独立事务) | @Transactional(REQUIRES_NEW) | REPEATABLE READ | `BudgetAlertProcessorServiceImpl` |

**隔离级别选择理由**: MySQL InnoDB 默认 REPEATABLE READ 即可满足需求。转账场景下，事务内 SELECT 余额和 INSERT 共享同一快照，配合悲观锁 (`SELECT ... FOR UPDATE`) 可完全防止并发透支。无需升级到 SERIALIZABLE (性能损失大) 或降级到 READ COMMITTED (可能幻读)。

**传播行为**: 大部分场景使用默认 REQUIRED。只有 BudgetScheduler 中单用户预警处理使用 REQUIRES_NEW，确保 A 用户的预算预警失败不回滚 B 用户的预警记录。

**AOP 自调用问题**: BudgetScheduler 内部原先直接 `this.processUserBudgetAlerts()` 导致 REQUIRES_NEW 失效 (Spring AOP 代理自调用穿透)。修复方案: 提取为独立的 `BudgetAlertProcessorServiceImpl`，通过注入的 Spring 代理调用。

### Q26: 并发控制怎么做的？悲观锁 vs 乐观锁的选择？

**悲观锁 (`SELECT ... FOR UPDATE`)**:

适用场景:
- 转账: 两个账户余额必须同时锁定，`accountMapper.selectByIdForUpdate()` 获取行级排他锁
- 理由: 转账是强一致性操作，不允许乐观冲突重试

**乐观锁 (DuplicateKeyException 捕获)**:

适用场景:
- 预算保存: 先 insert，并发冲突时 catch DuplicateKeyException -> 回退 updateById
- 理由: 预算设置是低冲突场景 (一个用户一月同一分类只设置一次)

**无锁 (快照读)**:

适用场景:
- 收支列表查询: `@Transactional(readOnly=true)` + InnoDB MVCC 快照读
- 理由: 查询类操作，利用 MVCC 实现无锁一致性读取

#### 代码证据表

| 并发策略 | 文件 | 行号 | 场景 |
|---|:---|:---:|---|
| SELECT FOR UPDATE | `TransactionServiceImpl.java` | 329-335 | 转账悲观锁 |
| DuplicateKeyException | `BudgetServiceImpl.java` | 176-192 | 预算并发保存 |
| MVCC 快照读 | `TransactionServiceImpl.java` | 124 | 列表查询 readOnly=true |

### Q27: 缓存策略是什么？Cacheable 怎么用的？

**Spring Cache + @Cacheable**:

1. **分类列表缓存** (`CategoryServiceImpl.java:63`): 13 条种子数据几乎不变，标注 `@Cacheable("categories")`。首次查询后缓存到内存，后续请求直接返回缓存，无需查 DB。

2. **ECharts 懒加载缓存** (`echarts-lazy.js`): 单例 Promise 全局变量，首次 `import('echarts/core')` 后缓存，后续调用直接返回。失败时清除缓存允许重试。

**缓存策略考量**:
- 分类数据低频变更，适合长期缓存
- 不使用 Redis (教学项目，减少组件依赖)
- 不使用 Caffeine (内置 ConcurrentHashMap + @Cacheable 足够)

### Q28: 前端的错误处理和加载状态怎么设计的？

**Axios 拦截器统一处理** (`request.js`):

**Chunk 加载容错** (`router/index.js:177-186`):

## S10 失败场景演练

### 场景 1: 数据库宕机

**故障**: MySQL 服务崩溃或网络不可达。

**系统行为**:
1. 后端 `DataSourceConfig` 连接池 (HikariCP) 检测到连接丢失 -> `SQLException`
2. `GlobalExceptionHandler` 未直接捕获 (HikariCP 内部重试 3 次)
3. 最终抛出 `DataAccessException` -> 前端收到 `Result.error(500, "服务器内部错误")`

**恢复**: MySQL 重新启动 -> docker-compose healthcheck 通过 -> backend 健康检查恢复 -> curl /api/v1/health 返回 200

**改进建议** (生产环境):
- HikariCP `connectionTimeout: 30000`
- Circuit Breaker (Resilience4j) 快速失败
- 前端显示 "服务暂不可用，请稍后重试"

---

### 场景 2: JWT Token 过期

**故障**: 用户 token 已过期 (7天后) 但仍在使用系统。

**系统行为 (双重防护)**:
1. **前端预检**: `router.beforeEach` -> `isTokenExpired(token)` -> 过期 -> 清 localStorage + 跳 /login
2. **后端拦截**: 即使前端未拦截，`LoginInterceptor` -> `parseTokenPayload()` 验签失败 -> 返回 401
3. **axios 拦截器**: `code === 401` -> 清 token + 跳 /login + `ElMessage.error('未登录')`

**用户体验**: 无缝跳转到登录页，黄色警告 "登录已过期，请重新登录"

**代码位置**:
- 前端预检: `router/index.js:126-128`
- 后端拦截: `interceptor/LoginInterceptor.java:67-71`
- axios 拦截器: `api/request.js:88-101`

### 场景 3: 并发转账死锁 (已预防)

**故障**: 用户 A 从账户 1 转账到账户 2，同时用户 B 从账户 2 转账到账户 1。

**旧代码行为**: 死锁 -> InnoDB 死锁检测器 (innodb_lock_wait_timeout=50s) -> 回滚一个事务 -> 用户看到错误

**新代码行为**: 按账户 ID 升序加锁 -> `Math.min(1,2)`=1 先锁, `Math.max(1,2)`=2 后锁 -> 两个线程加锁顺序均为 1->2 -> 无循环等待 -> 零死锁

**验证方法**: `CrossModuleIntegrationTest` 中模拟两个线程并发转账

### 场景 4: CSV 格式错误

**故障**: 用户上传格式错误的 CSV 文件 (缺列、金额非数字、日期格式错、分类ID不存在)。

**系统行为 (逐行容错)**:
1. 首行 (表头) 自动跳过
2. 列数不足 -> failRow: "列数不足，至少需要4列"
3. 分类ID不存在 -> failRow: "分类ID X 不存在"

5. 金额 <=0 -> failRow: "金额必须大于0"

8. 成功行正常插入，失败行不影响成功行

**返回**: `ImportResultDTO { successCount: 80, failCount: 20, failRows: [{row: 5, reason: "..."}, ...] }`

**前端**: `ImportPage.vue` 显示导入结果面板，成功数绿色，失败行红色列表展示

### 场景 5: 内存溢出 (OOM) 预防

**故障风险点**:
1. 收支列表查询: 无分页限制 -> 可能查询全表 -> 大内存占用
2. CSV 导入: 超大文件 -> 内存爆炸
3. 登录限流: 攻击者大量虚构用户名 -> ConcurrentHashMap 无限增长

**防护措施**:
1. **pageSize 上限**: `Math.min(pageSize, 100)` -> 最多一次查 100 条 (`TransactionServiceImpl.java:129`)
2. **文件大小上限**: 5MB (`TransactionServiceImpl.java:435`)
3. **导入条数上限**: 1000 条 (`TransactionServiceImpl.java:515`)
4. **限流器 TTL 过期**: 10 分钟无活动的限流器自动移除 (`LoginRateLimiter.java:30`)
5. **限流器节流清理**: 采样触发，size>=100 或距上次清理超 60s (`LoginRateLimiter.java:94-110`)

### 场景 6: 前端构建失败 / Vite 热更新失效

**故障描述**: 前端执行 `pnpm build` 报错, 或 `pnpm dev` 启动后 HMR 热更新不生效。

**排查步骤 (按优先级)**:

1. **检查 Node.js 版本**: `node -v` 必须 >= 22 (pnpm 11+ 硬要求)。版本过低会导致依赖解析失败 -> 升级到 Node 24 LTS
2. **检查 pnpm 版本**: `pnpm -v` 必须 >= 10.33。版本不匹配可能导致 lockfile 冲突 -> `pnpm install --frozen-lockfile` 验证
3. **清理构建缓存**: `rm -rf node_modules && pnpm install` (重新装所有依赖, 解决幽灵模块问题)
4. **检查 Vite 配置**: `vite.config.js` 中的 proxy 目标 `http://localhost:8080` 是否可达 (后端是否已启动)
5. **HMR 不生效**: 检查 `import` 路径大小写 (Windows 不区分大小写但 Vite/Rolldown 区分, 导致模块图断裂)
6. **构建成功但页面白屏**: 打开浏览器 DevTools Console 查看 JS 报错; 常见原因: 路由懒加载 chunk 加载失败 (检查 `router/index.js` 的 `router.onError` 是否触发)
7. **Element Plus 样式丢失**: 确认 `main.js` 中有 `import 'element-plus/dist/index.css'`, 且手动分包配置 (`vite.config.js:24-47`) 没有把 Element Plus CSS 遗漏在 vendor-element chunk 之外

**快速恢复命令**:
```bash

cd frontend
rm -rf node_modules .vite dist
pnpm install
pnpm dev   # 验证 dev 正常
pnpm build # 验证 build 正常
```

**预防措施**:
- `package.json` 中锁定版本号 (去掉 `^` / `~` 前缀), 用 `pnpm-lock.yaml` 保证可复现构建
- Vite 配置中 `chunkSizeWarningLimit` 不要设太低 (当前 1000KB 合理), 过低会引发不必要的警告

### 场景 7: MySQL 连接池耗尽 (HikariCP 排错)

**故障描述**: 后端突然所有接口返回 500, 日志中出现 `HikariPool-1 - Connection is not available, request timed out after 30000ms`。

**根因分析**:
1. **连接泄漏**: 某处代码获取了 Connection 但没有正确关闭 (如 `DataSource.getConnection()` 后未 `close()`)
2. **连接池太小**: `maximumPoolSize=10` (HikariCP 默认), 当前并发请求数 > 10 且每个请求持有连接时间很长
3. **慢 SQL 阻塞**: 某个查询执行 >30s, 占着连接不释放, 后续请求排队等待直到超时
4. **数据库锁等待**: 转账场景 SELECT FOR UPDATE 长时间持有行锁, 导致其他事务等待

**排查步骤**:

| 步骤 | 命令/操作 | 目的 |
|---|---|---|
| 1 | 查看后端日志 `docker logs finance-backend` | 找 HikariPool 超时日志, 确认是连接池耗尽 |
| 2 | MySQL 查锁等待 `SHOW ENGINE INNODB STATUS\G` | 看是否有长时间未提交的事务 |
| 3 | `SELECT * FROM information_schema.INNODB_TRX` | 列出当前活跃事务, 按 `trx_started` 排序, 找长时间未提交的 |
| 4 | 检查 HikariCP 配置 `application.yml` | 确认 `maximumPoolSize` 和 `connectionTimeout` |
| 5 | 如果发现慢事务: `KILL <trx_mysql_thread_id>` | 杀死长时间未提交的事务 |
| 6 | 重启后端 `docker compose restart backend` | 重置连接池 |

**预防措施**:

| 措施 | 配置/代码 | 效果 |
|---|---|---|
| 增大连接池 | `spring.datasource.hikari.maximumPoolSize: 20` | 允许更多并发数据库连接 |
| 缩短超时 | `spring.datasource.hikari.connectionTimeout: 10000` | 10s 拿不到连接就报错, 不无限等 |
| 泄漏检测 | `spring.datasource.hikari.leakDetectionThreshold: 30000` | 连接持有超过 30s 打印警告日志 |
| 事务超时 | `@Transactional(timeout=10)` | 单事务超 10s 自动回滚, 不阻塞连接 |
| 读操作分离 | `@Transactional(readOnly=true)` | 告诉 MySQL 这是只读, 走 MVCC 快照, 不加锁 |

**关键代码检查清单**:
- 所有 `@Transactional` 方法是否可能执行超过 30s? (CSV 导入是典型的耗时场景, 需评估)
- 是否有方法手动获取 Connection? (项目中应没有, HikariCP 自动管理)
- `SELECT FOR UPDATE` 是否都及时提交/回滚? (转账方法有 `@Transactional`, 方法结束即提交)

**验证恢复**:
```bash
# 检查后端健康
curl http://localhost:8080/api/v1/health
# 预期: {"code":200,"message":"OK","data":"Service is running"}

# 检查 HikariCP 连接池状态
# 后端日志中找 "HikariPool-1 - Start completed" (启动成功)
# 以及 "HikariPool-1 - After cleanup" (活跃 > 空闲 > 等待 的统计)
```

```bash
# wrk 安装 (Windows 通过 WSL 或下载预编译版)
# 测试登录接口
wrk -t4 -c10 -d30s --latency -s login.lua http://localhost:8080/api/v1/user/login

# 测试交易列表
wrk -t4 -c20 -d30s --latency \
  -H "Authorization: Bearer <TOKEN>" \
  http://localhost:8080/api/v1/transaction?pageNum=1&pageSize=20
```

| wrk 指标 | 登录接口 | 交易列表 | 健康检查 |
|---|:---:|:---:|:---:|
| Requests/sec | ~18 | ~120 | ~800 |
| Avg Latency | ~550ms | ~165ms | ~15ms |
| p99 Latency | ~1200ms | ~450ms | ~50ms |

#### 压力测试结论

1. **登录是瓶颈** (BCrypt cost=12 单次 ~50ms 是硬上限), 但登录频率本身低, 10 并发足以覆盖百人课堂同时登录
2. **读操作为主** 的系统 (记账软件的典型特征), 单实例 SpringBoot 轻松支撑 50-100 并发
3. **数据库索引** 对查询性能决定性: transaction 表的 user_id + account_id + create_time 联合索引使得列表查询稳定在 100ms 以内
4. **HikariCP 连接池** 默认 10 连接在 50 并发以下绰绰有余, 无需调参
5. **生产环境** 如需支撑 500+ 并发: 加 Redis 缓存用户 token 和分类数据、Nginx 反向代理多实例、MySQL 读写分离

### 12.7 被追问时的万能回答框架 (3 步法)

当被问到你不完全确定的问题时，使用以下 3 步框架：

**第 1 步: 承认 (Acknowledge)** -- 不要硬编

> "这个问题问得很好，让我想想..."

或

> "这个细节我目前没有在代码中直接实现，但我理解这个场景..."

**第 2 步: 分析 (Analyze)** -- 基于已有知识推理

> "基于本项目的架构，如果要处理这个问题，我会从以下几个角度考虑..."

或

> "类似的问题在项目中有相关的处理，比如在 XXX 模块中我们用了 YYY 方案，这里可以类推..."

**第 3 步: 方案 (Propose)** -- 给出具体但诚实的方案

> "如果让我现在实现，我会：1) ... 2) ... 3) ..."

或

> "这个超出了我目前项目的范围，但据我所知，业界通常用 XXX 方案来解决。如果将来扩展，我会优先考虑..."

**关键原则**:

| 原则 | 说明 |
|---|---|
| 不要编造 | 不知道就说"据我所知"、"我理解是"、"项目中目前没有实现", 不要斩钉截铁说一个错的 |
| 联系项目 | 即使不会, 也要把问题拉回你熟悉的项目场景 ("虽然项目中没有 X, 但类似的 Y 我们是这样处理的...") |
| 展示思维 | 评委更看重你的分析过程而非标准答案, 展示"你会怎么学、怎么想"比"你知道"更重要 |
| 保持从容 | 被追问是好事 (说明评委对你的项目有兴趣), 不是刁难, 放平心态 |

**常见追问类型与应对策略**:

| 追问类型 | 应对策略 | 示例话术 |
|---|---|---|
| "为什么不用 XXX?" (技术选型) | 对比优缺点 + 项目约束 | "我们确实评估过 XXX, 但考虑到本项目是单实例部署、教学场景, YYY 更合适因为..." |

#### 扣分点 3: 数据库数据不一致

| 项目 | 内容 |
|---|---|
| **扣分场景** | 评委查看数据库发现: 转账记录不成对、余额计算有误、超支标记与实际不符 |
| **扣分幅度** | -5 ~ -10 分 (数据正确性是财务系统的底线) |
| **应对策略** | ① 答辩前执行 "数据一致性自检 SQL": 见下方; ② 如果有不一致的数据，答辩前修复或重新初始化数据库 (`mysql -uroot -proot < sql/01-init.sql`); ③ 如果评委指出某处数据不对，不要狡辩——承认疏忽并解释根因 |
| **自查 SQL** | 见下方 |

```sql
-- 答辩前数据一致性自检 (逐条执行，预期结果均满足才可答辩)
-- 1. 转账记录成对检查 (每组 transfer_id 必须有 2 条记录)
SELECT transfer_id, COUNT(*) AS cnt FROM transaction
WHERE transfer_id IS NOT NULL GROUP BY transfer_id HAVING cnt != 2;
-- 预期: Empty set (0 rows)

-- 2. 密码全部为 BCrypt 哈希
SELECT username FROM user WHERE password NOT LIKE '$2a$%';
-- 预期: Empty set (0 rows)

-- 3. 预算仅引用支出分类 (type=1)
SELECT b.id, c.name, c.type FROM budget b
JOIN category c ON c.id = b.category_id WHERE c.type != 1;
-- 预期: Empty set (0 rows)

-- 4. 活跃账户的余额数据完整性
SELECT a.id, a.name, a.initial_balance,
  COALESCE(SUM(CASE WHEN t.type=1 THEN t.amount ELSE 0 END),0) AS income,
  COALESCE(SUM(CASE WHEN t.type=2 THEN t.amount ELSE 0 END),0) AS expense
FROM account a
LEFT JOIN transaction t ON t.account_id = a.id AND t.user_id = a.user_id
WHERE a.status = 1 GROUP BY a.id, a.name, a.initial_balance;
-- 预期: 每行 income/expense 与前端余额页一致

-- 5. 分类总数 = 13
SELECT COUNT(*) FROM category;
-- 预期: 13
```

#### 扣分点 4: 代码与演示功能不匹配

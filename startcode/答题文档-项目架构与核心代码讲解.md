#### 讲稿（可直接念 · ~3分钟）

> "现在我用转账功能把整条数据流串起来。总共 17 个节点，分四个阶段：前端 5 个、网络传输、后端 7 个、响应返回 4 个。
>
> **前端第一阶段——用户操作到 HTTP 发出，5 个节点。**
>
> 节点①，用户在 TransferPage.vue 转账页面选择转出账户、转入账户、输入金额，点'确认转账'。这是整个数据流的起点——用户的一次点击，触发后面 16 个节点依次执行。
>
> 节点②，handleSubmit() 方法里，el-form.validate() 先做前端校验——金额必须大于 0、转出和转入账户不能相同。这层校验是秒级反馈，校验不通过根本不发 HTTP 请求，省带宽。
>
> 节点③，校验通过后，调 api/transaction.js 里的 transfer() 函数。注意——页面没有直接写 axios.post，而是走 api/ 层的封装。好处是如果后端改了 URL，只改这一个文件，所有调用的页面自动生效。
>
> 节点④，transfer() 内部调的是 request.js 创建的 axios 实例。请求拦截器在这里自动注入 token——从 localStorage 读取，拼成 `Bearer xxx` 格式，写入 Authorization 请求头。10 个 API 模块全走这一个拦截器，不会出现哪个模块忘了加 token 导致 401。
>
> 节点⑤，HTTP 请求发出去了，但前端跑在 localhost:5173，后端在 localhost:8080——端口不同就是跨域。Vite 的 proxy 配置在这里起作用：匹配 `/api` 前缀的请求，Vite Dev Server 帮你转发到 `localhost:8080`。浏览器以为是同源请求，不触发 CORS 预检。这就是开发环境解决跨域的标准做法——不用改后端代码。
>
> **接下来是网络传输。** 一个 HTTP POST 请求，通过本地 TCP 从 5173 端口发到 8080 端口。请求体是 JSON：转出账户 ID、转入账户 ID、金额、备注。
>
> **后端第二阶段——请求进入到数据库写入，7 个节点。这是整个数据流最核心的部分。**
>
> 节点⑥，请求最先到达的是 CorsConfig 配置的 CorsFilter。它在 Filter 层——比 Interceptor 更早执行。检查请求来源是否在允许列表中。为什么 CORS 必须放 Filter 层？因为浏览器的 OPTIONS 预检请求不带 Authorization 头——如果放 Interceptor 层，LoginInterceptor 发现没 token 就返回 401，预检失败，所有跨域请求都被浏览器拦截。Filter 在协议层处理，预检直接放行。
>
> 节点⑦，通过 Filter 后，请求到达 Spring 的 DispatcherServlet。它根据 URL `/api/v1/transaction/transfer` 匹配到 TransactionController 的 transfer() 方法。这是 Spring MVC 的核心——前端控制器模式，所有请求由 DispatcherServlet 统一分发。
>
> 节点⑧，在进入 Controller 之前，LoginInterceptor 的 preHandle() 方法拦截请求，校验 JWT token。分 4 步：OPTIONS 请求直接放行、从请求头提取 Bearer token、HMAC 验签、把 userId 存入 request.setAttribute。token 无效直接返回 401，Controller 的代码根本不会执行。这就是 AOP 面向切面编程——把'校验身份'和'处理业务'分离。
>
> 节点⑨，校验通过后，终于进入 TransactionController.transfer()。Controller 只做两件事：@Valid 校验参数、调 transactionService.transfer()。不写业务逻辑、不 try-catch。
>
> 节点⑩，核心来了——TransactionServiceImpl 的 transfer() 方法。这是我自选的后端核心方法。7 步业务逻辑：fail-fast 校验金额和账户、死锁预防按账户 ID 升序加锁、FOR UPDATE 悲观锁查余额、BigDecimal 精确计算、查分类、UUID 生成关联 ID 做复式记账、组装 DTO 返回。35 行代码覆盖 7 个知识点——第二部分会逐行讲。
>
> 节点⑪，Service 调 Mapper。MyBatis-Plus 的 BaseMapper 生成 SQL——SELECT ... FOR UPDATE 锁两行、INSERT 转出记录、INSERT 转入记录。通过 JDBC 发送到 MySQL。
>
> 节点⑫，MySQL InnoDB 引擎执行：获取行级排他锁 → 检查余额是否足够 → INSERT 转出记录 → INSERT 转入记录 → COMMIT 提交。整个方法有 @Transactional 注解——任何一步失败，全部回滚。不会出现'钱扣了但没到账'的情况。
>
> **响应第三阶段——后端响应返回前端，4 个节点。**
>
> 节点⑬，Service 返回数据后，Controller 用 Result.success() 包装成统一格式 `{code: 200, message: '转账成功', data: {...}}`。全系统所有接口都是这个格式。
>
> 节点⑭，Spring Boot 默认的 Jackson 序列化器把 Result 对象转成 JSON 字符串，通过 HTTP 响应体返回。
>
> 节点⑮，前端 request.js 的响应拦截器接收到响应。三段处理：code===200 剥壳只返回 data——业务代码拿到的直接是纯数据，不用再判断 code；code===401 清 token 跳登录页；其他 code 弹出错误提示。10 个 API 模块、11 个页面全部自动走这个逻辑。
>
> 节点⑯，数据回到 TransferPage.vue。handleSubmit() 拿到响应后：ElMessage.success 弹出'转账成功'提示 → resetFields() 清空表单 → loadAccounts() 重新加载账户余额。用户看到的是：成功提示、表单清空、余额更新——一次完整的转账操作结束。
>
> 这 17 个节点串起来就是一条完整的请求-响应链路：Vue 3 发起 → SpringBoot 3 处理 → MySQL 8 存储 → JSON 返回 → Vue 3 更新页面。三层技术栈，各司其职。"

| 节点  | 设计决策                 | 一句话解释                                           |

### 1.5 模块边界规则（30秒）

> "我定了严格的边界规则，保证代码不混乱："

**前端边界：**

- views 页面不直接调 axios → 必须走 api/ 封装
- api/ 不操作 localStorage → 用户信息由 stores/ 统一管理
- 父子通信只用 props/emit → 不滥用 Pinia

**后端边界：**

- Controller 只校验+转发 → 不写业务逻辑
- Service 只调 Mapper → 不手写 SQL（复杂查询除外）
- Controller 不 try-catch → 异常全由 GlobalExceptionHandler 处理
- 接口传参返参只用 DTO → 不直接返回数据库 Entity

### 2.1 后端请求处理链

```
一个 HTTP 请求进入后端，依次经过：

  [Filter 层]    CorsFilter.doFilter()         ← CORS 跨域处理（最早执行）
  [Dispatcher]   DispatcherServlet             ← URL → Controller 映射
  [Interceptor]  LoginInterceptor.preHandle()  ← JWT 认证（校验 token）
  [Interceptor]  AdminInterceptor.preHandle()  ← 管理员授权（仅 /admin 路径）
  [Controller]   @Valid 参数校验 + 调用 Service
  [Service]      业务逻辑 + @Transactional
  [Mapper]       MyBatis-Plus → JDBC → MySQL
  [Exception]    GlobalExceptionHandler        ← 有异常时统一处理

前端收到的响应依次经过：
  axios 响应拦截器（成功分支）→ code===200 剥壳返回 data
  axios 响应拦截器（失败分支）→ code===401 清 token 跳 /login / 其他 code 弹错误提示
  axios 响应拦截器（错误分支）→ 网络异常指数退避重试（1s→2s→4s，最多 3 次）
```

#### ② LoginInterceptor.java（3分钟）

> 打开文件 → 滚到 `preHandle()` 方法

**讲稿：**

> "LoginInterceptor 校验 JWT token。10 个 Controller 如果每个都写一遍 JWT 校验，重复且容易漏。拦截器把'校验身份'和'处理业务'分离——这就是 AOP 面向切面编程的思想。
> preHandle() 分 4 步：
> **第 1 步** OPTIONS 请求直接 return true 放行——预检没有 Authorization 头，不放行就被误拦。
> **第 2 步** 从请求头提取 `Authorization: Bearer <token>`——RFC 6750 标准格式，检查 `startsWith("Bearer ")` 防非 JWT 格式攻击。

> writeError() 返回 HTTP 200 + body.code=401——前端 axios 检查 body.code 而非 HTTP status，全系统统一 body-code-first 机制。"

**开场白（直接念）：**

**逐行讲——代码里已标注 ★★，边滚边念：**

| 步骤                  | 讲什么（核心要点）                                                                                                                                                                                                                                               |
|:-------------------:| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 入口 `@Transactional` | "Spring 声明式事务——此方法内所有数据库操作要么全部成功提交，要么任何一步失败全部回滚。转账三步（查余额+INSERT转出+INSERT转入）不可分割——如果 INSERT 转入时数据库崩了，转出也会回滚，不会'钱扣了没到账'。"                                                                                                                                 |
| 第1步 fail-fast 校验    | "金额非空+必须>0+不能自己转自己。为什么放加锁之前？快速失败不占用数据库连接池——锁是最贵的资源，任何能在加锁前排除的错误都不应该占着锁。"                                                                                                                                                                                |
| 第2步 死锁预防            | "`Math.min/max` 按账户 ID 升序确定加锁顺序——先锁小 ID 再锁大 ID。线程1转账户1→5先锁1等5，线程2转5→1同样是 min(5,1)=1 先锁1再锁5——所有线程锁顺序一致，打破操作系统的'循环等待'条件，死锁就不可能发生。`selectByIdForUpdate()` = SELECT ... FOR UPDATE，InnoDB 行级排他锁。"                                                           |
| 第3步 账户校验            | "校验账户存在+归属当前用户+状态活跃。`Objects.equals` 比较 Integer 值而不是 `==`——Integer 在 -128~127 范围内有缓存，超出范围的 `==` 比较的是引用地址，可能两个值相同的 Integer 返回 false。"                                                                                                                    |
| 第4步 BigDecimal 余额计算 | "初始余额+收入总和-支出总和=当前余额。`BigDecimal.add().subtract()` 链式调用。为什么必须用 BigDecimal？如果用了 float/double——0.1+0.2=0.30000000000000004≠0.3——财务系统一分钱都对不上，生产事故。`compareTo(amount) < 0` 判断余额不足。"                                                                         |
| 第5步 查"其他"分类         | "`LambdaQueryWrapper` 查 name='其他' AND type=1（支出类）。为什么加 type 过滤？种子数据有 13 条分类，'其他'有两个——支出类(id=8)和收入类(id=13)。不加过滤 MyBatis-Plus 的 `selectOne` 查出来两行会抛 TooManyResultsException，转账直接崩溃。"                                                                      |
| 第6步 UUID + 复式记账     | "`UUID.randomUUID()` 生成转账关联 ID——在 Java 代码里提前生成，INSERT 之前就能设给两条记录。为什么不用数据库自增 ID？自增是 INSERT 后才有值——但我们需要 INSERT 之前就拿到关联 ID 同时赋给转出和转入两条记录。然后分别 INSERT 一条转出(type=2=支出)和一条转入(type=1=收入)，共享同一个 transferId——这就是'复式记账'：一出一入，资金流向完全可追溯。金额完全相等——转账不创造也不消灭金钱，只是转移。" |
| 收尾组装返回              | "组装 TransferDTO——包含 transferId、转出记录 DTO、转入记录 DTO。返回给 Controller → `Result.success()` 包装 → Jackson 序列化 JSON → 前端展示。"                                                                                                                                     |

**收尾总结（直接念）：**

> "这 35 行代码覆盖了 7 个知识点：fail-fast 校验、死锁预防、FOR UPDATE 悲观锁、BigDecimal 精度、@Transactional 原子性、复式记账、UUID 关联——都是后端开发必知必会。"

## 常见追问 & 应对

| 老师可能问                                       | 怎么答                                                                                                                                                  |
| ------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| "为什么选转账而不是登录？"                              | "登录只做查表+BCrypt+签发 token，3 步就完了，太简单。转账涉及死锁预防、悲观锁、BigDecimal 精度、事务原子性——7 个知识点，能展示技术深度。"                                                                |

| "body-code-first 是什么意思？"                    | "后端 HTTP 状态码始终返回 200，业务状态用 body 里的 code 字段区分——200 成功、401 未登录、其他业务错误。好处是前端 axios 拦截器只写一段 if-else，所有页面自动受益。不用区分 HTTP 401 走 error 回调、200 走 success 回调。" |

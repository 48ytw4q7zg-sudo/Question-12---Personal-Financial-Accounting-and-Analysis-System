|:---:|------|---------|--------|
| ⑥ | `config/CorsConfig.java` 的 `CorsFilter` | 检查请求来源是否在允许列表中，设置 `Access-Control-Allow-Origin` 等响应头 | **Filter 层**——HTTP 协议层处理，比 Interceptor 更早执行。OPTIONS 预检不带 Authorization 头，必须在此层处理 |
| ⑦ | Spring `DispatcherServlet` | URL `/api/v1/transaction/transfer` 映射到 `TransactionController.transfer()` | Spring MVC 核心——根据 @RequestMapping 匹配 Controller 方法 |
| ⑧ | `interceptor/LoginInterceptor.java` 的 `preHandle()` | 4步校验：① OPTIONS放行 ② 提取 `Bearer <token>` ③ `parseTokenPayload()` 一次验签 ④ `request.setAttribute("userId", ...)` | **Interceptor 层**——在 Controller 之前校验身份。如果 token 无效直接返回 401，业务方法根本不会执行 |
| ⑨ | `controller/TransactionController.java` 的 `transfer()` | `@Valid` 校验参数 → `LoginInterceptor.getUserId(request)` 取 userId → 调用 `transactionService.transfer(userId, request)` | Controller 只做参数校验和转发，不写业务逻辑——职责单一 |
| ⑩ | **`service/impl/TransactionServiceImpl.java` 的 `transfer()`** | 7 步核心业务（详见 §二·④）：fail-fast校验→死锁预防加锁→账户校验→BigDecimal余额计算→查分类→UUID+复式记账双条INSERT | **★ 30分核心**——所有业务逻辑在这里，与 HTTP/数据库解耦 |
| ⑪ | `mapper/TransactionMapper.java` + MyBatis-Plus | `SELECT ... FOR UPDATE` 锁行 → `accountMapper.selectByIdForUpdate()` 两次 → `transactionMapper.insert()` 两次 | MyBatis-Plus 代理生成 SQL，通过 JDBC 发送到 MySQL |
| ⑫ | MySQL InnoDB | 行锁获取 → 余额检查 → INSERT 转出记录 → INSERT 转入记录 → COMMIT 提交事务 | `@Transactional` 保证原子性——任何一步失败全部回滚 |

| ⑮ | `api/request.js` 响应拦截器 | 检查 `res.data.code`：code===200 → 剥壳只返回 `data` | 业务代码拿到的直接是纯数据，无需重复判断 code |
| ⑯ | `views/TransferPage.vue` 的 `handleSubmit()` | `ElMessage.success('转账成功')` → `formRef.resetFields()` → `loadAccounts()` 刷新余额 | 用户看到成功提示 + 表单清空 + 账户余额更新 |

#### 数据流总结（15 个节点 · 3 层技术栈）

```
前端 5 节点          网络    后端 7 节点               数据库     响应 3 节点
TransferPage        TCP     CorsFilter              MySQL      Result包装
  ↓                         ↓                       InnoDB     Jackson序列化
form.validate       →→→    DispatcherServlet        SELECT     JSON响应
  ↓                         ↓                       FOR UPDATE    ↓
api/transaction.js         LoginInterceptor         INSERT×2   request.js
  ↓                         ↓                       COMMIT     响应拦截器
request.js 拦截器          TransactionController       ↓         ↓
  ↓                         ↓                                TransferPage
Vite Proxy → :8080        TransactionServiceImpl             ElMessage
                           ↓                                 刷新余额
                          TransactionMapper
```

**必须记住的 6 个关键设计决策：**

| 节点 | 设计决策 | 一句话解释 |
|-----|---------|-----------|
| ④ | 请求拦截器统一注入 token | 10 个 API 模块不会遗漏——少一个就 401 |
| ⑤ | Vite Proxy 转发 /api | 开发环境用代理解决跨域，不用改后端代码 |
| ⑥ | CORS 放在 Filter 层 | OPTIONS 预检没有 Authorization 头，放 Interceptor 会被误拦 |
| ⑧ | JWT 放在 Interceptor 层 | 在 Controller 之前校验身份，校验不过业务代码不执行 |
| ⑩ | 业务逻辑放在 Service 层 | 与 HTTP 协议和数据库操作解耦——换个前端框架或数据库，Service 不用改 |
| ⑮ | 响应拦截器三段处理 | 200 剥壳 / 401 跳转 / 其他提示——前端所有页面统一受益 |

### 1.5 模块边界规则（30秒）

> "我定了严格的边界规则，保证代码不混乱："

**前端边界：**
- views 页面不直接调 axios → 必须走 api/ 封装
- api/ 不操作 localStorage → 用户信息由 stores/ 统一管理
- 父子通信只用 props/emit → 不滥用 Pinia

**后端边界：**
- Controller 只校验+转发 → 不写业务逻辑
- Service 只调 Mapper → 不手写 SQL（复杂查询除外）
- Controller 不 try-catch → 异常全由 GlobalExceptionHandler 处理
- 接口传参返参只用 DTO → 不直接返回数据库 Entity

### 2.1 后端请求处理链

```
一个 HTTP 请求进入后端，依次经过：

  [Filter 层]    CorsFilter.doFilter()         ← CORS 跨域处理（最早执行）
  [Dispatcher]   DispatcherServlet             ← URL → Controller 映射
  [Interceptor]  LoginInterceptor.preHandle()  ← JWT 认证（校验 token）
  [Interceptor]  AdminInterceptor.preHandle()  ← 管理员授权（仅 /admin 路径）
  [Controller]   @Valid 参数校验 + 调用 Service
  [Service]      业务逻辑 + @Transactional
  [Mapper]       MyBatis-Plus → JDBC → MySQL
  [Exception]    GlobalExceptionHandler        ← 有异常时统一处理

前端收到的响应依次经过：
  axios 响应拦截器（成功分支）→ code===200 剥壳返回 data
  axios 响应拦截器（失败分支）→ code===401 清 token 跳 /login / 其他 code 弹错误提示
  axios 响应拦截器（错误分支）→ 网络异常指数退避重试（1s→2s→4s，最多 3 次）
```

#### ② LoginInterceptor.java（3分钟）

> 打开文件 → 滚到 `preHandle()` 方法

**讲稿：**
> "LoginInterceptor 校验 JWT token。10 个 Controller 如果每个都写一遍 JWT 校验，重复且容易漏。拦截器把'校验身份'和'处理业务'分离——这就是 AOP 面向切面编程的思想。
> preHandle() 分 4 步：
> **第 1 步** OPTIONS 请求直接 return true 放行——预检没有 Authorization 头，不放行就被误拦。
> **第 2 步** 从请求头提取 `Authorization: Bearer <token>`——RFC 6750 标准格式，检查 `startsWith("Bearer ")` 防非 JWT 格式攻击。
> **第 3 步** `parseTokenPayload()` 一次 HMAC 验签，同时提取 userId 和 role——旧方案调用两次解析，合并后减少 50% 密码学运算。
> **第 4 步** `request.setAttribute("userId", ...)` 存入请求属性——为什么不用 ThreadLocal？Tomcat 线程池复用线程，ThreadLocal 没清干净会导致用户 A 读到用户 B 的数据——串号安全漏洞。setAttribute 每个请求独立对象，天然隔离。
> writeError() 返回 HTTP 200 + body.code=401——前端 axios 检查 body.code 而非 HTTP status，全系统统一 body-code-first 机制。"

▶ 下一个文件：`system/backend/.../config/WebMvcConfig.java`

#### ④ ★ TransactionServiceImpl.transfer()（5分钟 · 30分核心）

> 打开文件 → 滚到 `transfer()` 方法 → 逐行讲解

**开场白（直接念）：**
> "这是我选的后端核心方法——TransactionServiceImpl 的 transfer() 转账方法。35 行代码覆盖了 7 个知识点：fail-fast 校验、死锁预防、FOR UPDATE 悲观锁、BigDecimal 精度、@Transactional 原子性、复式记账、UUID 关联。比登录的技术深度高得多。"

**逐行讲——代码里已标注 ★★，边滚边念：**

| 步骤 | 讲什么（核心要点） |
|:---:|------|
| 入口 `@Transactional` | "Spring 声明式事务——此方法内所有数据库操作要么全部成功提交，要么任何一步失败全部回滚。转账三步（查余额+INSERT转出+INSERT转入）不可分割——如果 INSERT 转入时数据库崩了，转出也会回滚，不会'钱扣了没到账'。" |
| 第1步 fail-fast 校验 | "金额非空+必须>0+不能自己转自己。为什么放加锁之前？快速失败不占用数据库连接池——锁是最贵的资源，任何能在加锁前排除的错误都不应该占着锁。" |
| 第2步 死锁预防 | "`Math.min/max` 按账户 ID 升序确定加锁顺序——先锁小 ID 再锁大 ID。线程1转账户1→5先锁1等5，线程2转5→1同样是 min(5,1)=1 先锁1再锁5——所有线程锁顺序一致，打破操作系统的'循环等待'条件，死锁就不可能发生。`selectByIdForUpdate()` = SELECT ... FOR UPDATE，InnoDB 行级排他锁。" |
| 第3步 账户校验 | "校验账户存在+归属当前用户+状态活跃。`Objects.equals` 比较 Integer 值而不是 `==`——Integer 在 -128~127 范围内有缓存，超出范围的 `==` 比较的是引用地址，可能两个值相同的 Integer 返回 false。" |
| 第4步 BigDecimal 余额计算 | "初始余额+收入总和-支出总和=当前余额。`BigDecimal.add().subtract()` 链式调用。为什么必须用 BigDecimal？如果用了 float/double——0.1+0.2=0.30000000000000004≠0.3——财务系统一分钱都对不上，生产事故。`compareTo(amount) < 0` 判断余额不足。" |
| 第5步 查"其他"分类 | "`LambdaQueryWrapper` 查 name='其他' AND type=1（支出类）。为什么加 type 过滤？种子数据有 13 条分类，'其他'有两个——支出类(id=8)和收入类(id=13)。不加过滤 MyBatis-Plus 的 `selectOne` 查出来两行会抛 TooManyResultsException，转账直接崩溃。" |
| 第6步 UUID + 复式记账 | "`UUID.randomUUID()` 生成转账关联 ID——在 Java 代码里提前生成，INSERT 之前就能设给两条记录。为什么不用数据库自增 ID？自增是 INSERT 后才有值——但我们需要 INSERT 之前就拿到关联 ID 同时赋给转出和转入两条记录。然后分别 INSERT 一条转出(type=2=支出)和一条转入(type=1=收入)，共享同一个 transferId——这就是'复式记账'：一出一入，资金流向完全可追溯。金额完全相等——转账不创造也不消灭金钱，只是转移。" |
| 收尾组装返回 | "组装 TransferDTO——包含 transferId、转出记录 DTO、转入记录 DTO。返回给 Controller → `Result.success()` 包装 → Jackson 序列化 JSON → 前端展示。" |

**收尾总结（直接念）：**
> "这 35 行代码覆盖了 7 个知识点：fail-fast 校验、死锁预防、FOR UPDATE 悲观锁、BigDecimal 精度、@Transactional 原子性、复式记账、UUID 关联——都是后端开发必知必会。"

> "`loading.value = true` 开启表格 loading → 构建 params 对象，只有用户选了筛选条件才拼接——`if (filters.dateRange)`、`if (filters.accountId != null)`（用 `!= null` 而不是 `if(accountId)` 因为 0 也是合法 ID，但 0 是 falsy）→ `await getTransactionList(params)` 发 GET 请求 → 响应用可选链 `data?.records \|\| data?.list \|\| data \|\| []` 兼容后端三种可能的返回格式——万一后端重构改了字段名，前端不用改代码。catch 里 `ElMessage.error` 提示 + 清空列表兜底。finally 关闭 loading——保证一定执行。"

**函数 3：syncFiltersToUrl()（滚到第 394 行）**
> "把筛选条件写入 URL query 参数：`router.replace({ query: q })`。为什么用 replace 而不是 push？replace 替换当前历史记录，不产生新的 history 条目——用户点'后退'回到上一页，不是上一个筛选状态。如果用 push，用户筛选 10 次要按 10 次后退才能离开页面——这叫'历史污染'。`readFiltersFromUrl()` 是反向操作——页面初始化时从 URL 读取筛选条件恢复状态。URL 参数是字符串 `'3'`，el-select 的 value 是数字 `3`——`Number(q.accountId)` 做类型转换，否则匹配不上显示空白。这两个函数形成 URL↔状态双向同步闭环。"

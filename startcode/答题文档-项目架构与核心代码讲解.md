### 1.2 各目录职责一句话说明

**后端（12个目录/子目录）**：

| 目录 | 文件数 | 职责 | 通俗类比 |
|---|---|:---:|---|

### 2.2 过滤器 — CorsFilter（config/CorsConfig.java）

**为什么放在Filter层**：CORS是HTTP协议层面的事，浏览器发OPTIONS预检请求时没有Authorization头。如果放在Interceptor层，会被LoginInterceptor拦截返回401。

```java
// 第74-88行：创建CorsFilter Bean

    CorsConfiguration config = new CorsConfiguration();
    // 逐个添加允许源（为什么不用 * 通配符？因为 allowCredentials=true 时 OWASP 安全规范禁止通配）
    Arrays.stream(allowedOrigins.split(",")).map(String::trim).forEach(config::addAllowedOrigin);
    config.setAllowCredentials(true);   // 允许携带Cookie
    config.addExposedHeader("Authorization");  // 暴露Authorization头给前端读取
    config.setMaxAge(3600L);            // OPTIONS预检缓存1小时，减少预检请求

    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", config);  // 所有路径都适用
    return new CorsFilter(source);
}
```

### 2.3 拦截器 — LoginInterceptor（interceptor/LoginInterceptor.java）

**为什么需要拦截器**：如果10个Controller里都写JWT校验代码 → 重复、容易遗漏。拦截器把"谁可以访问"从"访问了什么"中分离。

**Filter vs Interceptor 区别**：

- Filter是Servlet级别，最早执行，不知道请求会去哪个Controller
- Interceptor是Spring MVC级别，知道目标Controller和方法，可以注入Bean

```java
@Component
public class LoginInterceptor implements HandlerInterceptor {

    private static final int HTTP_OK = 200;  // 为什么是200不是401？
    // 前端axios拦截器检查的是res.data.code（body里的code），不是HTTP status code
    // 如果返回HTTP 401，axios的error拦截器会捕获，中文错误消息会丢失

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
        // 第1步：OPTIONS预检请求直接放行（CORS预检没有Authorization头）
        if ("OPTIONS".equalsIgnoreCase(request.getMethod())) {
            return true;
        }

        // 第2步：提取Authorization头，格式为 "Bearer <token>"
        String authHeader = request.getHeader("Authorization");
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            writeError(response);  // 写入401 JSON → HTTP 200 + body {code:401}
            return false;          // 中断请求链，Controller不会执行
        }

        // 第3步：截取token，一次解析提取userId+role
        String token = authHeader.substring("Bearer ".length()).trim();
        JwtUtils.JwtPayload payload = JwtUtils.parseTokenPayload(token);
        // 为什么一次解析：旧代码先parseToken()再parseRole()，每次都要HMAC验签
        // 合并为一次解析，减少50%密码学运算
        if (payload == null) {
            writeError(response);
            return false;
        }

        // 第4步：将userId和role存入request属性（Controller通过getUserId()获取）
        request.setAttribute("userId", payload.getUserId());
        request.setAttribute("role", payload.getRole() != null ? payload.getRole() : 0);
        // 为什么用request.setAttribute而不是ThreadLocal？
        // 每个HTTP请求有独立的request对象 → 天然隔离
        // Tomcat线程池复用线程，如果用ThreadLocal可能串号（线程A的用户ID泄漏给线程B）

        return true;  // 放行
    }

    // Controller通过此静态方法获取当前用户ID
    public static Long getUserId(HttpServletRequest request) {
        Long userId = (Long) request.getAttribute("userId");
        if (userId == null) {
            throw new BusinessException(401, "未登录");  // fail-fast
        }
        return userId;
    }
}
```

### 2.4 后端核心 — TransactionServiceImpl.transfer() 逐行讲解

**文件**：`service/impl/TransactionServiceImpl.java`（604行）

为什么选这个方法：transfer() 覆盖了后端开发的核心知识点——前置校验、死锁预防、FOR UPDATE悲观锁、BigDecimal精度计算、@Transactional原子性、复式记账。代码约35行，比登录复杂得多，体现真实企业级后端开发的技术深度。

```java
@Override
@Transactional(readOnly = true)  // ← 只读事务。为什么？MySQL可以做读写分离优化
public LoginResponse login(UserLoginRequest request) {

    // ===== 第1步：登录限流 =====
    if (!LoginRateLimiter.tryAcquire(request.getUsername())) {
        throw new BusinessException(ErrorCode.LOGIN_RATE_LIMIT.getCode(),
            ErrorCode.LOGIN_RATE_LIMIT.getMsg());
    }
    // 这一行做什么：调用Guava RateLimiter，同一用户名每秒最多2次尝试
    // 为什么这样写：限流放在查数据库之前 —— 防止暴力破解消耗数据库连接池
    // 为什么RateLimiter.create(2.0)：正常用户不会每秒登录超过2次，超过说明可能是攻击

    // ===== 第2步：查询用户 =====
    User user = userMapper.selectOne(
        new LambdaQueryWrapper<User>().eq(User::getUsername, request.getUsername())
    );
    // 这一行做什么：用Lambda表达式构建SQL条件 WHERE username = ?
    // 为什么用LambdaQueryWrapper而不是写SQL：
    //   1. 类型安全 —— User::getUsername 编译时检查，字段名写错编译器报错
    //   2. 防SQL注入 —— MyBatis-Plus自动参数化，用PreparedStatement
    //   3. 重构友好 —— 改字段名时编译器自动提醒所有引用处

    // ===== 第3步：密码验证 + 防枚举攻击 =====
    if (user == null || !BCRYPT_ENCODER.matches(request.getPassword(), user.getPassword())) {
        throw new BusinessException(ErrorCode.PASSWORD_ERROR.getCode(),
            ErrorCode.PASSWORD_ERROR.getMsg());
    }
    // 这一行做什么：BCrypt不可逆哈希比对密码
    // 为什么"用户不存在"和"密码错误"统一返回"用户名或密码错误"？
    //   如果区分返回 → 攻击者可以枚举哪些用户名已注册（用户名枚举攻击）
    //   统一返回 → 攻击者无法判断用户名是否存在
    // 为什么BCrypt而不是MD5：
    //   1. BCrypt不可逆，即使数据库泄露也无法还原明文
    //   2. 自动加盐，相同密码产生不同哈希值
    //   3. 工作因子12 = 4096次迭代，暴力破解成本极高

    // ===== 第4步：签发JWT =====
    String token = JwtUtils.generateToken(user.getId(), user.getUsername(), user.getRole());
    // 这一行做什么：生成JWT token，包含 sub=userId, username, role, iat(签发时间), exp(过期时间)
    // 为什么JWT里存username和role：避免每次请求都查数据库
    // JWT自包含用户基本信息，拦截器解析token就能拿到身份 → 无状态设计

    // ===== 第5步：清理限流器 =====
    LoginRateLimiter.cleanup(request.getUsername());
    // 这一行做什么：登录成功后移除限流器，释放内存
    // 为什么：限流器只在登录失败时需要，成功后不再需要

    // ===== 第6步：返回 =====
    return new LoginResponse(token, user.getId(), user.getUsername(), user.getRole());
}
```

**这个方法涉及的知识点总结**：
| 知识点 | 在哪里体现 |
|--------|----------|
| 限流（Rate Limiting） | 第1步 Guava RateLimiter |
| ORM（对象关系映射） | 第2步 LambdaQueryWrapper |
| 防SQL注入 | 第2步 自动参数化查询 |
| 密码安全 | 第3步 BCrypt不可逆哈希 |
| 防枚举攻击 | 第3步 不区分用户不存在/密码错误 |
| JWT无状态认证 | 第4步 自包含token |
| 事务管理 | @Transactional(readOnly=true) |

```javascript
// 响应拦截器 — 每次收到后端响应时自动执行
request.interceptors.response.use(res => {
    const { code, message, data } = res.data || {}  // 解构统一响应结构 {code, message, data}

    if (code === 200) {
        return data  // 成功 → 返回纯业务数据
        // 为什么return data而不是整个res.data：所有API函数直接拿到业务数据
        // 不需要 .then(res => res.data.data)
    } else if (code === 401) {
        // token过期 → 清除登录状态 + 跳登录页
        const userStore = useUserStore()
        userStore.clearUser()                       // 清Pinia + localStorage
        ElMessage.error('登录已过期，请重新登录')
        router.push({ path: '/login', query: { redirect: currentPath } })
        // 为什么带redirect：登录后跳回原页面
        return Promise.reject(new Error(message))
    } else {
        // 其他业务错误码（如1001用户名重复）
        ElMessage.error(message || '请求失败')       // 弹出红色提示
        return Promise.reject(new Error(message))
    }
}, async error => {
    // 错误分支 — 处理网络异常（超时、断网、服务器5xx）
    // 指数退避重试：1秒 → 2秒 → 4秒，最多3次
    if (config && isRetryable(error) && config.retryCount < 3) {
        config.retryCount += 1
        const delayTime = Math.min(1000 * Math.pow(2, config.retryCount - 1), 10000)
        await delay(delayTime)                     // 等待后重试
        return request(config)
    }
    // 为什么指数退避：避免所有客户端同时重试造成"惊群效应"
    // 为什么加随机抖动：进一步分散重试时间
})
```

#### 2.5.4 前端路由守卫（router/index.js 第118-160行）

```javascript
router.beforeEach(async (to, from, next) => {
    const token = localStorage.getItem('token')

    // 第1步：JWT过期预检
    if (token) {
        tokenExpired = isTokenExpired(token)
        // 为什么在路由层预检而不等后端返回401：更快，不用发请求就能判断
    }

    // 第2步：需要登录但没token → 跳登录页
    if (requiresAuth && (!token || tokenExpired)) {
        next({ path: '/login', query: { redirect: to.fullPath } })

    // 第3步：已登录还访问/login → 跳首页（防止重复登录）
    } else if (to.path === '/login' && token && !tokenExpired) {
        next('/')

    // 第4步：管理员权限检查
    } else if (to.matched.some(record => record.meta.requiresAdmin)) {
        const userStore = useUserStore()
        if (userStore.role !== 1) {   // 非管理员
            ElMessage.warning('无权限访问该页面')
            next('/')
            return
        }
        next()
    } else {
        next()  // 正常放行
    }
})
```

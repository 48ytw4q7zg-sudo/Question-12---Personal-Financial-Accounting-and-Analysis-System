#### 节点 ⓪-B：路由守卫鉴权 → 路由懒加载 → TransferPage 渲染 → 加载账户列表

**触发**：`router.push('/transfer')` 触发 Vue Router 的导航解析流程。

**第 1 层 —— `router/index.js` 第 151 行 `router.beforeEach()` 全局前置守卫**：

这是 Vue Router 的核心安全机制——每次路由切换前（用户点击链接 / `router.push()` / 浏览器前进后退）都会执行此函数。守卫是一个 `async (to, from, next) => {}` 函数，共 6 步检查：

- **第 1 步（第 161 行）**：`const token = localStorage.getItem('token')` —— 从浏览器 localStorage 按 `'token'` 键读取 JWT。键名 `'token'` 与 `stores/user.js` 登录时写入的键名和 `api/request.js` 请求拦截器读取的键名保持一致——三个读写点闭环。为什么用 localStorage 而非 cookie：① JWT 无状态不需要 cookie 的自动携带 ② CSRF 防御——cookie 自动附加到同域请求会被 CSRF 利用，localStorage 只能由 JS 显式读取后放入 Authorization 头 ③ API 简洁——`getItem/setItem/removeItem` 比 `document.cookie` 字符串操作简单

- **第 2 步（第 173 行）**：`tokenExpired = isTokenExpired(token)` —— 调 `utils/jwt.js` 的 `isTokenExpired()`。该函数 Base64 解码 JWT payload 的 `exp` 字段（UNIX 秒级时间戳），与 `Date.now() / 1000` 比较。**纯本地计算 < 1ms，不等后端返回 401 就在路由层拦截。** 为什么不等后端：等后端 401 → 页面先渲染 → mounted 发 API → 后端校验 token → 返回 401 → 响应拦截器跳 /login → 用户看到页面闪烁后再跳到登录页。路由层预检 → 零渲染、零闪烁

- **第 3 步（第 186-202 行）**：`const requiresAuth = to.matched.some(record => record.meta.requiresAuth)` —— 遍历目标路由到根路由的完整匹配链。访问 `/transfer` 时 `to.matched` = `[{path:'/', meta:{requiresAuth:true}}, {path:'transfer', meta:{title:'转账'}}]`。父路由 AppLayout（`/`）标记了 `requiresAuth: true`，`some()` 返回 true。为什么用 `to.matched.some()` 遍历而非直接读 `to.meta.requiresAuth`：Vue Router 的 meta 不会自动从父路由继承到子路由——子路由 `/transfer` 的 meta 中只有 `title` 没有 `requiresAuth`，直接读得到 undefined。token 有效且未过期 → 跳过拦截

- **第 4 步（第 210-211 行）**：`to.path === '/login' && token && !tokenExpired` → 已登录用户访问 /login 直接跳首页 `/`。防止用户手动输入 `/login` 看到登录表单再次提交造成重复登录

- **第 5 步（第 215-229 行）**：`to.matched.some(record => record.meta.requiresAdmin)` → 仅 `/admin` 路由有此标记。从 Pinia store 读取 `userStore.role` 而非重新解析 JWT——避免重复 Base64 解码 + 保证单一数据源

- **第 6 步（第 234 行）**：`next()` 无参放行——导航继续，Vue Router 开始解析目标路由

**第 2 层 —— 路由懒加载（JS chunk 按需加载）**：

`router/index.js` 第 108 行定义了 `/transfer` 的路由：

```javascript
{ path: 'transfer', name: 'Transfer',
  component: () => import('../views/TransferPage.vue'),
  meta: { title: '转账' } }
```

`() => import('../views/TransferPage.vue')` 是 ES 动态 `import()` —— Vite 在打包时将 TransferPage.vue 编译为独立的 JS chunk 文件（如 `TransferPage-abc123.js`），**仅在用户首次访问 `/transfer` 时才发起 HTTP 请求加载这个 chunk**。好处：① 初始页面加载更快——登录时不需要下载所有 11 个页面的代码 ② 按需加载节省带宽——用户只下载实际访问的页面 ③ Vite 自动做代码分割——不需要手动配置

**第 3 层 —— TransferPage.vue 组件初始化（`<script setup>` 顶层代码执行）**：

Vue 3 的 `<script setup>` 是编译时语法糖——编译器将顶层变量自动暴露给 `<template>`，不需要手动 `return`。执行顺序：

1. **第 86 行** `import { ref, reactive, onMounted } from 'vue'` —— 导入 Vue 3 组合式 API 的三个核心函数。`ref()` 包装基本类型为响应式对象（通过 `.value` 读写），`reactive()` 包装对象为深层响应式代理（基于 ES6 Proxy），`onMounted()` 注册组件挂载后的回调

2. **第 88 行** `import { ElMessage } from 'element-plus'` —— 导入 Element Plus 的消息提示函数（弹窗式），不同于 `ElMessageBox`（确认对话框）。`ElMessage.success/error/warning` 分别显示绿/红/黄色提示

3. **第 89 行** `import { transfer } from '../api/transaction'` —— 按需导入（named import）转账 API 函数。`api/transaction.js` 导出了 6 个函数（getTransactionList / createTransaction / updateTransaction / deleteTransaction / transfer / importCsv），这里只需要 `transfer`

4. **第 90 行** `import { getAccountList } from '../api/account'` —— 导入账户列表 API 函数。`api/account.js` 第 47-51 行 `export function getAccountList() { return request.get('/account') }`

5. **第 91 行** `import { MIN_TRANSACTION_AMOUNT, MAX_NOTE_LENGTH, AMOUNT_STEP_ROUGH } from '../constants/finance'` —— 导入业务常量。`MIN_TRANSACTION_AMOUNT = 0.01`（最小转账金额 1 分钱），`MAX_NOTE_LENGTH = 200`（备注最大长度），`AMOUNT_STEP_ROUGH`（金额步进值）

6. **第 93 行** `import { logger } from '../utils/logger'` → 第 94 行 `const log = logger('TransferPage')` —— 创建带模块名前缀的日志实例。`logger()` 是一个高阶函数，返回的 `log` 对象在 `console.log/warn/error` 输出前自动加上 `[TransferPage]` 前缀

7. **第 95 行** `const loading = ref(false)` —— 创建响应式加载状态。`ref()` 返回 `{ value: false }` 的响应式代理。模板中可以直接写 `loading`（自动解包 `.value`），`v-loading="loading"` 绑定到 `el-card` 上，为 `true` 时卡片显示半透明遮罩 + 转圈动画

8. **第 96 行** `const submitting = ref(false)` —— 创建提交状态。模板第 79 行 `:loading="submitting"` 绑定到确认转账按钮上，为 `true` 时按钮禁用 + 转圈，防止用户连点发出重复请求

9. **第 97 行** `const formRef = ref(null)` —— 创建模板引用（Template Ref）。模板第 44 行 `<el-form ref="formRef">` 将该 ref 绑定到 el-form 组件实例。组件挂载后 `formRef.value` 指向 el-form 组件实例对象——可以调用其暴露的方法：`formRef.value.validate()`（触发表单校验）、`formRef.value.resetFields()`（重置表单字段为初始值）

10. **第 98 行** `const accountList = ref([])` —— 创建空数组的响应式引用。后续 `loadAccounts()` 中赋值 `accountList.value = data` 后，模板中 `v-for="acc in accountList"` 自动重新渲染 `<el-option>` 列表

11. **第 100-105 行** `const formData = reactive({...})` —— 创建表单数据响应式对象：

```javascript
const formData = reactive({
  fromAccountId: null,    // 转出账户 ID，初始 null（el-select 显示 placeholder）
  toAccountId: null,      // 转入账户 ID，初始 null
  amount: null,           // 转账金额，初始 null（el-input-number 显示空）
  note: ''                // 备注内容，初始空字符串
})
```

`reactive()` 基于 ES6 Proxy 实现深层响应式——修改 `formData.fromAccountId` 时，绑定了 `v-model="formData.fromAccountId"` 的 el-select 自动更新显示。与 `ref()` 的区别：`reactive()` 用于对象/数组，不需要 `.value` 访问；`ref()` 用于基本类型，需要 `.value` 访问

12. **第 107-127 行** `const formRules = {...}` —— 定义表单校验规则对象。`el-form` 的 `:rules="formRules"` 绑定此对象。每个 key（`fromAccountId` / `toAccountId` / `amount` / `note`）对应 `el-form-item` 的 `prop` 属性。每条规则是一个对象数组，Element Plus 按数组顺序依次校验：

```javascript
const formRules = {
  fromAccountId: [
    { required: true, message: '请选择转出账户', trigger: 'change' }
    // trigger: 'change' —— 用户切换选项时触发校验（而非 blur 失焦时）
  ],
  toAccountId: [
    { required: true, message: '请选择转入账户', trigger: 'change' },
    {
      validator: (rule, value, callback) => {
        // 自定义校验器——接收三个参数：rule（规则对象）、value（当前字段值）、callback（回调函数）
        if (value && value === formData.fromAccountId) {
          callback(new Error('转入账户不能与转出账户相同'))
          // callback 传入 Error 对象 → 校验失败 → el-form-item 显示红色错误文字
        } else {
          callback()  // callback 无参 → 校验通过
        }
      },
      trigger: 'change'
    }
  ],
  amount: [
    { required: true, message: '请输入转账金额', trigger: 'blur' },
    // trigger: 'blur' —— 输入框失去焦点时校验（而非每次输入都校验，体验更好）
    { type: 'number', min: MIN_TRANSACTION_AMOUNT, message: '转账金额必须大于0', trigger: 'blur' }
    // type: 'number' 对应 el-input-number 组件——Element Plus 自动转换值为数字类型
  ],
  note: [
    { max: MAX_NOTE_LENGTH, message: `备注长度不能超过${MAX_NOTE_LENGTH}`, trigger: 'blur' }
    // max: 200 —— 模板字符串动态拼接校验消息
  ]
}
```

**第 4 层 —— `<template>` 渲染（用户看到的界面如何生成）**：

`<template>` 中的 HTML 被 Vue 编译器编译为渲染函数（render function）。关键渲染细节：

- **第 42 行** `<el-card shadow="hover" class="transfer-card" v-loading="loading">` —— `v-loading` 指令是 Element Plus 的自定义指令。`loading` 为 `true` 时，指令在 el-card 的 DOM 上追加一个半透明遮罩层 + CSS 旋转动画。`loading` 为 `false` 时移除遮罩。这是声明式 loading——只需要改 `loading.value`，DOM 自动更新

- **第 44 行** `<el-form ref="formRef" :model="formData" :rules="formRules" label-width="90px">` —— `:model="formData"` 绑定整个表单的数据对象，`el-form` 内部通过 `model` prop 获取所有字段值。`label-width="90px"` 设置所有 `el-form-item` 的 label 宽度统一为 90px（"转出账户"四个中文字约 56px，90px 留有余量）

- **第 47-55 行 转出账户 `<el-select>`**：

```vue
<el-select v-model="formData.fromAccountId" placeholder="请选择转出账户"
           style="width: 100%"
           @change="() => formRef.validateField('toAccountId').catch(() => {})">
  <el-option
    v-for="acc in accountList"
    :key="acc.id"
    :label="acc.name"
    :value="acc.id"
    :disabled="acc.id === formData.toAccountId"
  />
</el-select>
```

- `v-model="formData.fromAccountId"` —— Vue 的双向绑定指令。等价于 `:model-value="formData.fromAccountId"` + `@update:model-value="val => formData.fromAccountId = val"`。用户选择一个账户 → el-select 触发 `update:model-value` 事件 → `formData.fromAccountId` 被更新 → 响应式系统通知所有依赖此值的组件重新渲染
- `v-for="acc in accountList"` —— 遍历 `accountList` 响应式数组（初始为空 `[]`），为每个账户生成一个 `<el-option>`。`:key="acc.id"` 给每个选项唯一标识——Vue 的 diff 算法依赖 key 做最小化 DOM 更新
- `:disabled="acc.id === formData.toAccountId"` —— 动态绑定 disabled 属性。表达式计算结果为 `true` 时该选项灰色不可选。**互斥约束**：已选为转入账户的选项在转出下拉中自动禁用，防止用户选同一个账户同时作为转出和转入
- `@change="() => formRef.validateField('toAccountId').catch(() => {})"` —— 转出账户变化时，手动触发转入账户字段的校验。场景：用户先选转入账户 A，再选转出账户也为 A——此时转入账户的自定义校验器需要重新判断 `value === formData.fromAccountId`。不手动触发 → 转入账户的校验状态残留（之前通过校验显示绿色，但实际现在应该报错了）。`.catch(() => {})` 吞掉 validateField 失败时的 rejection

- **第 59-67 行 转入账户 `<el-select>`**：逻辑同转出账户，`:disabled="acc.id === formData.fromAccountId"` 方向相反。没有 `@change` 手动触发表单校验——因为转入账户变化不影响转出账户的校验结果

- **第 69-71 行 金额 `<el-input-number>`**：

```vue
<el-input-number v-model="formData.amount" :precision="2"
                 :min="MIN_TRANSACTION_AMOUNT" :step="AMOUNT_STEP_ROUGH"
                 style="width: 100%" />
```

- `:precision="2"` —— 保留 2 位小数（对应数据库 DECIMAL(12,2)）。用户输入 `100` 显示为 `100.00`
- `:min="MIN_TRANSACTION_AMOUNT"`（0.01）—— 最小值为 1 分钱，Element Plus 内部控制加减按钮不会低于此值

- **第 79 行 确认转账按钮**：

```vue
<el-button type="primary" :loading="submitting"
           @click="handleSubmit" style="width: 100%;">
  确认转账
</el-button>
```

- `type="primary"` —— Element Plus 主题蓝色按钮
- `:loading="submitting"` —— 动态绑定 loading 状态。`submitting` 为 `true` 时按钮内部显示转圈动画 + `disabled` 变为 `true`（按钮灰色不可点击）——这是 Element Plus 的内置行为，不需要手动设置 `:disabled`
- `@click="handleSubmit"` —— 点击事件绑定 `handleSubmit` 函数（不是 `handleSubmit()`，传的是函数引用而非调用结果）

**第 5 层 —— `onMounted()` 自动加载账户列表**：

```javascript
// TransferPage.vue 第 201-203 行
onMounted(async () => {
  await loadAccounts()
})
```

- `onMounted()` 是 Vue 3 的生命周期钩子——在组件被挂载到 DOM 后执行。此时模板已渲染完毕，`formRef.value` 已指向 el-form 组件实例
- `async` 回调：`loadAccounts()` 内部有 `await`，不加 async 不能使用 await
- 执行时机：组件挂载 → `onMounted` 回调被推入微任务队列 → 当前同步代码执行完毕 → 微任务执行 → `loadAccounts()` 开始执行

**`loadAccounts()` 函数（第 130-142 行）**：

```javascript
async function loadAccounts() {
  loading.value = true                    // ① 开启骨架屏
  try {
    const data = await getAccountList()   // ② 调 API
    accountList.value = data || []        // ③ 响应式赋值 → UI 自动更新
  } catch (e) {
    log.warn('加载账户列表失败:', e)
    ElMessage.warning('加载账户列表失败，请刷新重试')
  } finally {
    loading.value = false                 // ④ 关闭骨架屏——保证一定执行
  }
}
```

**调用链展开**：

1. `loading.value = true` → Vue 响应式系统检测到 `loading` 变化 → 通知 `<el-card v-loading="loading">` → Element Plus 的 v-loading 指令更新 DOM → 卡片上出现半透明遮罩 + 旋转动画

2. `await getAccountList()` → 进入 `api/account.js` 第 47-51 行：

```javascript
export function getAccountList() {
  return request.get('/account')
}
```

`request` 是 `api/request.js` 第 58 行创建的 axios 实例：

#### 节点 ⑤：Vite Dev Server Proxy —— 解决跨域

**网络传输**：HTTP POST 请求通过本地回环 TCP 连接（127.0.0.1:5173 → 127.0.0.1:8080），请求体为 JSON 格式。

### 阶段二：后端——请求进入到数据库写入（7 个节点）

#### 节点 ⑥：CorsFilter —— Filter 层 CORS 处理

请求到达 `localhost:8080`，首先进入 **Servlet Filter 链**。Filter 是 Java Servlet 规范定义的组件，在 Spring MVC 的 DispatcherServlet 之前执行。

`config/CorsConfig.java` 第 103 行的 `corsFilter()` 方法被 `@Bean` 标注——Spring Boot 启动时调用此方法，将返回的 `CorsFilter` 实例注册到 Servlet 容器的 Filter 链中。`CorsFilter` 是 Spring 框架提供的实现，继承自 `OncePerRequestFilter`（保证每个请求只经过一次）。

**`doFilter()` 方法内部逻辑**（Spring 框架源码，非本项目代码）：
1. 检查请求的 `Origin` 头是否在 CORS 配置的 `allowedOrigins` 列表中
2. 如果是 **OPTIONS 预检请求**：设置 `Access-Control-Allow-Origin`、`Access-Control-Allow-Methods`、`Access-Control-Allow-Headers`、`Access-Control-Max-Age` 等响应头 → 返回 HTTP 200 → **不调用 `chain.doFilter()`**（预检请求不需要到达 Controller）
3. 如果是**普通请求（如本次 POST）**：设置 CORS 响应头 → 调用 `chain.doFilter(request, response)` → 将请求传递给 Filter 链中的下一个 Filter → 最终到达 Spring 的 `DispatcherServlet`

**为什么 CORS 必须放在 Filter 层（核心设计决策）**：浏览器的 CORS 协议规定——在发送带 `Authorization` 头的跨域 POST 请求之前，浏览器先发一个 `OPTIONS` 预检请求询问服务器是否允许。这个 OPTIONS 请求**不带 `Authorization` 头**（CORS 规范如此——预检的目的是确认服务器允许跨域，此时不应发送用户凭证）。如果 CORS 处理放在 Interceptor 层（Spring MVC 层），`LoginInterceptor.preHandle()` 在 OPTIONS 请求到达时发现没有 Authorization 头 → `writeError(response)` 返回 401 → `return false` 中断请求 → 浏览器收到 401 的预检响应 → 认为服务器拒绝了跨域 → 不发送正式的 POST 请求 → 前端所有带 token 的请求全部失败。Filter 在 HTTP 协议层处理——比 Spring MVC 更早执行——OPTIONS 预检在 Filter 层就被拦截并返回 200，不会到达 Interceptor。

#### 节点 ⑦：DispatcherServlet —— URL 映射到 Controller 方法

通过 Filter 链后，请求进入 Spring MVC 的核心——**`DispatcherServlet`**。这是 Spring MVC 的前端控制器（Front Controller），所有 HTTP 请求都由它统一分发。

**`DispatcherServlet` 的处理流程**（Spring 框架内部）：

1. **`getHandler(request)`**：遍历所有注册的 `HandlerMapping`。`RequestMappingHandlerMapping` 根据 `@RequestMapping` 注解找到匹配的处理器——匹配到 `TransactionController` 类上的 `@RequestMapping("/api/v1/transaction")` + `transfer()` 方法上的 `@PostMapping("/transfer")`。返回 `HandlerExecutionChain`（包含目标方法 + 拦截器链）

2. **`getHandlerAdapter(handler)`**：获取能执行该 handler 的适配器。`RequestMappingHandlerAdapter` 负责执行 `@RequestMapping` 标注的方法——处理参数解析（`@RequestBody`、`@Valid`、`HttpServletRequest`）和返回值处理（`@ResponseBody` → Jackson 序列化）

3. **执行拦截器链的 `preHandle()`**：按注册顺序依次调用。本项目注册了 `LoginInterceptor` 和 `AdminInterceptor`（`WebMvcConfig.java` 第 83、101 行）。先执行 `LoginInterceptor.preHandle()`

4. **调用目标方法**：拦截器全部返回 `true` 后，`HandlerAdapter` 通过反射调用 `TransactionController.transfer()`

5. **执行拦截器链的 `postHandle()`**（在视图渲染前，本项目前后端分离不涉及视图）

6. **执行拦截器链的 `afterCompletion()`**（在请求完成后，用于资源清理）

#### 节点 ⑧：LoginInterceptor —— Interceptor 层 JWT 校验

`interceptor/LoginInterceptor.java` 实现了 `HandlerInterceptor` 接口——这是 Spring MVC 层面的拦截器。第 92 行 `preHandle()` 方法在 Controller 方法执行之前被 `DispatcherServlet` 调用。

```java
@Override
public boolean preHandle(HttpServletRequest request, HttpServletResponse response,
                         Object handler) throws Exception {
```

三个参数：`request`（浏览器原始 HTTP 请求，由 Tomcat 构造）、`response`（HTTP 响应对象，用于写错误 JSON）、`handler`（匹配到的目标 Controller 方法，本方法未使用）。

**第 1 步（第 96 行）—— OPTIONS 预检放行**：

```java

    return true;
}
```

- `equalsIgnoreCase` 大小写不敏感比较——防御浏览器发送 `"options"` / `"Options"` 等变体
- `return true` 直接放行。CorsFilter 已处理 CORS 响应头，预检不需要进一步处理

**第 2 步（第 100-107 行）—— 提取 Authorization 请求头**：

```java
String authHeader = request.getHeader("Authorization");
if (authHeader == null || !authHeader.startsWith("Bearer ")) {
    writeError(response);   // → HTTP 200 + body {"code":401,"message":"未登录","data":null}
    return false;           // ★ 中断请求链——后续拦截器和 Controller 都不执行
}
```

- `request.getHeader("Authorization")` —— 从 HTTP 请求头中读取 Authorization 字段。前端 `request.js` 请求拦截器写入的值就在这里被读到
- `authHeader == null` —— 请求头中根本没有 Authorization 字段。短路求值：先判 null 防止后续 `startsWith` 抛 NPE
- `!authHeader.startsWith("Bearer ")` —— 有 Authorization 字段但格式不是 `"Bearer xxx"`。防止攻击者用 `"Basic base64encoded"` 等其他认证方案绕过 JWT 校验
- `writeError(response)` —— 写入统一 401 错误响应（详见下方 writeError 分析）
- `return false` —— 中断 Spring MVC 请求处理链。后续的 `AdminInterceptor.preHandle()` 和 `TransactionController.transfer()` 都不会执行

**第 3 步（第 110-118 行）—— 截取 token + HMAC 验签**：

```java
String token = authHeader.substring("Bearer ".length()).trim();
JwtUtils.JwtPayload payload = JwtUtils.parseTokenPayload(token);
if (payload == null) {
    writeError(response);
    return false;
}
```

- `"Bearer ".length()` = 7（含空格）。`substring(7)` 从第 8 个字符开始截取，得到纯 token 字符串。`.trim()` 去除意外首尾空格（如前端代码拼写失误多加了空格）
- `JwtUtils.parseTokenPayload(token)` —— 一次方法调用完成两件事：① HMAC-SHA256 验证签名（判断 token 是否被篡改）② Base64 解码 payload 提取 `userId` 和 `role` 字段。返回值 `JwtPayload` 是一个内部静态类，包含 `userId`（Long）和 `role`（Integer）两个字段。**为什么合并为一次调用**：旧方案 `parseToken()` 取 userId + `parseRole()` 取 role → 每次调用做一次完整 HMAC 运算 → 两次 HMAC。合并后一次 HMAC 同时提取两个字段 → **减少 50% 密码学运算**（HMAC-SHA256 是 CPU 密集型操作，高并发下节省显著）
- `payload == null` → 三种情况：token 过期（`exp` 字段时间戳 < 当前时间）、签名被篡改（攻击者修改 payload 后 HMAC 不匹配）、格式错误（不是合法的三段式 JWT）。任一种都返回 null → 返回 401

**第 4 步（第 124-132 行）—— 存入 request 属性**：

```java
request.setAttribute("userId", payload.getUserId());
request.setAttribute("role",
    payload.getRole() != null ? payload.getRole() : UserRole.NORMAL.getValue());
return true;
```

- `request.setAttribute("userId", ...)` —— 将 userId 存入当前 `HttpServletRequest` 对象的属性 Map。下游 Controller 通过 `LoginInterceptor.getUserId(request)` 静态方法取出——`request.getAttribute("userId")` → Long 类型转换
- `request.setAttribute("role", ...)` —— 存入角色。null 时默认为普通用户（`UserRole.NORMAL.getValue()` = 0）——遵循最小权限原则：宁可误降为普通用户（能看的少），绝不误升为管理员（能改系统设置、看所有用户数据）
- **为什么用 `request.setAttribute` 而不是 `ThreadLocal`**：Tomcat 使用线程池处理请求——线程 A 处理完请求 1 后归还线程池，线程 B 取到同一线程处理请求 2。如果线程 A 在 ThreadLocal 中存了 userId=1 但异常退出没清理 → 线程 B 复用时读到 userId=1 → 用户 B 被误认为是用户 A → **串号安全漏洞**。`request.setAttribute` 每个请求有独立的 `HttpServletRequest` 对象——即使线程复用，不同请求的 request 对象不同——天然隔离

**`writeError()` 方法分析（第 154-175 行）**：

```java
private void writeError(HttpServletResponse response) throws Exception {
    if (response.isCommitted()) {
        log.warn("LoginInterceptor: 响应已 committed,跳过 401 错误写入");
        return;
    }
    response.setStatus(HTTP_OK);  // HTTP 200——不是 401
    response.setContentType("application/json;charset=UTF-8");
    var writer = response.getWriter();
    writer.write(objectMapper.writeValueAsString(
        Result.error(ErrorCode.UNAUTHORIZED.getCode(), ErrorCode.UNAUTHORIZED.getMsg())
    ));
    writer.flush();
}
```

- `response.isCommitted()` —— 防御性检查：如果前置 Filter 已经提交了响应（写入了响应头），再修改会抛 `IllegalStateException`
- **`response.setStatus(HTTP_OK)` —— 为什么 HTTP 状态码写 200 而不是 401**：这是**全栈 body-code-first 约定**。前端 axios 响应拦截器（`request.js` 第 136 行）统一检查 `res.data.code`（body 里的业务状态码）而非 HTTP status code。如果后端返回 HTTP 401，axios 走 error 回调而非 success 回调——后端返回的中文错误消息"未登录"会被 `HttpError` 对象吞掉——前端只能拿到英文默认消息 `"Request failed with status code 401"`。统一用 HTTP 200 + body.code 区分业务状态——好处是前端只需一段 if-else，所有页面自动受益
- `writer.flush()` —— 强制将缓冲区数据立即发送到 TCP。不 flush 可能滞留缓冲区 → 客户端迟迟收不到响应 → 触发 axios timeout

- `addPathPatterns("/api/v1/**")` —— `**` 是 Ant 风格通配符，匹配任意多级路径（如 `/api/v1/user/login`、`/api/v1/transaction/transfer?id=1`、`/api/v1/admin/user/5`）。不能用 `/*`——`/*` 只匹配一级路径

#### 节点 ⑨：TransactionController —— 参数校验 + 转发 Service

`LoginInterceptor.preHandle()` 返回 `true` 后，`DispatcherServlet` 调用 `TransactionController.transfer()`（`controller/TransactionController.java` 第 197-209 行）：

```java
@PostMapping("/transfer")
public Result<TransferDTO> transfer(@Valid @RequestBody TransferRequest request,
                                    HttpServletRequest httpRequest) {
    Long userId = LoginInterceptor.getUserId(httpRequest);
    TransferDTO transfer = transactionService.transfer(userId, request);
    return Result.success(transfer, "转账成功");
}
```

**三件事，每件一行**：

1. **`@Valid @RequestBody TransferRequest request`** —— Spring 的参数解析器（`RequestResponseBodyMethodProcessor`）做两件事：
   - `@RequestBody`：读取 HTTP 请求体的 JSON 字符串 → Jackson `ObjectMapper.readValue()` 反序列化为 `TransferRequest` Java 对象。`TransferRequest` 是一个 DTO（Data Transfer Object），定义在 `entity/dto/TransferRequest.java`，包含 `fromAccountId`（Long, @NotNull）、`toAccountId`（Long, @NotNull）、`amount`（BigDecimal, @NotNull）、`note`（String）
   - `@Valid`：触发 JSR-303 Bean Validation。Spring 调用 `jakarta.validation.Validator.validate(request)`，检查每个字段上的约束注解（`@NotNull`、`@DecimalMin` 等）。如果校验失败 → Spring 抛出 `MethodArgumentNotValidException` → `GlobalExceptionHandler.handleMethodArgumentNotValid()`（`@ExceptionHandler` 标注的方法）捕获 → 返回 `Result.error(400, "参数校验失败: 具体字段和原因")`。**Controller 方法体不执行**

2. **`LoginInterceptor.getUserId(httpRequest)`** —— 静态辅助方法（LoginInterceptor.java 第 139-146 行），内部 `request.getAttribute("userId")` → Long 类型转换。如果为 null（非正常调用路径——请求没经过 LoginInterceptor），抛 `BusinessException` 防 NPE

3. **`transactionService.transfer(userId, request)`** —— 转发到 Service 层。Controller 不写业务逻辑、不 try-catch。`transactionService` 是 `TransactionService` 接口的代理对象（Spring 注入的实际类型是 `TransactionServiceImpl`）

#### 节点 ⑩ ★：TransactionServiceImpl.transfer() —— 转账核心业务逻辑

`service/impl/TransactionServiceImpl.java` 第 389-521 行。**这是自选的后端核心方法，逐行讲解见第二部分 §2.2 ⑤。** 这里概述 7 步：

| 步骤 | 行号 | 做什么 | 调用了什么 | 产生的 SQL |
|:---:|:---:|------|------|------|
| 1 | 409-418 | fail-fast 校验：金额非空 + >0 + 不能自己转自己。**全部放在加锁之前** | `BigDecimal.compareTo()`, `Long.equals()` | 无（校验不过不查数据库） |
| 2 | 426-432 | 死锁预防：`Math.min/max` 按账户 ID 升序排序 → `selectByIdForUpdate()` 加锁 | `AccountMapper.selectByIdForUpdate()` | `SELECT * FROM account WHERE id=? FOR UPDATE` × 2 |
| 3 | 435-441 | 账户归属 + 状态校验 | `Objects.equals()` 比较 Integer 值（不用 `==`） | 无 |
| 4 | 450-457 | BigDecimal 余额计算：初始余额 + 收入总和 - 支出总和 → 比较 | `TransactionMapper.selectAccountIncome/Expense()` | `SELECT COALESCE(SUM(amount),0) FROM transaction WHERE ... AND type=?` × 2 |
| 5 | 461-469 | `LambdaQueryWrapper` 查"其他"支出分类（加 `type=1` 过滤防 `TooManyResultsException`） | `CategoryMapper.selectOne()` | `SELECT * FROM category WHERE name='其他' AND type=1` |
| 6 | 479-513 | UUID 关联 + 复式记账：INSERT 转出(type=2=支出) + INSERT 转入(type=1=收入) | `UUID.randomUUID()`, `TransactionMapper.insert()` × 2 | `INSERT INTO transaction (...) VALUES (...)` × 2 |
| 7 | 516-520 | 组装 TransferDTO 返回 | `toDTO()` 私有方法（Entity→DTO，附带 accountName/categoryName） | 无 |

**关键设计**：第 389 行 `@Transactional` —— Spring 声明式事务。此方法内所有数据库操作都在一个事务中。方法正常返回 → Spring AOP 通知 `PlatformTransactionManager.commit()` → MySQL `COMMIT`。方法内任何位置抛 `RuntimeException`（包括 `BusinessException`）→ Spring AOP 通知 `PlatformTransactionManager.rollback()` → MySQL `ROLLBACK`。转账三步（查余额 + INSERT 转出 + INSERT 转入）不可分割——如果 INSERT 转入时数据库崩溃，转出也自动回滚。

#### 节点 ⑪：Mapper 层 —— SQL 生成与执行

Service 调用的 Mapper 是 MyBatis-Plus 在启动时通过 JDK 动态代理生成的实现类。每个 Mapper 方法对应一条 SQL：

| Mapper 接口方法 | 注解/来源 | 生成的 SQL（PreparedStatement 参数化） | 作用 |
|------|------|------|------|
| `AccountMapper.selectByIdForUpdate(id)` | `@Select`（AccountMapper.java 第 36-37 行） | `SELECT * FROM account WHERE id=? FOR UPDATE` | InnoDB 行级排他锁（X Lock），阻止其他事务修改此行直到当前事务 COMMIT/ROLLBACK |
| `TransactionMapper.selectAccountIncome(userId, accountId)` | MyBatis XML（`TransactionMapper.xml`） | `SELECT COALESCE(SUM(amount),0) FROM transaction WHERE user_id=? AND account_id=? AND type=1` | 统计该账户所有收入（type=1）的金额总和，COALESCE 保证无数据时返回 0 |
| `TransactionMapper.selectAccountExpense(userId, accountId)` | MyBatis XML | `SELECT COALESCE(SUM(amount),0) FROM transaction WHERE user_id=? AND account_id=? AND type=2` | 统计该账户所有支出（type=2）的金额总和 |
| `TransactionMapper.insert(entity)` | MyBatis-Plus `BaseMapper` 内置 | `INSERT INTO transaction (user_id,account_id,category_id,type,amount,note,time,transfer_id,create_time,update_time) VALUES (?,?,?,?,?,?,?,?,?,?)` | 插入转出记录，MyBatis-Plus 自动根据实体字段生成 INSERT 语句 |

所有 SQL 使用 `PreparedStatement` 参数化查询——`?` 占位符 + 参数绑定——**防止 SQL 注入**。MyBatis-Plus 的 `LambdaQueryWrapper` 也是参数化查询——`eq(Category::getName, "其他")` 生成的 SQL 是 `WHERE name = ?`（参数化），而非 `WHERE name = '其他'`（拼接）。

#### 节点 ⑫：MySQL InnoDB —— 事务执行

MySQL InnoDB 引擎在 `@@tx_isolation = REPEATABLE-READ`（MySQL 默认隔离级别）下执行：

```
1. BEGIN  — @Transactional 触发，Spring 通过 DataSourceTransactionManager 向 MySQL 发送 START TRANSACTION

2. SELECT * FROM account WHERE id=1 FOR UPDATE;  — 获取 id=1 的行级排他锁（X Lock）
   - InnoDB 在聚簇索引（主键索引）上对 id=1 的记录加 X Lock
   - 其他事务的 SELECT ... FOR UPDATE / UPDATE / DELETE WHERE id=1 被阻塞，等待此锁释放
   - 其他事务的普通 SELECT（不加锁读）在 REPEATABLE-READ 下读的是 undo log 中的快照版本——不受影响

3. SELECT * FROM account WHERE id=2 FOR UPDATE;  — 获取 id=2 的行级排他锁
   - 因为步骤 2 按 ID 升序加锁（先锁小 ID 再锁大 ID），所有并发转账的锁顺序一致
   - 破坏了死锁四条件之一的"循环等待" → 死锁不可能发生

4. SELECT COALESCE(SUM(amount),0) FROM transaction
   WHERE user_id=1 AND account_id=1 AND type=1;  — 收入合计
   - COALESCE 保证 NULL → 0，不会返回 NULL

5. SELECT COALESCE(SUM(amount),0) FROM transaction
   WHERE user_id=1 AND account_id=1 AND type=2;  — 支出合计

   [Java 层 BigDecimal 计算: initialBalance + income - expense >= amount?]

6. INSERT INTO transaction (...) VALUES (..., type=2, ...);  — 转出记录
   - 写入 InnoDB 的 redo log（持久化保证）+ undo log（回滚保证）
   - 数据页在 buffer pool 中修改，标记为脏页，尚未刷盘

7. INSERT INTO transaction (...) VALUES (..., type=1, ...);  — 转入记录

8. COMMIT  — Spring AOP 在 transfer() 正常返回后调用 commit
   - redo log 写入磁盘（两阶段提交：prepare → commit）
   - 锁释放
   - 脏页由后台线程异步刷盘（checkpoint 机制）
```

**如果任一步失败**（如步骤 7 INSERT 时发生死锁/磁盘满/主键冲突）→ MySQL 自动 ROLLBACK → undo log 回放撤销步骤 2-6 的所有修改 → 锁释放 → Spring 收到异常 → 抛 `DataAccessException` → 被 `@Transactional` 包装 → `BusinessException` 等 RuntimeException 触发回滚 → `GlobalExceptionHandler` 处理

### 阶段三：响应返回——后端响应到前端更新（4 个节点）

#### 节点 ⑬：Result.success() 统一响应包装

Controller 第 208 行：

```java
return Result.success(transfer, "转账成功");
```

调 `common/Result.java` 第 99-101 行的静态工厂方法：

```java
public static <T> Result<T> success(T data, String message) {
    return new Result<>(CODE_SUCCESS, message, data);
    // CODE_SUCCESS = 200
}
```

构造出的对象：`Result(code=200, message="转账成功", data=TransferDTO{...})`。`Result` 类上有 `@Data`（Lombok）——自动生成 getter/setter/toString/equals/hashCode。`TransferDTO` 内的 `BigDecimal` 字段（amount）、`LocalDateTime` 字段（time/createTime/updateTime）由 Jackson 后续序列化。

#### 节点 ⑭：Jackson 序列化 → HTTP 响应

Spring Boot 的 `@RestController` = `@Controller` + `@ResponseBody`。`@ResponseBody` 触发 `HttpMessageConverter` 链——找到 `MappingJackson2HttpMessageConverter`（处理 `application/json`）→ 调用 Jackson `ObjectMapper.writeValueAsString(result)`：

- `Integer code` → JSON number `200`
- `String message` → JSON string `"转账成功"`
- `TransferDTO data` → JSON object `{transferId, outRecord:{...}, inRecord:{...}}`
- `BigDecimal amount` → JSON number `100.00`
- `LocalDateTime time` → JSON string `"2026-05-29T10:30:00"`（ISO 8601 格式）

序列化后的 JSON 写入 HTTP 响应体。Tomcat 发送 HTTP 响应：

```
HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8

{"code":200,"message":"转账成功","data":{...}}
```

TCP 传输：`:8080` → `:5173`（Vite Dev Server）→ 浏览器。

#### 节点 ⑮：`request.js` 响应拦截器 —— 三段处理

浏览器收到响应 → axios 接收 → 触发**响应拦截器的成功回调**（`request.js` 第 136-166 行）：

```javascript
request.interceptors.response.use(res => {
  const { code, message, data } = res.data || {}
  if (code === 200) {
    return data
    // ★ 剥壳——只返回 data。TransferPage 的 await transfer(formData) 拿到的是纯 TransferDTO
  } else if (code === 401) {
    // 未登录处理：isRedirecting 去重 → userStore.clearUser() → router.push('/login')
    // ...
  } else {
    // 业务错误：ElMessage.error + Promise.reject（标记 noRetry=true 防无效重试）
    // ...
  }
}, async error => {
  // 网络异常/超时/5xx → 指数退避重试（1s→2s→4s + 10-30% 随机抖动，最多 3 次）
  // 4xx / 业务错误（noRetry=true）→ 不重试，直接弹错误提示
})
```

**成功分支**：`code === 200` → `return data`。TransferPage 的 `await transfer(formData)` resolve，拿到的是 `{transferId: "uuid...", outRecord: {...}, inRecord: {...}}`。外层的 `code: 200` 和 `message: "转账成功"` 被丢弃——业务代码只需要数据。

**为什么三段处理放在拦截器中**：10 个 API 模块 + 11 个页面统一受益——不需要每个页面重复写 `if (res.data.code === 200) { ... } else if (...) { ... }`。代码集中在一个地方，修改逻辑只改一处。

#### 节点 ⑯：TransferPage.vue 接收响应并更新页面

`await transfer(formData)` resolve，`handleSubmit()` 第 172 行之后的代码继续执行（仍在 try 块中）：

```javascript
ElMessage.success('转账成功')       // ① 绿色成功提示
formRef.value.resetFields()        // ② 清空表单
await loadAccounts()               // ③ 重新加载账户列表
```

1. **`ElMessage.success('转账成功')`** → Element Plus 在页面顶部弹出绿色消息条，3 秒后自动消失。`ElMessage` 是命令式 API——不依赖模板中的组件标签，直接调用函数即可

2. **`formRef.value.resetFields()`** → el-form 组件方法——将所有 `el-form-item` 绑定的字段重置为 `formData` 中定义的初始值：
   - `formData.fromAccountId` → `null`（el-select 显示 placeholder "请选择转出账户"）
   - `formData.toAccountId` → `null`
   - `formData.amount` → `null`（el-input-number 显示空）
   - `formData.note` → `''`（el-input 显示空）

3. **`await loadAccounts()`** → 重新执行 `getAccountList()` → `GET /api/v1/account` → 后端查询最新的账户余额 → 返回更新后的账户列表。**为什么必须刷新**：转账后转出账户的余额 = 原余额 - 100，转入账户的余额 = 原余额 + 100——两个账户的余额都变了。不刷新 → 用户再次转账时下拉框显示的余额是旧的 → 用户可能以为还有足够的钱 → 再次转出但实际余额已不足。

**`catch` 块（第 180-191 行）**——只处理网络层异常：

```javascript
} catch (e) {
  log.error('转账异常:', e)
  if (e.code === 'ERR_NETWORK' || e.code === 'ECONNABORTED') {
    ElMessage.error('网络异常，转账失败')
  }
}
```

- `ERR_NETWORK`：网络断开/DNS 解析失败/服务器宕机——请求根本没到达服务器
- `ECONNABORTED`：axios 的 10 秒 timeout 触发——服务器响应太慢
- **为什么只处理这两种**：业务异常（"余额不足"、"账户不存在"等）已被 axios 响应拦截器统一 `ElMessage.error` 处理——如果这里也弹提示，用户会看到**两条重复的错误消息**

**`finally` 块（第 192-196 行）**：

```javascript
} finally {
  submitting.value = false
}
```

`submitting.value = false` → Vue 响应式更新 → `:loading="submitting"` 重新求值为 false → el-button 移除转圈动画 + 启用按钮。**为什么放 finally**：JS 的 `finally` 保证无论 try 成功、catch 处理了异常、还是 catch 中又抛了新异常——都一定执行。如果 `submitting.value = false` 放在 try 末尾——catch 中抛了新异常 → 这行不执行 → 按钮永远转圈 → 用户无法再次转账 → 只能刷新页面

**最终用户看到**：✅ 绿色"转账成功"提示 → 表单已清空 → 两个下拉框中账户余额已更新 → 按钮恢复可点击 → 一次完整的转账操作结束。

### 模块边界规则

**前端 4 条**：
- views 页面不直接调 axios → 必须走 `api/` 封装
- `api/` 不操作 localStorage → 用户信息由 `stores/` 统一管理
- 父子通信只用 props/emit → 不滥用 Pinia
- 页面文件 `*Page.vue` → 可复用组件不加 Page 后缀

**后端 4 条**：
- Controller 只校验 + 转发 → 不写业务逻辑
- Service 只调 Mapper → 不手写 SQL（复杂多表 join 走 XML 例外）
- Controller 不 try-catch → 异常全由 `GlobalExceptionHandler` 统一处理
- 接口传参返参只用 DTO → 不直接返回 Entity（防字段泄漏 + 解耦数据库变更）

#### ① CorsConfig.java（2分钟）

**讲稿**："这是 CorsConfig.java，CORS 跨域配置。`@Bean` 让 Spring 把返回的 `CorsFilter` 注册为全局过滤器。**为什么 CORS 必须放 Filter 层**：浏览器的 OPTIONS 预检请求不带 Authorization 头——如果放 Interceptor 层，LoginInterceptor 发现没 token 就返回 401，预检失败，所有跨域请求都被浏览器拦截。Filter 比 Interceptor 更早执行——在 HTTP 协议层就把预检处理完了。"

#### ② LoginInterceptor.java（3分钟）

**讲稿**："LoginInterceptor 校验 JWT token，4 步：
① `"OPTIONS".equalsIgnoreCase(request.getMethod())` → `return true` 放行预检；
② `request.getHeader("Authorization")` 提取请求头 → `startsWith("Bearer ")` 检查 RFC 6750 标准格式；

**为什么两处都用 `Objects.equals(a, b)` 而非 `a == b` 或 `a.equals(b)`**：

| 写法 | 问题 |
|------|------|
| `a == b` | Integer 在 -128~127 有缓存，范围内 `==` 比较同一对象 OK。超出范围（如 userId=128）`==` 比较引用地址——**值相同的两个 Integer 可能返回 false**——隐蔽 bug |

## 常见追问 & 应对

| 老师可能问 | 怎么答 |
|------|------|
| "为什么选转账而不是登录？" | "登录只做查表+BCrypt+签发 token，3 步太简单。转账涉及死锁预防、悲观锁、BigDecimal 精度、事务原子性——7 个知识点。" |
| "Filter 和 Interceptor 区别？" | "Filter 是 Servlet 规范，最早执行——CORS 放 Filter。Interceptor 是 Spring MVC 的，能依赖注入——JWT 校验放 Interceptor。" |
| "为什么 FOR UPDATE 不用乐观锁？" | "转账写冲突高发——两人同时转出。乐观锁冲突后重试整个方法，悲观锁一次锁住效率更高。" |
| "BigDecimal 为什么不用 double？" | "double 二进制浮点——0.1+0.2=0.30000000000000004。财务 1 分误差=生产事故。" |
| "ThreadLocal vs setAttribute？" | "ThreadLocal 线程级——Tomcat 线程池复用→没清干净→串号漏洞。setAttribute 每请求独立对象天然隔离。" |
| "body-code-first 是什么？" | "HTTP 始终 200，body.code 区分业务状态。前端拦截器只写一段 if-else，所有页面受益。" |
| "死锁预防具体怎么做？" | "`Math.min(from,to)` 取小先锁，`Math.max` 取大后锁。所有线程锁顺序一致→打破循环等待→死锁不可能发生。操作系统课在数据库层的落地。" |
| "复式记账为什么？" | "一出一入两条记录共享 transferId——资金从哪来到哪去完全可追溯。金额完全相等——转账不创造也不消灭金钱。" |

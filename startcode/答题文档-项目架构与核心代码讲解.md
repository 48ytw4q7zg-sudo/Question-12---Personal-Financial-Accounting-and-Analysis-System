#### 后端鉴权：CorsFilter → LoginInterceptor → Controller

**"请求到达 8080 端口，先经过 Servlet Filter 层。"** → Ctrl+P `CorsConfig.java`。

**"为什么 CORS 放 Filter 层？浏览器在发带 Authorization 头的跨域 POST 之前会先发一个 OPTIONS 预检——这个预检请求不带 Authorization 头。如果 CORS 放 Interceptor 层，LoginInterceptor 看到没 token 就返回 401，浏览器认为跨域被拒绝，正式的 POST 根本不会发出。Filter 比 Interceptor 更早执行——OPTIONS 在 Filter 层就处理完了。"**

**"接下来是 JWT 校验。"** → Ctrl+P `LoginInterceptor.java`，第 92 行 `preHandle()`，4 步：

1. **第 96 行** — `"OPTIONS".equalsIgnoreCase(...)` → `return true` 放行预检
2. **第 100-107 行** — 提取 `Authorization` 头 → 检查 `startsWith("Bearer ")`（RFC 6750 标准格式）
3. **第 114 行** — `JwtUtils.parseTokenPayload(token)`：**一次 HMAC 验签同时提取 userId 和 role**。旧方案调两次各做一次完整 HMAC 运算，合并后减少 50% 密码学运算
4. **第 124-131 行** — `request.setAttribute("userId", ...)` + `request.setAttribute("role", ...)`

**"为什么用 request.setAttribute 不用 ThreadLocal？Tomcat 线程池复用线程——线程 A 处理完请求，ThreadLocal 没清干净——线程 B 复用时读到线程 A 的用户 ID——这就是串号漏洞。setAttribute 每个请求有独立的 HttpServletRequest 对象，天然隔离。"**

**"token 无效时调 writeError()——返回 HTTP 200 + body.code=401。为什么 HTTP 状态码写 200 不写 401？前端 axios 响应拦截器检查的是 `res.data.code` 而不是 HTTP status。如果后端返回 HTTP 401，axios 走 error 回调，后端的中文错误消息被吞掉，前端只能拿到英文的 'Request failed with status code 401'。全系统统一 body-code-first——HTTP 始终 200，body.code 区分业务状态。"**

**"拦截器在哪里注册？"** → Ctrl+P `WebMvcConfig.java`，第 78 行 `addInterceptors()`：

- `LoginInterceptor` 拦截 `/api/v1/**`，白名单放行 login / register / health / category
- `AdminInterceptor` 只拦 `/api/v1/admin/**`，读 LoginInterceptor 存入的 role

**"执行顺序不能反——先 LoginInterceptor 存 role，再 AdminInterceptor 读 role。反了 AdminInterceptor 读到 null，所有管理员请求被误判。"**

**"校验通过后到 Controller。"** → Ctrl+P `TransactionController.java`，第 196 行：

```java
@PostMapping("/transfer")
public Result<TransferDTO> transfer(@Valid @RequestBody TransferRequest request,
                                    HttpServletRequest httpRequest) {
    Long userId = LoginInterceptor.getUserId(httpRequest);
    TransferDTO transfer = transactionService.transfer(userId, request);
    return Result.success(transfer, "转账成功");
}
```

**"三件事，每件一行：① @Valid @RequestBody——Jackson 反序列化 JSON 并校验；② getUserId——从 JWT 取 userId，不是从请求体取的——防止攻击者伪造 userId 转走别人的钱；③ 转发 Service。Controller 不写业务逻辑、不 try-catch。"**

#### 核心业务：TransactionServiceImpl.transfer() → Mapper → MySQL

**"核心来了。"** → Ctrl+P `TransactionServiceImpl.java`，滚到第 388 行。7 步：

| 步骤  | 行号      | 做什么                                                         | 关键设计决策                                                                                                                                                            |
|:---:|:-------:| ----------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ①   | 409-418 | 校验金额非空 + >0 + 不能自己转自己                                       | **fail-fast**：放加锁前——锁是最贵的资源，校验不过不占锁                                                                                                                               |
| ②   | 426-429 | `Math.min/max` 按 ID 升序 → `selectByIdForUpdate()`            | **死锁预防**：线程1转1→5先锁1再锁5，线程2转5→1同样是 `min(5,1)=1` 先锁1再锁5——锁顺序一致 → 打破循环等待 → 死锁不可能发生。`SELECT ... FOR UPDATE` 是 InnoDB 行级排他锁。为什么用悲观锁？转账写冲突高发——乐观锁冲突后要重试整个方法，悲观锁一次锁住效率更高 |
| ③   | 431-441 | 还原 from/to 方向 + `Objects.equals` 三重校验                       | 加锁顺序（先小后大）是并发考量，业务方向（from→to）是业务语义——两套逻辑解耦。为什么 `Objects.equals`？Integer 在 -128~127 有缓存，超出范围 `==` 比较引用而非值——隐蔽 bug                                                  |
| ④   | 450-457 | `BigDecimal currentBalance = initial + income - expense`    | **必须用 BigDecimal**：double 是 IEEE 754——`0.1+0.2=0.30000000000000004`。财务 1 分误差 = 生产事故。数据库 `DECIMAL(12,2)` ↔ Java `BigDecimal` 零精度丢失                                 |
| ⑤   | 461-469 | LambdaQueryWrapper 查 `name='其他' AND type=1`                 | **必须加 type 过滤**：种子数据"其他"有两条（支出 id=8 + 收入 id=13）。不加 → `selectOne()` 查 2 行 → `TooManyResultsException` → 转账崩溃。`Category::getName` 是方法引用——字段改名 → 编译报错                |
| ⑥   | 479-513 | `UUID.randomUUID()` → INSERT 转出(type=2) + INSERT 转入(type=1) | 为什么 UUID 而非自增？自增是 INSERT 后才生成，但需要 INSERT 之前把关联 ID 赋给两条记录。**复式记账**：一出一入共享 transferId，金额完全相等——转账不创造也不消灭金钱                                                           |
| ⑦   | 516-520 | 组装 TransferDTO → `toDTO()` Entity→DTO                       | `toDTO()` 传已加载引用——2 次调用共用，零额外查询                                                                                                                                   |

**"第 389 行 @Transactional——转账三步（查余额 + INSERT 转出 + INSERT 转入）在一个事务中。INSERT 转入时数据库崩了 → ROLLBACK → 转出也撤销——不会'钱扣了没到账'。"**

**"Mapper 层：AccountMapper 第 36-37 行 `@Select("SELECT * FROM account WHERE id = #{id} FOR UPDATE")`——自定义 SQL 覆盖 MyBatis-Plus 默认查询。TransactionMapper.insert() 是 BaseMapper 内置方法，生成参数化 INSERT。TransactionMapper.selectAccountIncome/Expense() 走 XML——`SELECT COALESCE(SUM(amount),0) FROM transaction WHERE ...`——COALESCE 保证无记录时返回 0 而非 NULL。"**

**"MySQL InnoDB：BEGIN → 按 ID 升序获取行锁 → 查余额 → INSERT × 2 → COMMIT（redo log 两阶段提交）→ 锁释放。"**

### 3.1 后端核心 ★ transfer() 逐行讲解（5分钟）

> **"这是我选的后端核心方法——TransactionServiceImpl 的 transfer()。35 行代码覆盖 7 个知识点。逐行讲——每一行做什么、为什么这样写。"**

---

**第 389 行 `@Transactional`**

| 做什么                                                  | 为什么                                             |
| ---------------------------------------------------- | ----------------------------------------------- |
| Spring 声明式事务。正常返回→COMMIT，抛 RuntimeException→ROLLBACK | 转账三步不可分割——INSERT 转入时数据库崩了，转出也回滚——**不会"钱扣了没到账"** |

---

**第 395 行** `public TransferDTO transfer(Long userId, TransferRequest request)`

| 做什么                                                                     | 为什么                                                                           |
| ----------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| 接收 userId（JWT 解析）和 request（fromAccountId / toAccountId / amount / note） | **userId 来自 JWT 而非请求体**——JWT 经 HMAC-SHA256 签名不可篡改。若来自请求体，攻击者可伪造 userId 转走别人的钱 |

---

**第 409-418 行** 前置校验（fail-fast，全放在加锁前）

```java
if (request.getAmount() == null)                             // 第409行
    throw new BusinessException(..., "转账金额不能为空");
if (request.getAmount().compareTo(BigDecimal.ZERO) <= 0)    // 第412行
    throw new BusinessException(..., "转账金额必须大于0");
if (request.getFromAccountId().equals(request.getToAccountId())) // 第416行
    throw new BusinessException(..., "转出和转入账户不能相同");
```

| 做什么               | 为什么                                                                                                                        |
| ----------------- | -------------------------------------------------------------------------------------------------------------------------- |
| 3 项前置校验，任一失败直接抛异常 | **fail-fast 模式**：锁是最贵的资源——任何能在加锁前排除的错误都不应该占着锁。`BigDecimal.compareTo()` 返回 -1/0/1——不能用 `<` `>` 运算符。`<= 0` 同时拦截"金额=0"和"金额<0" |

---

**第 426-429 行** ★ 死锁预防 + FOR UPDATE 悲观锁

```java
Long firstId = Math.min(request.getFromAccountId(), request.getToAccountId());
Long secondId = Math.max(request.getFromAccountId(), request.getToAccountId());
Account firstAccount = accountMapper.selectByIdForUpdate(firstId);
Account secondAccount = accountMapper.selectByIdForUpdate(secondId);
```

| 做什么                                                                                  | 为什么                                                                                                                           |
| ------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------- |
| `Math.min/max` 按 ID 升序确定加锁顺序，`selectByIdForUpdate()` = `SELECT ... FOR UPDATE` 行级排他锁 | **死锁预防**：线程1转1→5先锁1后锁5；线程2转5→1同样是 `min(5,1)=1` 先锁1。**所有线程锁顺序一致 → 打破循环等待 → 死锁不可能发生**。为什么悲观锁？转账写冲突高发——乐观锁冲突后要重试整个方法，悲观锁一次锁住效率更高 |

---

**第 431-441 行** 还原方向 + 账户校验

```java
Account fromAccount = request.getFromAccountId().equals(firstId) ? firstAccount : secondAccount;
Account toAccount = request.getToAccountId().equals(firstId) ? firstAccount : secondAccount;
if (fromAccount == null || !Objects.equals(fromAccount.getUserId(), userId)
    || !Objects.equals(fromAccount.getStatus(), Status.ACTIVE.getValue()))
    throw new BusinessException(...);
// 转入账户同样三重校验
```

| 做什么                            | 为什么                                                                                                                                                                |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 还原 from/to 方向 + 三重校验（存在/归属/活跃） | 加锁顺序（先小后大）是并发考量——业务方向（from→to）是业务语义——**两套逻辑完全解耦**。为什么 `Objects.equals`？Integer 在 -128~127 有缓存，超出范围 `==` 比较引用而非值——`Objects.equals` 内部先 `a==b` 再 `a.equals(b)`，安全且正确 |

---

**第 450-457 行** ★ BigDecimal 精度计算

```java
BigDecimal totalIncome = transactionMapper.selectAccountIncome(userId, fromAccount.getId());
BigDecimal totalExpense = transactionMapper.selectAccountExpense(userId, fromAccount.getId());
BigDecimal safeIncome = totalIncome != null ? totalIncome : BigDecimal.ZERO;
BigDecimal safeExpense = totalExpense != null ? totalExpense : BigDecimal.ZERO;
BigDecimal currentBalance = fromAccount.getInitialBalance()
    .add(safeIncome).subtract(safeExpense);
if (currentBalance.compareTo(request.getAmount()) < 0)
    throw new BusinessException(..., "余额不足");
```

| 做什么                                           | 为什么                                                                                                                                                                                        |
| --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 初始余额 + 收入 - 支出 = 当前余额 → `compareTo < 0` 判余额不足 | **必须用 BigDecimal**：`double` 是 IEEE 754——`0.1+0.2=0.30000000000000004`。财务 1 分误差 = 生产事故。`add()/subtract()` 返回新对象（BigDecimal 不可变类）。null 兜底：新账户无记录时 Mapper 返回 null → `BigDecimal.ZERO` 替换防 NPE |

---

**第 461-469 行** 查"其他"支出分类

```java
Category transferCategory = categoryMapper.selectOne(

        .eq(Category::getName, "其他")
```

| LambdaQueryWrapper 查 `name='其他' AND type=1`（支出类） | **必须加 type 过滤**：种子数据"其他"有两条——支出(id=8)和收入(id=13)。不加过滤 → `selectOne()` 查出 2 行 → `TooManyResultsException` → **转账崩溃**。`Category::getName` 是方法引用——**编译期检查**，比字符串 `"name"` 安全 |

**第 479-513 行** ★ UUID + 复式记账（双条 INSERT）

```java
String transferId = UUID.randomUUID().toString();               // 第479行
LocalDateTime now = LocalDateTime.now();                        // 第481行

Transaction outTransaction = new Transaction();                 // 第488行
outTransaction.setUserId(userId);
outTransaction.setAccountId(request.getFromAccountId());
outTransaction.setType(TransactionType.EXPENSE.getValue());     // type=2=支出
outTransaction.setAmount(request.getAmount());
outTransaction.setTransferId(transferId);                       // ★ 共用 transferId
// ... setCategoryId/Note/Time/CreateTime/UpdateTime ...
transactionMapper.insert(outTransaction);                       // 第499行 INSERT 转出

Transaction inTransaction = new Transaction();                  // 第502行
inTransaction.setUserId(userId);
inTransaction.setAccountId(request.getToAccountId());
inTransaction.setType(TransactionType.INCOME.getValue());       // type=1=收入
inTransaction.setAmount(request.getAmount());                   // ★ 金额完全相等
inTransaction.setTransferId(transferId);                        // ★ 同一个 transferId
// ...
transactionMapper.insert(inTransaction);                        // 第513行 INSERT 转入
```

| 做什么                                                        | 为什么                                                                                              |
| ---------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| UUID 生成关联 ID → INSERT 转出(支出) + INSERT 转入(收入)，共享 transferId | **为什么 UUID 而非自增**：自增 ID 是 INSERT 后才生成——但需要 INSERT 之前把关联 ID 赋给两条记录。UUID 在 Java 代码里生成，INSERT 之前就确定 |
| 一出一入两条记录                                                   | **复式记账**：资金从哪来到哪去完全可追溯。金额完全相等——转账不创造也不消灭金钱                                                       |
| @Transactional 包裹                                          | **INSERT 转入时数据库崩了** → ROLLBACK → 第 1 条 INSERT 也撤销——**"要么都成功，要么都不成功"**                            |

**函数 1：loadTransactions() — 第 326-361 行**

```javascript
326: async function loadTransactions() {
327:   loading.value = true                    // el-table v-loading 遮罩

331:     const params = { pageNum: pagination.pageNum, pageSize: pagination.pageSize }
```

|:-------:| ------------------------------------------------- | ------------------------------------------------ |
| 338-339 | 日期拼接 `" 00:00:00"` / `" 23:59:59"`                | `el-date-picker` 只返日期不带时间——不拼 23:59:59 当天数据全被排除  |


|:---:| --------------------------------- | ------------------------------------------------------------------------------------------ |

**函数 3：handleSubmit() — 第 493-521 行**

```javascript
493: async function handleSubmit() {
495:   const valid = await formRef.value.validate().catch(() => false)
496:   if (!valid) return                       // ★ 校验失败不发 HTTP 请求
498:   submitting.value = true                  // 按钮转圈+disabled
499:   try {
500:     if (isEdit.value) {
502:       await updateTransaction(editId.value, formData)  // PUT /api/v1/transaction/{id}

517:     }
518:   } finally {

521: }
```

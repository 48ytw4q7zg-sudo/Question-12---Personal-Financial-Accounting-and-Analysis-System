**目录职责一句话总结**:

| 目录 | 职责 | 类比 |
|------|------|------|
| `config/` | 系统配置（CORS、拦截器注册、JWT初始化、分页插件） | 电路开关 |
| `controller/` | 接收HTTP请求，参数校验，转发Service | 前台接待 |
| `service/` + `service/impl/` | 所有业务逻辑 | 业务部门 |
| `mapper/` | 数据库CRUD（MyBatis-Plus代理） | 数据管理员 |
| `entity/` + `entity/dto/` | 数据库表映射 + 接口传输对象 | 表单/档案 |
| `interceptor/` | 请求拦截（JWT认证 + 管理员授权） | 门禁系统 |
| `common/` | 通用组件（Result/异常/错误码） | 公共设施 |
| `util/` | 工具类（JWT/限流） | 工具箱 |
| `scheduler/` | 定时任务 | 自动值班 |

### 1.3 完整数据流：从前端到数据库

```

│                          完整数据流（以"登录"为例）                                      │
│                                                                                       │
│  前端                                 网络                    后端                    │
│  ────                                 ────                    ────                    │
│                                                                                       │
│  ① 用户点击"登录"按钮                                                                   │
│     │                                                                                 │
│     ▼                                                                                 │
│  ② LoginPage.vue                                                                      │
│     handleLogin()                                                                     │

│     │  ├── api/user.js login()         ← 调用 API 模块                                 │
│     │  └── .then(res => {              ← 成功后处理                                    │
│     │        userStore.setUser(res)     ← 存入 Pinia + localStorage                     │
│     │        router.push('/')          ← 跳转首页                                      │
│     │      })                                                                          │
│     ▼                                                                                 │
│  ③ api/user.js                                                                        │
│     request.post('/user/login', data)                                                 │
│     │                                                                                 │
│     ▼                                                                                 │
│  ④ api/request.js — 请求拦截器                                                          │
│     │  ├── localStorage.getItem('token')  ← 读取token（首次为null）                     │
│     │  ├── config.headers.Authorization = `Bearer ${token}` ← 注入JWT                  │
│     │  └── config.retryCount = 0          ← 初始化重试计数                              │
│     ▼                                                                                 │
│  ⑤ axios 发送 HTTP 请求                                                                │
│     POST /api/v1/user/login                            ──────────────────────────────→ │
│     Content-Type: application/json                  ⑥ Vite Dev Server (端口5173)       │
│     Body: {"username":"zhangsan","password":"123456"}   │ 检测 /api 前缀 → proxy转发   │
│                                                          │  → localhost:8080            │
│                                                          └───────────────────────────→│
│                                                                                       │
│                                                                  ⑦ Tomcat (端口8080)   │
│                                                                     │                  │
│                                                                     ▼                  │
```

### 2.2 过滤器层 — CorsFilter

**文件**: `config/CorsConfig.java`

#### 为什么需要 CorsFilter

浏览器有同源策略：`localhost:5173`（前端）不能直接请求 `localhost:8080`（后端），因为端口不同就是跨域。CorsFilter 在后端响应中加入 `Access-Control-Allow-Origin` 头，告诉浏览器"这个前端可以访问我"。

#### 为什么 CorsFilter 放在 Filter 层而不是 Interceptor 层

CORS 是 HTTP 协议层面的事，需要在 Spring MVC 之前处理。浏览器会先发一个 OPTIONS 预检请求，这个请求没有 Authorization 头，如果放在 Interceptor 层会被 LoginInterceptor 拦截返回 401。

#### 逐段讲解

```java
// 为什么: 浏览器对同一个 URL 的 OPTIONS 预检请求可以缓存，不用每次请求都发 OPTIONS
```

```java
// 第 74-88 行: 创建 CorsFilter Bean
@Bean
public CorsFilter corsFilter() {
    CorsConfiguration config = new CorsConfiguration();
    // 逐个添加允许源，禁止通配符 *
    Arrays.stream(allowedOrigins.split(",")).map(String::trim).forEach(config::addAllowedOrigin);
    // 为什么不用 addAllowedOriginPattern("*"): OWASP A01 安全规范要求，allowCredentials=true 时禁止通配
    config.setAllowCredentials(true);
    // 为什么需要 credentials: 前端 axios 请求可能携带 Cookie
    config.addExposedHeader("Authorization");
    // 为什么暴露 Authorization 头: 前端需要读取响应中的 Authorization 头（如 token 刷新场景）
    config.setMaxAge(CORS_PREFLIGHT_CACHE_SECONDS);
    // 为什么缓存: 减少 OPTIONS 预检请求，提升性能

    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", config);
    // 为什么 /** : 所有 API 路径都需要 CORS 支持
    return new CorsFilter(source);
}
```

### 2.3 拦截器层 — LoginInterceptor + AdminInterceptor

**文件**: `interceptor/LoginInterceptor.java` + `interceptor/AdminInterceptor.java`

#### 为什么需要拦截器（而不是在 Controller 里做）

如果每个 Controller 方法里都写 JWT 校验代码：
1. 重复代码（10 个 Controller × N 个方法）
2. 容易遗漏（新加接口忘了加校验）
3. 校验逻辑变了要改所有地方

拦截器把"谁可以访问"从"访问了什么"中分离出来。

#### Filter vs Interceptor 对比

| 维度 | Filter (Servlet) | Interceptor (Spring MVC) |
|------|-------------------|--------------------------|
| 执行时机 | 最早，在 DispatcherServlet 之前 | 知道目标 Controller 之后 |
| 能否注入 Bean | 不能直接 @Autowired | 可以 @Autowired |
| 能知道目标 Handler 吗 | 不能 | 能（知道是哪个 Controller 的哪个方法） |
| 本项目用途 | CorsFilter (HTTP 协议层) | LoginInterceptor (认证) + AdminInterceptor (授权) |
| 排除路径 | 需要自己写判断 | WebMvcConfig.excludePathPatterns() |

#### LoginInterceptor 逐行讲解


```java
// 第 53-81 行: preHandle() — JWT 认证的核心
@Override
public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
    // 第 55-57 行: OPTIONS 预检直接放行
    if ("OPTIONS".equalsIgnoreCase(request.getMethod())) {
        return true;
    }
    // 为什么: 浏览器 CORS 预检请求没有 Authorization 头，不放行会导致跨域失败

    // 第 59-63 行: 提取 Authorization 头
    String authHeader = request.getHeader("Authorization");
    if (authHeader == null || !authHeader.startsWith("Bearer ")) {
        writeError(response);  // 写入 401 JSON 响应
        return false;          // 中断请求链，Controller 不会执行
    }
    // 为什么检查 startsWith("Bearer "): JWT 标准格式是 "Bearer <token>"，
    // 后面要截取 Bearer 之后的部分

    // 第 66-71 行: 一次解析 token
    String token = authHeader.substring("Bearer ".length()).trim();
    JwtUtils.JwtPayload payload = JwtUtils.parseTokenPayload(token);
    if (payload == null) {
        writeError(response);
        return false;
    }
    // 为什么 parseTokenPayload() 一次解析而不是两次:
    // 旧代码先调用 parseToken() 再调用 parseRole()，每次都要 HMAC 验签，
    // 合并为一次解析，减少 50% 密码学运算

    // 第 74-79 行: 将 userId 和 role 存入 request 属性
    request.setAttribute("userId", payload.getUserId());
    request.setAttribute("role", payload.getRole() != null ? payload.getRole() : UserRole.NORMAL.getValue());
    // 为什么存到 request.setAttribute 而不是 ThreadLocal:
    // 每个 HTTP 请求有独立的 request 对象 → 天然隔离 → 不需要手动清理
    // Tomcat 线程池复用线程，如果用 ThreadLocal 可能出现串号

    return true;  // 放行，进入下一个拦截器或 Controller
}
```

```java
private void writeError(HttpServletResponse response) throws Exception {
    // 第 104-106 行: 检测响应是否已提交
    if (response.isCommitted()) {
        log.warn("响应已 committed，跳过 401 写入");
        return;
    }
    // 为什么检查 isCommitted: 如果前置 Filter 已经写入了响应，再写会抛 IllegalStateException

    response.setStatus(HTTP_OK);  // HTTP 200 (不是 401!)
    response.setContentType("application/json;charset=UTF-8");
    var writer = response.getWriter();
    writer.write(objectMapper.writeValueAsString(
        Result.error(ErrorCode.UNAUTHORIZED.getCode(), ErrorCode.UNAUTHORIZED.getMsg())
    ));
    writer.flush();

    // 为什么不 close writer: Servlet 容器统一管理，交给容器关闭
}
```

### 2.4 全局异常处理 — GlobalExceptionHandler

**文件**: `common/GlobalExceptionHandler.java`

#### 为什么需要全局异常处理器

没有全局异常处理时，每个 Controller 方法都要 try-catch：
```java
// 不好的写法
@PostMapping("/login")
public Result<?> login(@Valid @RequestBody UserLoginRequest request) {
    try {
        LoginResponse resp = userService.login(request);
        return Result.success(resp, "登录成功");
    } catch (BusinessException e) {
        return Result.error(e.getCode(), e.getMessage());
    } catch (Exception e) {
        return Result.error(500, "服务器内部错误");
    }
}
```

有了全局异常处理，Controller 只需要：
```java
@PostMapping("/login")
public Result<LoginResponse> login(@Valid @RequestBody UserLoginRequest request) {
    return Result.success(userService.login(request), "登录成功");
}
```

异常统一由 `@RestControllerAdvice` 处理。

#### body-code-first 约定

```
传统做法:                         本项目做法:
HTTP 401 → axios error 拦截器    HTTP 200 + body {code: 401, message: "未登录"}
→ 拿不到中文错误消息              → axios 响应拦截器检查 body.code
→ 用户体验差                      → 可以正确显示中文错误消息

为什么这样做:
Spring Boot 默认的 BasicErrorController 对 401/403/404 返回的是英文 HTML 或简单 JSON，
不包含业务错误码和中文消息。统一用 HTTP 200 + body code 让前端拦截器能统一处理。
```

#### 14 个异常处理者逐行讲解

```java
// JWT 结构: header.payload.signature
public static String generateToken(Long userId, String username, Integer role) {
    ensureInitialized();
    Date now = new Date();
    return Jwts.builder()
        .subject(String.valueOf(userId))     // sub: 用户ID
        .claim("username", username)          // 自定义声明
        .claim("role", role)                  // 自定义声明
        .issuedAt(now)                        // iat: 签发时间
        .expiration(new Date(now.getTime() + EXPIRE))  // exp: 过期时间 (7天)
        .signWith(KEY)                        // HMAC-SHA256 签名
        .compact();                           // Base64URL 编码
    // 为什么 subject 存 userId: JWT 标准的 sub 字段放主体标识
    // 为什么 username 和 role 放 claims 而不是重复解析数据库:
    // 避免每次请求都查库，JWT 自包含用户基本信息
}
```

#### LoginRateLimiter — 登录限流

**文件**: `util/LoginRateLimiter.java`

```java
public class LoginRateLimiter {
    // 每个用户名一个限流器，最多缓存 100 个，30 分钟过期
    private static final ConcurrentHashMap<String, RateLimiter> limiters = new ConcurrentHashMap<>();

    public static boolean tryAcquire(String username) {
        evictExpired();  // 定期清理过期条目
        RateLimiter limiter = limiters.computeIfAbsent(username,
            k -> RateLimiter.create(2.0));  // 每秒 2 次
        return limiter.tryAcquire();
    }
    // 为什么 RateLimiter.create(2.0):
    // Guava SmoothBursty 算法，每秒生成 2 个令牌
    // 正常用户不会每秒登录超过 2 次，超过说明可能是暴力破解

    // 为什么登录成功后调用 cleanup:
    // 成功后移除限流器，释放内存
    public static void cleanup(String username) {
        limiters.remove(username);
    }
}
```

```java
@Override
@Transactional
public TransferDTO transfer(Long userId, TransferRequest request) {
    // 第 317-322 行: 金额校验
    if (request.getAmount() == null || request.getAmount().compareTo(BigDecimal.ZERO) <= 0) {
        throw new BusinessException(ErrorCode.PARAM_INVALID.getCode(), "转账金额必须大于0");
    }
    // 为什么用 BigDecimal.compareTo 而不是 >:
    // BigDecimal 不能用 > < 运算符，必须用 compareTo

    // 第 324-326 行: 不能转给自己
    if (request.getFromAccountId().equals(request.getToAccountId())) {
        throw new BusinessException(ErrorCode.SAME_TRANSFER_ACCOUNT.getCode(), ...);
    }

    // 第 328-335 行: 死锁预防 — 按 ID 升序加锁
    Long firstId = Math.min(request.getFromAccountId(), request.getToAccountId());
    Long secondId = Math.max(request.getFromAccountId(), request.getToAccountId());
    Account firstAccount = accountMapper.selectByIdForUpdate(firstId);
    Account secondAccount = accountMapper.selectByIdForUpdate(secondId);
    // 为什么按 ID 排序加锁:
    // 如果线程1转账 A→B，线程2转账 B→A
    // 旧实现: 线程1先锁A后锁B，线程2先锁B后锁A → 死锁
    // 新实现: 两个线程都先锁 min(A,B) 再锁 max(A,B) → 永远是同一顺序 → 不死锁
    // 这是操作系统"破坏循环等待条件"的经典死锁预防方法

    // 第 334-335 行: 还原 from/to 引用
    Account fromAccount = request.getFromAccountId().equals(firstId) ? firstAccount : secondAccount;
    Account toAccount = request.getToAccountId().equals(firstId) ? firstAccount : secondAccount;
    // 为什么需要还原: 锁的顺序（first/second）和业务方向（from/to）是解耦的

    // 第 338-344 行: 校验账户状态
    if (fromAccount == null || !Objects.equals(fromAccount.getUserId(), userId)
        || !Objects.equals(fromAccount.getStatus(), Status.ACTIVE.getValue())) {
        throw new BusinessException(ErrorCode.ACCOUNT_NOT_FOUND_OR_DISABLED.getCode(), ...);
    }
    // 为什么用 Objects.equals: Integer 用 == 比较的是引用，值相等会误判

    // 第 346-355 行: 校验余额充足
    BigDecimal totalIncome = transactionMapper.selectAccountIncome(userId, fromAccount.getId());
    BigDecimal totalExpense = transactionMapper.selectAccountExpense(userId, fromAccount.getId());
```

```javascript
router.beforeEach(async (to, from, next) => {
    const token = localStorage.getItem('token')

    // 第 1 步: JWT 过期预检
    if (token) {
        tokenExpired = isTokenExpired(token)
        // 为什么在路由层预检而不是等后端返回 401:
        // 1. 速度更快：不用发请求就能判断
        // 2. 用户体验更好：切换路由时立即提示，不用等 API 返回
    }

    // 第 2 步: 鉴权检查
    const requiresAuth = to.matched.some(record => record.meta.requiresAuth)
    if (requiresAuth && (!token || tokenExpired)) {
        if (tokenExpired) {
            localStorage.removeItem('token')  // 清除过期 token
            ElMessage.warning('登录已过期，请重新登录')
        }
        next({ path: '/login', query: { redirect: to.fullPath } })
        // 为什么带 redirect 参数: 登录成功后跳回原页面，而不是总是到首页

    // 第 3 步: 已登录用户访问 /login → 跳到首页
    } else if (to.path === '/login' && token && !tokenExpired) {
        next('/')  // 防止重复登录

    // 第 4 步: 管理员权限检查
    } else if (to.matched.some(record => record.meta.requiresAdmin)) {
        const userStore = useUserStore()
        if (userStore.role !== ROLE_ADMIN) {
            ElMessage.warning('无权限访问该页面')
            next('/')  // 普通用户尝试访问 /admin → 重定向首页
            return
        }
        next()

    // 第 5 步: 正常放行
    } else {
        next()
    }
})
```

#### 2.9.4 stores/user.js — 用户状态管理

**文件**: `src/stores/user.js`

```javascript
export const useUserStore = defineStore('user', () => {
    // 状态（ref = 响应式基本类型）
    const userId = ref(null)
    const username = ref('')
    const role = ref(ROLE_NORMAL)
    const token = ref('')

    // 计算属性
    const isLoggedIn = computed(() => !!token.value)
    // 为什么用 computed 而不是直接读 token:
    // 其他组件可以 watch(isLoggedIn) 来响应登录/退出状态变化

    // 操作
    function setUser(data) {
        userId.value = data.userId
        username.value = data.username
        role.value = data.role
        token.value = data.token
        // Pinia state 和 localStorage 同时更新
        localStorage.setItem('token', data.token)
        localStorage.setItem('userInfo', JSON.stringify({
            userId: data.userId,
            username: data.username,
            role: data.role
        }))
        // 为什么存 localStorage: 刷新页面后 Pinia state 会丢失
        // localStorage 持久化，刷新后可以从 localStorage 恢复
    }

    function clearUser() {
        userId.value = null
        username.value = ''
        role.value = ROLE_NORMAL
        token.value = ''
        localStorage.removeItem('token')
        localStorage.removeItem('userInfo')
        // 为什么同时清 Pinia 和 localStorage:
        // 避免退出登录后刷新页面又从 localStorage 恢复登录状态
    }

    return { userId, username, role, token, isLoggedIn, setUser, clearUser }
})
```

#### 2.9.11 utils/jwt.js — JWT 过期预检

**文件**: `src/utils/jwt.js`

```javascript
export function isTokenExpired(token) {
    try {
        const payload = JSON.parse(atob(token.split('.')[1]))
        // 为什么解码 payload 而不是发给后端验证:
        // 1. 纯前端操作，毫秒级完成
        // 2. 减少一次不必要的 HTTP 请求
        // 3. 路由切换时即拦截，用户体验更好
        const exp = payload.exp * 1000  // JWT exp 是秒，JS Date 是毫秒
        return Date.now() > exp
    } catch (e) {
        return true  // 解码失败视为过期
    }
}
```

#### 2.9.12 utils/echarts-lazy.js — ECharts 按需加载

```javascript
let echartsPromise = null

export function loadEcharts() {
    if (echartsPromise) return echartsPromise  // Promise 缓存，只加载一次

    echartsPromise = Promise.all([
        import('echarts/core'),
        import('echarts/charts'),     // 按需引入图表类型
        import('echarts/components'), // tooltip/legend/grid
        import('echarts/renderers')   // Canvas 渲染器
    ]).then(([core, charts, components, renderers]) => {
        // 注册到 echarts
        core.use([charts, components, renderers])
        return core
    }).catch(err => {
        echartsPromise = null  // 失败时重置，下次重试
        throw err
    })

    return echartsPromise
}
// 为什么 Promise 缓存:
// DashboardPage 和 AnalyticsPage 都需要 echarts
// 第一个页面加载后缓存 Promise，第二个页面直接用
// 不重复加载 1MB 的 echarts 包
```

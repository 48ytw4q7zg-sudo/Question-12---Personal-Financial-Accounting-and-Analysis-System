### 1. 口头描述 JWT 的认证完整流程。为什么用 JWT 而不用传统 Session？

**参考答案：**

流程：① 用户提交账号密码 → `LoginPage.vue` 调用 `api/user.js` 的 `login()`，POST `/api/v1/user/login`；② 后端 `UserController.login()` → `UserServiceImpl.login()` 用 `LambdaQueryWrapper` 查用户 → `BCryptPasswordEncoder.matches()` 验证密码 → 调用 `JwtUtils.generateToken(userId, username, role)` 生成 JWT（HMAC-SHA256 签名，7天过期，claims 含 userId + username + role），返回给前端；③ 前端把 token 存 localStorage，后续请求由 `api/request.js` 的**请求拦截器**自动从 localStorage 读 token、拼成 `Bearer xxx` 写入 Authorization 头，10 个 API 模块统一注入不会遗漏；④ 后端 `LoginInterceptor.preHandle()`（第 92 行）拦截 `/api/v1/**`，4 步：OPTIONS 放行 → 提取 Bearer token → 调用 `JwtUtils.parseTokenPayload()` 一次 HMAC 验签同时提取 userId + role（性能优化：旧方案两次解析，合并后减少 50% 密码学运算）→ `request.setAttribute("userId", ...)` 存入请求属性（用 setAttribute 而非 ThreadLocal，Tomcat 线程池复用会导致串号漏洞）；⑤ 校验失败返回 HTTP 200 + body `{code: 401}`（body-code-first 约定），前端**响应拦截器**检测到 code===401 → 清 token + 清 Pinia userStore + router.push('/login')。

JWT vs Session 优势：① **无状态**：服务器不保存会话，本系统未引入 Redis，纯 JWT 即支持集群；② **跨域友好**：前后端分离（5173 ↔ 8080），JWT 通过 Header 传递不受同源限制；③ **可携带载荷**：userId/role 直接编码在 claims 里，`LoginInterceptor` 解析后 Controller 无需再查库。

安全实践：密钥通过环境变量 `jwt.secret` 注入（`JwtConfig.java`），生产环境检测到默认密钥阻止启动；role 为 null 时降级为普通用户而非默认提升管理员。

### 2. @Transactional 加在 private 方法上能生效吗？同一个类里 A 方法直接调用 B 方法（B 上有 @Transactional），事务能生效吗？分别说明原因。

**参考答案：**

- 加在 private 方法上**不生效**。Spring AOP 基于 CGLIB 动态代理，private 方法无法被子类代理覆盖。
- 同类内部 this.B() 调用**也不生效**，因为没经过代理对象，直接调用绕过了 AOP 拦截器链。
- 解决方法：把 B 方法拆到另一个 @Service 类，或通过 `AopContext.currentProxy()` 拿到代理对象再调。

本系统的验证：所有 `@Transactional` 都加在 public 方法上（`TransactionServiceImpl.transfer()`、`AccountServiceImpl.create()` 等），只读查询用 `@Transactional(readOnly = true)`。`BudgetScheduler` 中原本内部 `this.processSingleUser()` 调用导致 `@Transactional(REQUIRES_NEW)` 失效，后来特意抽成独立的 `BudgetAlertProcessorServiceImpl`，经过 Spring 代理后事务才正确生效——这就是同类内部调用事务失效的真实案例。`MybatisPlusConfig.java` 上有 `@EnableTransactionManagement` 显式启用。

### 3. Axios 怎么封装一个统一的请求实例？请求拦截器和响应拦截器分别在里面做什么事？

**参考答案：**

整体思路：① `api/request.js` 中用 `axios.create({ baseURL: '/api/v1', timeout: 10000 })` 创建实例——baseURL 对齐 `vite.config.js` 的 proxy（`/api` 前缀转发到 `:8080` 绕过跨域），timeout 10 秒；② 在实例上注册请求拦截器和响应拦截器；③ export 出去，10 个业务模块（user.js / account.js / transaction.js 等）统一引用。

**请求拦截器**做的事：
- 从 `localStorage.getItem('token')` 读取 JWT，拼成 `Bearer xxx` 格式写入 `config.headers.Authorization`
- 初始化重试计数器（`config.retryCount = 0`）
- 10 个 API 模块统一注入，防止遗漏导致 401

**响应拦截器**做的事（三段处理 + 指数退避重试）：
- `code === 200` → 剥壳只返回 `data`（业务代码拿到纯数据）
- `code === 401` → 清 token + 清 Pinia userStore + `router.push('/login?redirect=原页面')` 跳登录页。关键设计：`isRedirecting` 去重标志防 Dashboard 并发 3 个请求重复跳转，`redirect` 参数让登录后自动回到原页面
- 其他 code → `ElMessage.error(message)` 弹提示 + reject
- 错误分支：网络异常/超时/5xx → 指数退避重试（1s→2s→4s，最多 3 次，每次加 10-30% 随机抖动防"惊群效应"），4xx 和业务错误（标记 `noRetry=true`）不重试

### 4. 浏览器的同源策略限制了什么？解决跨域的常见方式有哪些？项目里 Spring Boot 是怎么配 CORS 的？

**参考答案：**

- **同源策略**：协议、域名、端口三者必须完全一致才算同源，否则浏览器限制 AJAX 请求。本系统前端 `localhost:5173`，后端 `localhost:8080`——端口不同就是跨域。
- **跨域常见方案**：① **CORS**（推荐）：后端在响应头加 `Access-Control-Allow-Origin` 等；② **JSONP**：只支持 GET，已淘汰；③ **Nginx 反向代理**：统一域名。

**本系统 Spring Boot 配置（`CorsConfig.java`）：**

配置了 `CorsFilter` Bean（**Filter 层**，比 Interceptor 更早执行）——允许源从配置文件注入（dev: `localhost:5173,5174` / prod: 环境变量注入实际域名），允许方法 GET/POST/PUT/DELETE/OPTIONS，允许请求头 Authorization/Content-Type/Accept，`allowCredentials=true`，预检缓存 1 小时。

**为什么 CORS 必须放 Filter 层？** 浏览器的 OPTIONS 预检请求不带 Authorization 头——如果放 Interceptor 层，`LoginInterceptor` 发现没 token 就返回 401，预检失败，所有跨域请求都被浏览器拦截。Filter 在协议层处理，预检直接放行。

安全加固：生产环境启动时检测到 localhost 阻止启动（`@PostConstruct validateCorsOrigins()`），必须通过环境变量替换为实际域名；`allowCredentials=true` 时不用通配符，逐个明确指定允许源（OWASP A01）。

### 5. 口头说明 @RequestParam、@PathVariable、@RequestBody 三种参数接收方式分别用于什么场景，前端发请求的形式分别是什么样。

**参考答案：**

- **@RequestParam**：接收 URL 查询参数（?key=value）。如本系统 `TransactionController.list()` 接收 `?accountId=1&keyword=超市&pageNum=1&pageSize=10` 等筛选条件，可选参数加 `required=false`，分页参数加 `defaultValue`。前端形如 `GET /api/v1/transaction?accountId=1&pageNum=2`。
- **@PathVariable**：接收 URL 路径中的变量。如本系统 `AccountController.update(@PathVariable @Min(1) Long id)`，对应 `@PutMapping("/{id}")`。前端形如 `PUT /api/v1/account/3`。
- **@RequestBody**：接收请求体里的 JSON。如本系统 `TransactionController.create(@Valid @RequestBody TransactionRequest request)`，前端 POST + `Content-Type: application/json` + body 写 JSON 对象 `{"accountId":1, "type":2, "amount":100.00, ...}`。

本系统还有 `@RequestParam("file")` 接收文件上传（`TransactionController.importCsv()`），前端 `multipart/form-data` 表单。

### 6. 全局异常处理是怎么实现的？口头说说要用到哪些注解、放在哪个类型的类上、典型会处理哪些异常。

**参考答案：**

- 在 `GlobalExceptionHandler.java` 上加 `@RestControllerAdvice`（= `@ControllerAdvice` + `@ResponseBody`），继承 `ResponseEntityExceptionHandler`（覆盖父类方法以正确替换默认异常处理），让它能拦截所有 Controller 抛出的异常；
- 类里写多个方法，每个加 `@ExceptionHandler(异常类型.class)` 或 `@Override` 覆盖父类方法，指定处理哪种异常；
- 所有方法统一返回 HTTP 200 + `Result.error(code, msg)`（body-code-first 约定，与 `LoginInterceptor` 保持一致）。

本系统处理的异常：① `BusinessException` — 提取 code + message 返回（1001-7xxx 错误码）；② `MethodArgumentNotValidException` — @Valid 校验失败，收集字段错误拼接返回 400；③ `MethodArgumentTypeMismatchException` — 参数类型不匹配；④ `ConstraintViolationException` — @Validated 路径参数校验失败；⑤ `HttpMessageNotReadableException` — JSON 请求体解析失败；⑥ `DataIntegrityViolationException` — 数据库约束违反；⑦ `NoResourceFoundException` — 资源不存在返回 404（不暴露路径防探测）；⑧ `NullPointerException` — 代码缺陷返回"内部数据异常"；⑨ `Exception.class` — 兜底，返回 500，堆栈不泄露给前端。

### 7. MyBatis-Plus 的分页插件怎么生效？口头说出配置步骤和调用步骤。

**参考答案：**

配置步骤（`MybatisPlusConfig.java`，第 28-33 行）：① 写一个配置类，注入 `MybatisPlusInterceptor` Bean；② 给它添加内层拦截器 `new PaginationInnerInterceptor(DbType.MYSQL)` 指定 MySQL 方言；③ 类上同时加 `@EnableTransactionManagement` 启用事务。

调用步骤（`TransactionServiceImpl.list()`，第 203-239 行）：① 创建 `Page<TransactionDTO> page = new Page<>(pageNum, pageSize)`；② 调 `transactionMapper.selectTransactionList(userId, ..., new RowBounds(offset, pageSize))` + 独立 count 查询获取总数（交易列表因动态 SQL + 动态 ORDER BY 走 XML 自定义分页，复杂筛选用 RowBounds 是标准 MyBatis 分页模式）；③ `page.getRecords()` 拿数据列表，`page.getTotal()` 拿总记录数。简单查询直接用 `mapper.selectPage(page, wrapper)`，MP 自动追加 LIMIT。

常见坑：忘了配 `PaginationInnerInterceptor`，分页参数被忽略，查全量。本系统 Controller 层做了 `@Min(1) @Max(100)` 校验，Service 层还有 `Math.min(pageSize, 100)` 上限保护防 OOM。

### 9. 父子组件通信中，父传子用什么？子传父用什么？口头描述完整流程。

**参考答案：**

- **父传子**：父组件在使用子组件标签时通过属性绑定传值（如 `:active-menu="route.path"`），子组件用 `defineProps` 声明接收（如本系统 `SidebarMenu.vue` 第 97 行：`defineProps({ activeMenu: String, collapsed: Boolean })`）。
- **子传父**：子组件用 `defineEmits` 声明可触发的事件（如 `SidebarMenu.vue` 第 103 行 `defineEmits(['navigate'])`），需要时调用 `$emit('事件名', 参数)`；父组件在子组件标签上用 `@事件名="处理函数"` 监听。
- **方法暴露**：子组件用 `defineExpose` 暴露方法，父组件通过 `ref` 调用。如本系统 `ConfirmDialog.vue`（第 106 行）暴露 `resetLoading()`，父组件通过 `confirmRef.value.resetLoading()` 重置按钮状态。

完整流程示例（删除账户）：父组件 `AccountPage.vue` 传 `v-model="deleteDialogVisible"` + `content="确定删除？"` 给 `ConfirmDialog`——这是父传子；用户点确认 → `ConfirmDialog` emit `confirm` 事件——这是子传父；父组件 `@confirm="handleDelete"` 执行删除 → 删除完成后 `confirmRef.value.resetLoading()`——这是通过 `defineExpose` 父调子。

### 11. 口头描述 Spring MVC 处理一个请求的完整流程（DispatcherServlet 的工作过程，至少 5 步）。

**参考答案：**

以 `GET /api/v1/transaction?pageNum=1` 为例：① 请求到达 Tomcat → 先经过 **CorsFilter**（Filter 层，检查 Origin 允许跨域，OPTIONS 预检放行）；② 进入 **DispatcherServlet**；③ `HandlerMapping` 根据 URL + HTTP 方法找到 `TransactionController.list()`；④ 按注册顺序执行**拦截器链**：`LoginInterceptor.preHandle()` 提取 Bearer token → HMAC 验签 → 解析 userId/role → `request.setAttribute()` 存入 → return true 放行；`AdminInterceptor` 检查路径不是 `/api/v1/admin/**` 跳过；⑤ `HandlerAdapter` 绑定参数（`@RequestParam` 映射到方法形参），调用 `TransactionController.list()` 方法；⑥ Controller → Service → Mapper → MySQL；⑦ Service 返回 `IPage<TransactionDTO>` → Controller 用 `Result.success(page)` 包装为 `{code:200, data:{...}}` → Jackson 序列化为 JSON → HTTP 200 响应；⑧ 前端 axios 响应拦截器三段处理（code===200 剥壳返回 data）。

关键：`LoginInterceptor` 先注册先执行（校验 token + 存 userId/role），`AdminInterceptor` 后注册后执行（读 role 校验管理员）——顺序不能反，反了 AdminInterceptor 读到 role 为 null，所有管理员请求被误判。

### 12. ref 和 reactive 都能创建响应式数据，怎么选？

**参考答案：**

- **ref**：包装基本类型（number/string/boolean），也可包装对象和数组；JS 中访问/修改要 `.value`，模板中自动解包不用写 `.value`。
- **reactive**：包装对象/数组，返回 Proxy 代理；不需要 `.value`，但解构后会丢失响应性（要用 `toRefs`）。

本系统的实践：基本类型和数组一律用 `ref`——`AccountPage.vue` 中 `const loading = ref(false)`、`const accountList = ref([])`；表单这种一组强关联字段用 `reactive`——`const formData = reactive({ name:'', type:1, initialBalance:0, currency:'CNY' })`；`DashboardPage.vue` 中月度汇总用 `reactive({ income:0, expense:0, balance:0 })`；Pinia store（`stores/user.js`）全部用 `ref`（组合式 store 风格）。

推荐：基本类型用 `ref`；数组和大多数对象也用 `ref`（统一加 `.value` 心智一致）；只有表单这类"一组强关联字段"用 `reactive`（且不解构）。

### 13. 什么是回调地狱？Promise 怎么解决它？async/await 又比 Promise 好在哪？

**参考答案：**

- **回调地狱**：多个异步操作嵌套（一个回调里再写一个回调），代码向右无限缩进，难读难维护，错误处理也麻烦。
- **Promise**：用 `.then().then().then()` 链式调用，把嵌套展平为线性，错误用 `.catch` 统一捕获。
- **async/await**：让异步代码看起来像同步代码（`const data = await api.get()`），无 `.then` 链；用 `try/catch` 直接捕获异常，调试时也能像同步代码一样看堆栈。

本系统的体现：10 个 API 模块（`api/user.js` 等）全部返回 Promise（`return request.post('/user/login', data)`），组件层用 async/await 消费。`DashboardPage.vue` 中 `async function loadBudgetAlerts()`（第 166 行）：`const data = await getBudgetAlert({ year, month })` → `try/catch` 捕获异常 → ElMessage 提示 + 降级兜底。axios 拦截器本身就是 Promise 链：请求拦截器 `return config`（resolve）或 `return Promise.reject(error)`（reject），响应拦截器 `return data`（剥壳），重试逻辑 `await delay(delayTime)` + `return request(config)`（递归 Promise）。

### 14. 口头描述：要查询「年龄大于 18、姓名以"张"开头、按创建时间倒序」的用户列表，用 LambdaQueryWrapper 怎么实现？涉及哪些方法？

**参考答案：**

思路：`new LambdaQueryWrapper<User>()` 创建包装器，链式调用：
- `.gt(User::getAge, 18)` → 大于 18
- `.likeRight(User::getName, "张")` → 姓名以"张"开头（likeRight = 右模糊，等价 `LIKE '张%'`）
- `.orderByDesc(User::getCreateTime)` → 按创建时间倒序

最后调 `userMapper.selectList(wrapper)` 拿结果。

本系统中 LambdaQueryWrapper 大量使用，如 `BudgetServiceImpl.list()`（第 94-98 行）：
```java
new LambdaQueryWrapper<Budget>()
    .eq(Budget::getUserId, userId)
    .eq(Budget::getMonth, monthStr)
```
`BudgetServiceImpl.save()`（第 163-168 行）中并发冲突处理也用 `.eq().eq().eq()` 链式调用精确匹配三条作唯一键。简单 CRUD 走 LambdaQueryWrapper，复杂多表 join/动态排序的交易列表走 XML 自定义 SQL——遵循 MyBatis-Plus 最佳实践。

关键优势：用 `User::getAge` 方法引用，**编译期类型安全**——字段改名编译不过，不会出现运行时 SQL 字符串拼错的 bug。

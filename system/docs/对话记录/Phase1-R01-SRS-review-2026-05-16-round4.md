### 维度 1: 完整性

- **issue-1** [严重度: 高]: §5 映射表中 CategoryPage 和 UserSettingsPage 在 §3 功能需求列表中无对应功能定义
  - **位置**: PRD §5 行 481-482 vs §3 全文（无分类浏览页 / 无用户设置功能）
  - **问题**: §5 有 18 行映射但 §3 只有 16 个功能（P0-1 到 P0-5 + P1-1 到 P1-6 + P2-1 到 P2-5）。CategoryPage 和 UserSettingsPage 是 Phase 1 Final Audit 后追加到 §5 的，但未在 §3 补充对应功能定义，导致 §3→§5 映射断裂
  - **修复建议**: 在 §3 新增 P0-6「分类浏览页」（描述：展示种子分类 + 各分类本月消费金额）和 P1-7「用户设置」（描述：查看用户信息 + 修改密码），保持 8 字段完整性

- **issue-2** [严重度: 中]: §5 新增的 CategoryPage 和 UserSettingsPage 无功能编号（`-` 占位），无法与 §3 建立对应关系
  - **位置**: PRD §5 行 481-482（功能编号列 = `-`）
  - **问题**: §5 其他行的功能编号列均为 `P0-N` / `P1-M` / `P2-K` 格式，唯独最后两行用 `-` 占位，破坏了 §5 内部编号一致性
  - **修复建议**: 随 issue-1 修复——§3 补充 P0-6 / P1-7 后，§5 功能编号列改为 `P0-6` / `P1-7`

  - **修复建议**: P1-4 异常流程②补充「若关联账户已禁用，编辑时账户下拉框允许更换为其他活跃账户；一键生成时拒绝并返回 5002」

### 维度 7: 反例推演

#### 7.1 删除依赖推演

| 实体 | 推演场景 | 推演链 | 结论 |
|---|---|---|---|
| account | 用户有 10 条 transaction + 3 个活跃 recurring_bill 引用账户 A → 点击禁用 | P0-2 异常流程②检查 transaction 和 recurring_bill → 拒绝 + 提示 | ✅ 已覆盖 |
| account | 用户有 10 条 transaction + 0 个活跃 recurring_bill → 点击禁用 | transaction 存在 → 拒绝 + 提示「有 N 条收支记录」 | ✅ 已覆盖 |
| account | 用户有 0 条 transaction + 3 个已停用 recurring_bill → 点击禁用 | transaction=0 且活跃 recurring_bill=0 → 允许禁用 | ✅ 正确（已停用账单不阻塞禁用） |
| account | 禁用后发现还有历史 transaction 需要查看余额 | P0-5 余额汇总只统计 status=1 账户 → 禁用账户不出现 | ⚠️ 历史交易记录仍可通过 transaction 列表查看（userId 过滤），但余额卡片不再展示该账户——PRD 未说明此行为 |
| recurring_bill | 停用后 → 一键生成按钮应消失 | P1-4 仅检查 bill.status=1 → 停用账单不可生成 | ✅ 已覆盖 |

- **issue-8** [严重度: 中]: 账户禁用后，关联的已停用周期性账单的「异常标记」语义不清
  - **位置**: PRD §3 P0-2 行 66 + P1-4 行 246
  - **推演**: 假设用户先停用周期性账单 X（status=0），再禁用关联账户 A（status=0）。P1-4 业务规则③说「引用的账户被禁用后，该周期性账单应标记为异常」——但 X 已经是停用状态，再标记「异常」语义重复。PRD 未区分「活跃+账户禁用=异常」和「已停用+账户禁用=无影响」两种场景
  - **修复建议**: P1-4 业务规则③改为「活跃的周期性账单（status=1）引用的账户被禁用后，该账单应标记为异常；已停用的账单（status=0）不受影响」

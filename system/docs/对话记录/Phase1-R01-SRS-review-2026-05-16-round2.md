#### 6.2 教学简化的边界声明

第二轮逐项复查:

- **issue-8** [严重度: 中]: P1-4 周期性账单的业务规则②与教学简化声明存在矛盾。业务规则②写「status 取值 {1: 活跃, 0: 停用} · 用户可停用/**重新启用**」，但 P0-2 账户的「禁用后不可恢复」模式在 §6 未被显式对标。P1-4 的「重新启用」意味着从 status=0 改回 status=1，但 PRD 的 API 形态中 `DELETE /api/recurring-bill/{id}` 对应停用，**没有 `PUT` 恢复接口**。业务规则说"可重新启用"但 API 未提供恢复路径 → 下游 coder 不知道怎么实现恢复。

### 维度 7:反例推演（第二轮 · 聚焦第一轮未覆盖的场景）

#### 7.1 删除依赖推演（第二轮）

**account 删除 + recurring_bill 引用（第一轮遗漏）**:
- 假设场景: 用户创建「支付宝」账户 → P1 阶段创建周期性账单「月工资」关联该账户 → 回到账户管理页点击「删除」
- 推演链: 系统检查 transaction 表 → 该账户下有 0 条收支记录 → 允许删除（改 status=0）→ recurring_bill 的 account_id 仍指向已禁用账户 → P1-4 业务规则说「标记为异常」→ 但异常标记是在列表展示时计算还是存储时触发？→ 如果是展示时计算（查账户 status），recurring_bill 本身无变化 → 用户看到「月工资」账单显示异常警告 → 但用户没有「更换关联账户」的入口 → **周期性账单永久异常**
- 结论: issue-7 已覆盖。

**recurring_bill 停用**:
- 假设场景同第一轮推演，结论无变化 ✓

#### 7.2 NULL / 空集合推演（第二轮）

**P1-4 recurring_bill.next_due_date**:
- PRD 主流程①要求填写「下次到期日」→ 该字段应该 NOT NULL
- 假设场景: 用户创建周期性账单时不填到期日 → 异常流程①说「名称为空 → 400」，但未提及到期日为空的校验
- 推演: 如果到期日为空，@Scheduled 定时任务不知道何时到期 → 无法提醒/标记异常
- 结论: PRD 异常流程①应补充「下次到期日为空 → 400」。但这是输入校验类的小遗漏，不影响核心功能。

- **issue-9** [严重度: 低]: P1-4 异常流程①输入校验未覆盖「下次到期日为空」场景。主流程①要求填写到期日，但异常流程仅校验了名称/金额/周期。
  - **位置**: PRD §3 · P1-4 异常流程①
  - **修复建议**: 异常流程①补充「下次到期日为空 → 400」。

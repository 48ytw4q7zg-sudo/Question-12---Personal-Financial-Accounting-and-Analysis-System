### 维度 7:反例推演

#### 7.1 删除依赖推演（逐实体）

**account 软删除**:
- 假设场景: 用户创建了「支付宝」账户，记了 50 笔收支记录，点击「删除」
- 推演链: 系统检查 transaction 表中 account_id = 该账户 → 发现 50 条 → 返回业务错误「该账户下有 50 条收支记录，请先处理后再禁用」→ 用户无法删除该账户 → 用户也没有「删除收支记录」或「转移收支记录到其他账户」的操作入口 → **账户永久无法禁用**
- 结论: P0-2 的删除依赖处理逻辑正确但体验有改进空间。不影响 P0 验收，属于低优先级优化点（未来可加「批量转移收支记录到其他账户」功能）。

**recurring_bill 停用**:
- 假设场景: 用户有一个周期性账单「月房租」，已通过「生成」按钮产生了 3 条收支记录，点击「停用」
- 推演链: 系统将 recurring_bill status 改为 0 → 已生成的 3 条收支记录为独立记录不受影响 → 正常
- 结论: 无 issue。

**transaction / budget / user 无删除操作**:
- 设计意图明确（不做删除），无外键约束问题。
- 结论: 无 issue。

#### 7.2 NULL / 空集合推演（逐可空字段）

**transaction.transfer_id（可空）**:
- 假设场景: 用户查看流水列表，列表中混有普通收支记录（transfer_id=NULL）和转账记录（transfer_id=非空）
- 推演链: NULL 的记录按正常收支展示（金额+分类+账户）→ 非 NULL 的记录如何展示？PRD 未定义 → 不同 coder 可能猜出不同实现（有的隐藏一半转账记录只显示一条、有的显示两条但无特殊标识、有的加「转出/转入」标签）
- 结论: NULL 语义未明确定义，下游 coder 实现会不一致。

- **issue-4** [严重度: 中]: P1-5 转账生成的收支记录在流水列表中的展示方式未定义。transaction.transfer_id 为 NULL（普通记录）vs 非 NULL（转账记录）时在列表中的显示区别未说明。P0-4 的异常流程/业务规则未涉及此场景。
  - **位置**: PRD §3 · P0-4 业务规则 · P1-5 业务规则
  - **修复建议**: P1-5 业务规则补充「转账记录在流水列表中标记为「转出/转入」，展示关联账户信息」；或 P0-4 业务规则补充「transfer_id 非空的记录在列表中加标记」。

**recurring_bill.account_id 关联账户被禁用（status=0）**:
- 假设场景: 用户将「支付宝」账户禁用，但该账户下有一个活跃的周期性账单「月工资」
- 推演链: 周期性账单列表展示该账单时显示异常警告标识 → 用户点击「生成」按钮 → `POST /api/recurring-bill/{id}/generate` → 系统需要检查关联账户状态 → PRD 未明确是否允许生成 → 不同 coder 可能实现不同（有的允许生成到已禁用账户、有的拒绝）
- 结论: 生成操作的前置校验未定义。

- **issue-5** [严重度: 中]: P1-4 周期性账单的「一键生成」操作（`POST /api/recurring-bill/{id}/generate`）未定义关联账户被禁用时的行为。业务规则仅写了「列表展示警告标识」，未说明生成操作是否允许/拒绝。
  - **位置**: PRD §3 · P1-4 异常流程
  - **修复建议**: P1-4 异常流程补充「生成操作时，若关联账户已禁用（status=0），拒绝生成并提示「关联账户已禁用，请先启用或更换账户」」。

**P2-4 currency（可空）**:
- PRD 已明确定义: 币种为空时默认 CNY ✓
- 结论: 无 issue。

#### 7.3 并发 / 重复操作推演（逐状态变更操作）

**P1-5 转账（状态变更类操作）**:
- 假设场景: 用户在转账页面填写「支付宝 → 微信 1000 元」，快速连点 2 次「确认转账」按钮
- 推演链: 第 1 次请求进入 → 检查支付宝余额 1000 ≥ 1000 → 生成两条记录（支付宝-1000 / 微信+1000）→ 余额更新为支付宝 0 / 微信 2000 → 第 2 次请求进入 → 检查支付宝余额 0 < 1000 → 返回余额不足 → 实际效果为 1 次转账成功（余额不足兜底）
- 但如果用户余额充足（如 10000 元）: 第 1 次请求 → 扣 1000 → 余额 9000 → 第 2 次请求 → 扣 1000 → 余额 8000 → **用户实际转账了 2000 元，但只打算转 1000 元**
- 结论: PRD 教学简化声明「不做幂等」，但连点导致实际金额翻倍的风险应在异常流程中说明。

- **issue-6** [严重度: 中]: P1-5 转账操作无幂等保护。余额充足时用户连点 2 次确认按钮，实际转账金额翻倍。虽然 PRD 已声明「教学简化：不做幂等」，但异常流程未覆盖「余额充足时重复提交」场景，仅写了「用户短时间内重复提交相同转账 → 无特殊处理」。
  - **位置**: PRD §3 · P1-5 异常流程
  - **修复建议**: 异常流程③补充完整描述:「用户连点 2 次确认按钮且余额充足 → 两次转账均成功（教学简化：不做幂等），前端应通过按钮 loading 状态防连点」。

**P1-3 预算保存**:
- 假设场景: 用户快速保存同一分类同一月份的预算 2 次（先设 1000，立刻改为 2000）
- 推演链: 第 1 次 → 创建/更新为 1000 → 第 2 次 → 覆盖为 2000 → 最终值 2000 → PRD 已定义「以最后提交为准（覆盖写入）」✓
- 结论: 无 issue。

**P0-1 用户注册**:
- 假设场景: 两个用户同时注册同名账号
- 推演链: 两条请求同时通过业务层校验（均未发现重复）→ 同时 INSERT → 第一条成功 → 第二条触发唯一索引冲突 → 返回用户名已存在 → PRD 已定义「数据库唯一索引兜底」✓
- 结论: 无 issue。

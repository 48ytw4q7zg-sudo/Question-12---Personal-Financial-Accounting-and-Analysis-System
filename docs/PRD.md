> 版本: v1.0 · 生成日期: 2026-05-16 · 依据: `docs/00-选题标定.md`（R-00 已审已修）

## §1 项目概述

个人财务记账与分析系统是一个面向个人用户的前后端分离 Web 应用，帮助用户管理多账户（现金/支付宝/微信等）的收支流水。系统支持用户注册登录（JWT 鉴权）、多账户 CRUD 管理、收支分类查询、分页流水记录、按账户汇总余额等基础功能（P0）。在此基础上，扩展预算管理、周期性账单提醒、多条件筛选、月度/年度汇总统计、ECharts 可视化图表（P1/P2）等功能，为用户提供从「记一笔」到「看趋势」的完整个人财务管理体验。

技术栈：SpringBoot 3.5.14 + MyBatis-Plus 3.5.15 + MySQL 8.4 LTS + Vue 3.5.34 + Element Plus 2.13.7 + Axios 1.15.2 + Pinia 3.0.4 + ECharts。

## §2 用户角色定义

- **普通用户** · 登录后可管理自己的账户、分类、收支记录、预算等全部功能模块 · 无管理员 / 其他角色

> 教学简化: 系统为单用户角色设计，所有数据按用户隔离，不存在跨用户共享场景。

### P0-1 · 登录/JWT

- **实现优先级**: P0 必做
- **描述**: 用户注册并登录，系统签发 JWT token 用于后续接口鉴权
- **前置条件**: 无
- **主流程**:
  1. 用户输入用户名 + 密码 → 系统校验用户名格式和密码强度
  2. 系统检查用户名是否已注册 → 若未注册则创建账户（BCrypt 加密密码），若已注册则校验密码
  3. 校验通过后签发 JWT token（含 userId）→ 返回 token 给前端
  4. 前端存入 localStorage，后续请求通过 Authorization: Bearer token 鉴权
- **异常流程**:
  - ① 输入校验: 用户名为空 / 密码为空 → 返回 400 + 提示信息；用户名格式不合法（含特殊字符）→ 返回 400
  - ② 数据关联约束: 注册时用户名已存在 → 返回业务错误码（如 1001: 用户名已存在）；登录时用户名不存在或密码错误 → 返回业务错误码（如 1002: 用户名或密码错误）· **不区分「用户名不存在」和「密码错误」以防枚举攻击**
  - ③ 并发/重复操作: 同一用户名同时注册 → 仅第一条成功（数据库唯一索引兜底）
  - ④ 权限边界: 未登录访问需要鉴权的接口 → 401
- **业务规则**:
  - ① 数据约束: 用户名 3-20 字符（字母/数字/下划线），密码 6-20 字符
  - ② 状态机: 无状态流转
  - ③ 可空关联: 无
  - ④ 教学简化声明: 使用硬编码 JWT 密钥（放 application.yml），token 有效期固定 7 天，不做 token 刷新机制
- **API 形态**:
  - `POST /api/v1/user/register` → `Result<UserDTO>`（注册）
  - `POST /api/v1/user/login` → `Result<LoginResponse>`（登录，返回 token）
- **关联页面**: LoginPage

- **实现优先级**: P0 必做
- **描述**: 用户管理多账户（现金/支付宝/微信等），支持列表查看、新增、修改、软删除（改 status）
- **前置条件**: 用户已登录
- **主流程**:
  1. 用户进入账户管理页 → 系统查询该用户所有账户列表
  2. 用户点击「新增账户」→ 填写名称、初始余额 → 系统创建账户
  3. 用户点击「编辑」→ 修改名称或初始余额 → 系统更新
  4. 用户点击「删除」→ 系统将账户 status 标记为禁用（软删除）· 账户保留但不再出现在活跃列表
- **异常流程**:
  - ① 输入校验: 账户名称为空 → 400；初始余额非数字或为负数 → 400
  - ② 数据关联约束: 删除账户时，若该账户下存在关联收支记录 → 拒绝删除并提示「该账户下有 N 条收支记录，请先处理后再禁用」；若该账户被活跃周期性账单引用（recurring_bill 中 status=1 且 account_id=该账户）→ 拒绝删除并提示「该账户下有 N 个活跃周期性账单，请先停用后再禁用」（三选一：拒绝 + 提示 · 不做级联删除）
  - ③ 并发/重复操作: 同一用户创建同名账户 → 允许（不同账户可同名，如两个「现金」账户）
  - ④ 权限边界: 用户只能操作自己的账户 · 通过 JWT userId 过滤，无行级权限隔离问题
- **业务规则**:
  - ① 数据约束: 账户名称 1-20 字符，初始余额 DECIMAL(12,2) · 精度保留 2 位；type 取值 {1: 现金, 2: 银行卡, 3: 支付宝, 4: 微信}
  - ② 状态机: status 取值 {1: 正常, 0: 禁用} · 新增时默认 1 · 禁用后不允许重新启用（教学简化）
  - ③ 可空关联: 无
  - ④ 教学简化声明: 删除采用改 status（软删除），不做物理删除；禁用后不可恢复
- **API 形态**:
  - `GET /api/v1/account` → `Result<List<AccountDTO>>`（账户列表，仅 status=1 的活跃账户）
  - `POST /api/v1/account` → `Result<AccountDTO>`（新增账户）
  - `PUT /api/v1/account/{id}` → `Result<AccountDTO>`（修改账户）
  - `DELETE /api/v1/account/{id}` → `Result<Void>`（软删除，改 status=0）
- **关联页面**: AccountPage

  - ① 数据约束: 金额 DECIMAL(12,2) > 0，备注 ≤ 200 字符，时间格式 ISO 8601

  - ③ 可空关联: 无

  - `POST /api/v1/transaction` → `Result<TransactionDTO>`（记一笔）

### P0-5 · 按账户汇总余额

- **实现优先级**: P0 必做
- **描述**: 统计每个活跃账户的当前余额（初始余额 + 收入 - 支出）
- **前置条件**: 用户已登录；至少有一个账户
- **主流程**:
  1. 用户进入账户管理页或余额卡片 → 系统查询每个账户的初始余额 + 累计收支
  2. 系统计算当前余额 = 初始余额 + SUM(收入金额) - SUM(支出金额) → 返回余额列表
- **异常流程**:
  - ① 输入校验: 无额外入参
  - ② 数据关联约束: 无（纯统计查询）
  - ③ 并发/重复操作: 无
  - ④ 权限边界: 只统计当前用户自己的账户和收支
- **业务规则**:
  - ① 数据约束: 余额精度 DECIMAL(12,2)
  - ② 状态机: 无
  - ③ 可空关联: 账户下无收支记录时，余额 = 初始余额
  - ④ 教学简化声明: 余额每次实时计算（不做缓存/冗余字段存储）
- **API 形态**:
  - `GET /api/v1/account/balance` → `Result<List<AccountBalanceDTO>>`（各账户余额汇总）
- **关联页面**: AccountPage（账户列表含余额列）

### P1-1 · 多条件筛选

- **实现优先级**: P1 应做
- **描述**: 对收支流水列表支持按时间范围、账户、分类、关键词多条件组合筛选
- **前置条件**: 用户已登录；已有收支记录数据
- **主流程**:
  1. 用户在流水列表页输入筛选条件（起止时间、账户、分类、关键词）→ 系统组合条件查询
  2. 查询结果仍走分页 → 返回筛选后的分页列表
  3. 筛选条件持久化到 URL query params → 支持刷新页面后保留筛选状态
- **异常流程**:
  - ① 输入校验: 起始时间 > 终止时间 → 400；关键词长度 > 50 → 400
  - ② 数据关联约束: 选择的账户/分类已被删除或不存在 → 忽略该条件（不报错，返回空结果）
  - ③ 并发/重复操作: 无
  - ④ 权限边界: 只筛选当前用户自己的数据
- **业务规则**:
  - ① 数据约束: 时间范围最大跨度 1 年（防止全表扫描）；关键词模糊匹配备注字段
  - ② 状态机: 无
  - ③ 可空关联: 账户/分类筛选条件为空时 = 不过滤该维度
  - ④ 教学简化声明: 关键词仅匹配备注字段，不做金额范围筛选
- **API 形态**:
  - `GET /api/v1/transaction` → `Result<Page<TransactionDTO>>`（复用分页接口，增加 query params: startTime / endTime / accountId / categoryId / keyword / pageNum / pageSize）
- **关联页面**: TransactionListPage（筛选器区域复用列表页）

### P1-3 · 预算管理

- **实现优先级**: P1 应做
- **描述**: 用户按分类设置月预算，系统标记超支情况并展示预算消耗进度
- **前置条件**: 用户已登录；分类数据已初始化
- **主流程**:
  1. 用户进入预算设置页 → 为每个支出分类设置月预算金额
  2. 系统保存预算 → 后续用户查看预算进度时，对比实际支出与预算
  3. 超支时标记为红色警告 → 前端展示进度条（已用/总额）
- **异常流程**:
  - ① 输入校验: 预算金额 ≤ 0 → 400；预算金额超出 DECIMAL(12,2) 范围 → 400
  - ② 数据关联约束: 引用的分类不存在或为收入分类 → 返回业务错误（预算仅适用于支出分类）
  - ③ 并发/重复操作: 用户短时间内重复保存同一分类同一月份的预算 → 以最后提交为准（覆盖写入）
  - ④ 权限边界: 用户只能操作自己的预算数据
- **业务规则**:
  - ① 数据约束: 预算金额 DECIMAL(12,2) > 0，按月+分类唯一约束（同一用户+同一月+同一分类仅一条预算）
  - ② 状态机: 无（预算创建后可编辑，无状态流转）
  - ③ 可空关联: 某支出分类未设置预算时，不参与超支判断（不标记超支）
  - ④ 教学简化声明: 预算粒度为月，不做周预算或自定义周期
- **API 形态**:
  - `GET /api/v1/budget` → `Result<List<BudgetDTO>>`（参数 year + month，该月所有分类的预算列表）
  - `POST /api/v1/budget` → `Result<BudgetDTO>`（创建或更新预算，对应「保存」按钮）
  - `GET /api/v1/budget/progress` → `Result<List<BudgetProgressDTO>>`（预算进度，含已用金额和超支标记）
- **关联页面**: BudgetPage（预算设置 + 进度条展示）

- **调整后**: P0-4 异常流程补充「修改时校验 transfer_id，转账记录仅允许修改备注，禁止修改金额」
- **理由**: 转账生成两条关联记录，修改其一金额会导致总额不一致

###### 调整 2: 统一错误码规范（issue-3 · 中）
- **原 PRD**: 未定义错误码规范，仅举例 1001/1002
- **调整后**: §3 末尾新增「统一错误码规范」段，定义 5 模块号段（1xxx=用户/2xxx=账户/3xxx=交易/4xxx=预算/5xxx=周期账单）+ HTTP 状态码映射
- **理由**: 下游 coder 需要统一的错误码表，避免各模块编号冲突

###### 调整 3: 转账记录展示 + NULL 语义（issue-4 · 中）
- **原 PRD**: P1-5 可空关联写「无」，未定义 transfer_id NULL 语义
- **调整后**: P1-5 业务规则③补充「transfer_id 非空 → 流水列表标记转出/转入并展示关联账户名称」
- **理由**: 不同 coder 对转账记录展示方式会猜出不同实现

###### 调整 4: 周期性账单一键生成约束（issue-5 · 中）
- **原 PRD**: P1-4 异常流程②仅覆盖创建/编辑时的账户状态校验
- **调整后**: 异常流程②补充「一键生成时关联账户已禁用 → 拒绝生成并返回 5002」
- **理由**: 生成操作的前置校验缺失，可能导致收支记录关联到已禁用账户

###### 调整 5: 转账连点防重复（issue-6 · 中）
- **原 PRD**: P1-5 异常流程③仅写「无特殊处理（教学简化）」
- **调整后**: 补充「余额充足时连点 2 次 → 两次均成功，前端应通过按钮 loading 状态防连点」
- **理由**: 教学简化不做后端幂等，但需在异常流程中完整说明后果 + 前端防护措施

###### 调整 6: P0-5 页面映射修正（issue-1 · 低）
- **原 PRD**: §5 P0-5 映射到「AccountPage, HomePage（余额卡片）」
- **调整后**: §5 P0-5 映射改为「AccountPage（余额列）」；P0-5 关联页面同步移除 HomePage
- **理由**: HomePage 不在标定卡 P0 页面列表中，余额卡片应挂靠 AccountPage

###### 调整 7: 账户删除检查 recurring_bill 引用（issue-7 · 高 · 第二轮）
- **原 PRD**: P0-2 异常流程②仅检查 transaction 表引用
- **调整后**: 补充「若该账户被活跃周期性账单引用 → 拒绝删除并提示先停用周期性账单」
- **理由**: 仅查 transaction 遗漏了 recurring_bill 的 account_id 外键引用

###### 调整 8: 周期性账单停用不可恢复（issue-8 · 中 · 第二轮）
- **原 PRD**: 业务规则②写「用户可停用/重新启用」但 API 无恢复接口
- **调整后**: 业务规则②改为「停用后不允许重新启用（教学简化: 同账户禁用模式）」
- **理由**: 规则与 API 形态矛盾，统一为「禁用后不可恢复」与 P0-2 一致

###### 调整 9: 周期性账单到期日校验（issue-9 · 低 · 第二轮）
- **原 PRD**: P1-4 异常流程①未校验「下次到期日为空」
- **调整后**: 异常流程①补充「下次到期日为空 → 400」
- **理由**: 到期日是 @Scheduled 定时任务判断是否到期的依据，空值会导致逻辑异常

###### 调整 10: §3 新增分类浏览页 + 用户设置（issue-1 + issue-6 · 高 · 第四轮）
- **原 PRD**: §5 映射表有 CategoryPage 和 UserSettingsPage 但 §3 无对应功能定义，change-password API 无归属
- **调整后**: §3 新增 P0-6「分类浏览页」（展示种子分类 + 各分类本月消费金额）和 P1-7「用户设置（修改密码）」；§5 功能编号改为 P0-6 / P1-7
- **理由**: Phase 1 Final Audit 追加页面时遗漏 §3 功能定义，导致 §3→§5 映射断裂 + 孤儿 API

###### 调整 11: P0-4 关联页面修正（issue-3 · 高 · 第四轮）

- **调整后**: 关联页面改为「TransactionListPage（记一笔为内嵌弹窗）」；§5 P0-4 同步修改

###### 调整 12: P0-2 补充账户 type 枚举（issue-4 · 高 · 第四轮）
- **原 PRD**: P0-2 业务规则①仅写「账户名称 1-20 字符，初始余额 DECIMAL(12,2)」

###### 调整 13: P1-4 异常流程补充账户禁用时的编辑/生成流程（issue-5 · 中 · 第四轮）

- **调整后**: 补充「若关联账户已禁用，编辑时账户下拉框允许更换为其他活跃账户；一键生成时拒绝并返回 5002」
- **理由**: P0-2 禁用检查与 P1-4 异常标记之间存在流程缝隙

###### 调整 14: P1-4 业务规则区分活跃/已停用账单异常标记（issue-8 · 中 · 第四轮）
- **原 PRD**: 业务规则③写「引用的账户被禁用后，该周期性账单应标记为异常」，未区分活跃/已停用
- **调整后**: 改为「活跃的周期性账单（status=1）引用的账户被禁用后，该账单应标记为异常；已停用的账单（status=0）不受影响」
- **理由**: 已停用账单再标记「异常」语义重复，需区分两种场景

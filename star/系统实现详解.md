### 4.3 删除交易 (特殊约束: 转账记录禁止删除)

```
DELETE /api/v1/transaction/{id}
  → TransactionController.java:144 delete(id)
    → TransactionServiceImpl.java:271 delete(userId, id)
      → getTransactionById() (校验归属)
      → transferId != null → 抛 TRANSFER_RECORD_NOT_DELETABLE(3007) (第 276 行)
      → 普通记录: deleteById (第 279 行)
```

**为什么转账记录禁止直接删除**：转账记录成对出现（一出一进），单独删一条会破坏余额一致性。必须同时处理两条。

### 4.4 分页查询 (List)

```

      → 时间范围 ≤ 1 年校验 (第 135 行)
      → LIKE 通配符转义 (第 148 行 · escapeLikeKeyword)
      → TransactionMapper.xml selectTransactionList (JOIN account+category)

```

**XML 映射**: `backend/.../mapper/TransactionMapper.xml:25-55` — 动态 SQL 拼接筛选条件 (accountId/categoryId/startTime/endTime/keyword) + 动态 ORDER BY (time/amount_asc/amount_desc)。使用 `<include refid="transactionFilterConditions">` 复用条件片段 (第 6-22 行)。

**金额校验**: DTO `@DecimalMin("0.01")` (第 44 行) + Service 层 `compareTo(ZERO) <= 0` 双重校验。

### 11.2 死锁预防机制 (P1-1 Q-CR Loop1 修复)

**问题**: 线程1 转账 A→B，线程2 转账 B→A。原实现先锁 fromAccount 再锁 toAccount，形成循环等待 → 死锁。

**修复** (`TransactionServiceImpl.java:329-334`):
```java
Long firstId = Math.min(fromAccountId, toAccountId);    // 较小的 ID 先锁
Long secondId = Math.max(fromAccountId, toAccountId);   // 较大的 ID 后锁
Account firstAccount = selectByIdForUpdate(firstId);    // 先锁小 ID
Account secondAccount = selectByIdForUpdate(secondId);   // 再锁大 ID
// 根据 from/to 还原引用 (锁顺序与业务方向解耦)
fromAccount = (fromAccountId.equals(firstId)) ? firstAccount : secondAccount;
toAccount = (toAccountId.equals(firstId)) ? firstAccount : secondAccount;
```

**原理**: 所有转账线程都按 ID 升序加锁，破坏死锁的「循环等待」必要条件（操作系统四大必要条件之一）。

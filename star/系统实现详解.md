### 0.4 技术选型决策树

以下决策树记录了每个核心技术点的"为什么选 A 而不是 B"，供答辩时解释技术选型依据。

```
Q1: 后端框架选什么？
├─ SpringBoot 3.5.14 (LTS) ✅
│  理由: Spring 生态成熟 + 国内就业市场主流 + MyBatis-Plus 集成无缝
├─ 为什么不选 Quarkus?
│  Quarkus 启动更快但社区中文资料少，不适合课堂学习型项目
└─ 为什么不选 Gin/Express?
   本项目定位 Java 全栈，非多语言混合项目

Q2: 数据库选什么？
├─ MySQL 8.4 LTS ✅
│  理由: 关系型 + 事务(ACID) + 免费 + 国内使用率最高
├─ 为什么不选 PostgreSQL?
│  PG 更强大但国内使用率低，MySQL 在课堂环境中运维更简单
└─ 为什么不选 MongoDB?
   MongoDB 无 schema 不适合财务数据（金额必须精确约束）

Q3: ORM 框架选什么？
├─ MyBatis-Plus 3.5.15 ✅
│  理由: LambdaQueryWrapper 防 SQL 注入 + 自动分页 + 代码生成器
├─ 为什么不选 JPA/Hibernate?
│  JPA 自动建表/级联操作是黑盒，课堂项目中不容易理解 SQL 执行细节
└─ 为什么不选原生 JDBC?
   JDBC 模板代码太多，手写 ResultSet 解析容易出错且不安全

Q4: 前端框架选什么？
├─ Vue 3.5 + Element Plus 2.13 ✅
│  理由: 中文生态最好 + Element Plus 企业级后台组件库 + Composition API 先进
├─ 为什么不选 React?
│  React 搭配 Ant Design 也是一种选择，但 Vue + Element Plus 中文文档更友好
└─ 为什么不选 Angular?
   Angular 学习曲线陡峭，全量 TypeScript 对课堂项目过重

Q5: JWT 方案选什么？
├─ JJWT 0.13.0 + HS256 ✅
│  理由: 对称加密简单高效 + 无状态(无需 Redis session) + 模块化依赖
├─ 为什么不选 RSA 非对称?
│  RSA 生成密钥对复杂，无状态的 HS256 足够满足单服务场景
├─ 为什么不选 OAuth2/Session+Redis?
│  本项目是单体应用，OAuth2 过重，Session 需要额外 Redis 运维
└─ JWT 过期策略: 固定 7 天过期 + 前端 exp 预检（不做 refresh token）
   理由: 课堂项目用户量小，简化 token 管理，不引入 refresh token 复杂性

Q6: 前端状态管理选什么？
├─ Pinia 3.0 ✅
│  理由: Vuex 已停止维护，官方推荐 Pinia；组合式 store 与 Composition API 风格一致
└─ 为什么仅 1 个 store (user.js)?
   本项目状态以"用户认证"为核心，其余数据通过 API 实时获取，不需要复杂状态同步

Q7: 密码加密选什么？
├─ BCrypt (cost=12, spring-security-crypto 6.3.4) ✅
│  理由: BCrypt 自带 salt + 可调节 cost + 抵抗彩虹表
├─ 为什么不选 MD5/SHA-256?
│  MD5 已破解，SHA-256 无 salt 无法抵抗彩虹表
├─ 为什么不选 scrypt/Argon2?
│  BCrypt 生态更成熟，JJWT+BCrypt 都被 spring-security-crypto 覆盖
└─ 为什么只引 crypto 子模块不引完整 security?
   完整 Spring Security 会注入默认 Filter Chain（CSRF/Session 等），干扰本项目自定义 JWT 拦截器

Q8: 构建工具选什么？
├─ 后端 Maven 3.9 ✅ | 前端 pnpm 10.33 ✅
│  Maven: SpringBoot 官方推荐 + 国内 Java 项目事实标准
│  pnpm: 严格依赖隔离 + 磁盘节省 + Vite 官方推荐
├─ 为什么不选 Gradle?
│  本项目命令行工具链已定型为 Maven，Gradle 的 Groovy DSL 增加学习曲线
└─ 为什么不选 npm/yarn?
   pnpm 硬链接机制避免幽灵依赖，node_modules 体积同比缩减 50%+

Q9: 图表库选什么？
├─ ECharts 5.5 ✅
│  理由: 百度开源 + 中文文档最好 + Apache 项目 + 按需加载体积可控
├─ 为什么不选 Chart.js?
│  Chart.js 交互少、中文资料少，ECharts drill-down/联动更强
└─ 为什么不选 D3.js?
   D3.js 灵活但学习曲线极高（需理解 SVG 层级），适合定制可视化而非快速报表

Q10: CSV 解析选什么？
├─ OpenCSV ✅
│  理由: 轻量 + 注解驱动 + Bean 映射 + 无需额外依赖
├─ 为什么不选 Apache POI?
│  POI 针对 Excel(.xlsx)，本项目 CSV(.csv) 用 POI 过于庞大
└─ 为什么不选 SuperCSV?
   SuperCSV 已数年未更新，OpenCSV 仍在维护
```

### 1.4 常见坑点与解决方案

**坑点 1: JWT 密钥过短导致启动失败**

现象: Spring Boot 启动时抛出 `IllegalArgumentException: JWT secret must be at least 256 bits (32 bytes)`。

根因: `JwtConfig.java:80-98` 在 `@PostConstruct` 阶段校验密钥长度。HS256 要求密钥 >= 256 bits (32 bytes)。开发环境中 `application-dev.yml` 的默认占位密钥 `your-secret-key-change-in-production-do-not-use-this` 仅有部分满足要求，具体取决于字符串长度。

解决方案:
1. 确认 `application-dev.yml` 中 `jwt.secret` 的值长度 >= 32 字符
2. 正式环境通过环境变量 `JWT_SECRET` 注入 >= 64 字符的随机值
3. 生成随机密钥命令: `openssl rand -base64 64`

**坑点 2: 前端 token 过期后仍停留在页面，请求持续 401**

现象: 用户登录后长时间不操作，token 过期后前端仍然停留在当前页面，所有 API 请求返回 401。

根因: `router/index.js:118-160` 的 `beforeEach` 守卫只在路由切换时做 token 过期预检。如果用户页面闲置但未切换路由，已过期的 token 不会被立即检测。

解决方案:
1. `api/request.js:82-125` 响应拦截器中 `code === 401` 分支执行 `router.push('/login')`——这是**运行时**检测
2. 在 `request.js` 中用 `setInterval` 每 60 秒检查一次 token 过期（可选增强）
3. 前端 `jwt.js:isTokenExpired()` 提前 5 分钟判定过期，留缓冲时间

**坑点 3: 注册时用户名重复报 `DuplicateKeyException` 而非友好提示**

现象: 两个用户几乎同时注册同一用户名，其中一个收到 `500 Internal Server Error` 而非 `用户名已存在`。

根因: 多线程并发注册时，两个线程同时通过 `selectOne` 检查（都发现不存在），然后都执行 `insert`。第一个 insert 成功，第二个触发数据库 `uk_user_username` 唯一索引冲突。

解决方案: `UserServiceImpl.java:74-78` 已实现——`catch(DuplicateKeyException)` 兜底，转换为 `BusinessException(ErrorCode.USERNAME_EXISTS, "用户名已存在")`。同时配合 `LoginRateLimiter` 降低并发注册概率。

### 2.5 常见坑点与解决方案

**坑点 1: 账户 initialBalance 传入 null 导致 NullPointerException**

现象: 前端创建账户时如果未填初始余额（空字符串），axios 将字段设为 `null` 而非 `0`，后端 `BigDecimal` 运算时报 NPE。

根因: Java 的 `BigDecimal` 不能与 `null` 做算术运算。`AccountRequest.java` 中 `initialBalance` 字段虽设置了 `@DecimalMin("0")`，但 `null` 值不会触发此校验（`@NotNull` 与 `@DecimalMin` 是独立的）。

解决方案: `AccountServiceImpl.java:113` 已实现 `null` 安全兜底——
```java
BigDecimal initialBalance = request.getInitialBalance() != null
    ? request.getInitialBalance() : BigDecimal.ZERO;
```
前端侧也应在 `AccountPage.vue` submit 前将空值转为 `0`。

**坑点 2: 删除账户时只检查了 transaction 但有转账记录的残留**

现象: 账户 A 与账户 B 之间存在转账记录（`transfer_id` 非 null），删除 A 时检查 `transaction` 表发现存在记录但 `transfer_id` 非 null 被排除，导致删除"成功"——但后续查转账详情时关联账户已不存在。

根因: `AccountServiceImpl.java:163` 的删除前检查排除了 `transfer_id IS NOT NULL` 的记录（因为每笔转账生成 2 条 transaction）。这是有意为之——转账记录不应阻止账户删除，因为转账是"中间账户"行为，不直接新增余额。

现状与权衡: 系统当前接受这个边界情况——删除账户后，历史转账记录的关联信息变为"僵尸引用"。前端展示时已做防御（`transaction.accountName` 为 null 时显示"已删除账户"）。如需更严格处理，可改为检查 `status=1` 的关联账户并提示用户先处理转账。

### 4.5 金额并发安全

财务系统中，金额一致性是最高优先级的安全要求。本节对比两种并发控制策略。

#### 4.5.1 悲观锁 vs 乐观锁 — 选择理由

| 维度 | 悲观锁 (`SELECT ... FOR UPDATE`) | 乐观锁 (`@Version` 或 `CAS UPDATE`) |
|---|:---|:---|
| **原理** | 读取时加行锁，阻止其他事务修改 | 更新时检查版本号，冲突则重试或失败 |
| **适用场景** | 写冲突频繁（转账、扣款） | 读多写少（配置修改、个人信息更新） |
| **并发性能** | 低（阻塞等待） | 高（无锁等待） |
| **数据安全性** | 强（不会丢失更新） | 弱（冲突时需重试逻辑） |
| **实现复杂度** | 低（SQL 加 FOR UPDATE 即可） | 中（需 version 字段 + 重试机制） |
| **死锁风险** | 有（多锁无序时） | 无 |
| **本项目选择** | ✅ **悲观锁 — 转账场景** | 不采用（写冲突概率高） |

**选择理由（详情）**:

1. **转账场景写冲突概率高**: 用户 A 向用户 B 转账时，两个账户余额可能同时被其他转账线程修改。悲观锁保证"读余额 → 校验 → 扣减 → 写入"整个过程无其他事务插入。

2. **乐观锁在财务场景的缺陷**: 假设账户余额 1000，线程 T1 转出 800，线程 T2 转出 500。乐观锁下两个线程都可能读到余额 1000，T1 先提交(余额变 200)，T2 提交时 version 已变 → 失败重试 → 重新读余额 200 → 校验通过(200 < 500? No!) → 正确拒绝。这个流程依赖重试逻辑的正确性，一旦重试逻辑有 bug，会导致余额透支。

3. **悲观锁代码对齐数据库事务**: MySQL InnoDB 行锁与 Spring `@Transactional` 天然配合，锁在事务提交时自动释放，无需手动 unlock。

#### 4.5.2 悲观锁实现代码（转账场景）

```java
// AccountMapper.java:19
@Select("SELECT * FROM account WHERE id = #{id} FOR UPDATE")
Account selectByIdForUpdate(@Param("id") Long id);

// TransactionServiceImpl.java:329-334 (转账流程)
@Transactional(rollbackFor = Exception.class)
public TransferDTO transfer(Long userId, TransferRequest request) {
    // 死锁预防: 按 ID 升序加锁
    Long firstId = Math.min(fromAccountId, toAccountId);
    Long secondId = Math.max(fromAccountId, toAccountId);
    Account firstAccount = accountMapper.selectByIdForUpdate(firstId);   // 先锁小 ID
    Account secondAccount = accountMapper.selectByIdForUpdate(secondId); // 再锁大 ID

    // 根据 from/to 还原引用
    Account fromAccount = (fromAccountId.equals(firstId)) ? firstAccount : secondAccount;
    Account toAccount = (toAccountId.equals(firstId)) ? firstAccount : secondAccount;

    // === 以下为锁保护下的临界区 ===
    // 余额校验
    BigDecimal currentBalance = calculateBalance(fromAccount);
    if (currentBalance.compareTo(request.getAmount()) < 0) {
        throw new BusinessException(ErrorCode.BALANCE_INSUFFICIENT);
    }
    // INSERT 转出记录 + INSERT 转入记录 ...
}
```

#### 4.5.3 为什么不选乐观锁的代码对比

如果采用乐观锁，转账流程会变成（示意对比）:

```java
// 乐观锁版本（本项目不采用，仅作对比）
@Transactional
public TransferDTO transferOptimistic(Long userId, TransferRequest request) {
    int maxRetries = 3;
    for (int retry = 0; retry < maxRetries; retry++) {
        Account from = accountMapper.selectById(fromId); // 无锁读取
        Account to = accountMapper.selectById(toId);     // 无锁读取

        BigDecimal balance = calculateBalance(from);
        if (balance.compareTo(request.getAmount()) < 0) {
            throw new BusinessException(ErrorCode.BALANCE_INSUFFICIENT);
        }

        // UPDATE ... SET version = version + 1 WHERE id = ? AND version = ?
        int updated = accountMapper.updateWithVersion(from); // CAS 更新
        if (updated == 1) {
            insertTransaction(...);  // 成功
            return result;
        }
        // version 冲突 → 重试
    }
    throw new BusinessException(ErrorCode.CONCURRENT_CONFLICT);
    // ❌ 问题1: 重试 3 次仍可能失败，用户体验差
    // ❌ 问题2: 版本号字段侵入 account 表结构
    // ❌ 问题3: 如果忘记重试逻辑，金额数据可能不一致
}
```

#### 4.5.4 死锁预防的必要性

悲观锁的最大风险是**死锁**。两个事务相互等待对方持有的行锁 → 无限等待。本项目通过**加锁顺序规范**化解:

```
所有转账线程强制按 account.id 升序加锁:
  转出账户 ID=5, 转入账户 ID=3
  → Math.min(5,3)=3 先锁 → Math.max(5,3)=5 后锁
  → 无论业务方向(from→to 还是 to→from)，锁顺序永远一致
  → 破坏死锁四必要条件之一的"循环等待"
```

详见 §11 P1-5 转账 + 死锁预防。

### 4.6 常见坑点与解决方案

**坑点 1: transfer_id 字段导致统计 SQL 虚增金额**

现象: Dashboard 月收入/支出统计明显高于实际消费金额，如实际花了 5000 但统计显示 8000。

根因: 每笔转账生成 2 条 transaction 记录（一条支出标记转出，一条收入标记转入）。如果统计 SQL 未加 `WHERE transfer_id IS NULL` 过滤，转账的两条记录都会被计入收入/支出，导致虚增。

解决方案: 所有统计接口的 SQL 均需 `AND t.transfer_id IS NULL` 排除转账记录。`StatisticsServiceImpl` 的 4 个统计方法以及 `AccountServiceImpl.getBalance()` 的 GROUP BY 聚合均已遵循此规则。这是 Q-CR Loop1 早期修复的关键 Bug。

**坑点 2: 转账记录允许修改备注但禁止修改金额的实现细节**

现象: 更新转账关联的 transaction 时传了新的 amount，期望只改备注，但金额被悄悄改了。

根因: `TransactionServiceImpl.java:220-255` update 方法。早期实现仅判断了 `transferId != null` 就整体拒绝修改，未区分"只改备注"和"改金额"两种情况。

解决方案: 修改后的实现 (`TransactionServiceImpl.java:224-233`) 采用精细化校验:
```java
if (transaction.getTransferId() != null) {
    // 允许修改备注
    if (request.getAmount() != null
        && transaction.getAmount().compareTo(request.getAmount()) != 0) {
        throw new BusinessException(ErrorCode.TRANSFER_RECORD_NOT_MODIFIABLE); // 3006
    }
    // 仅更新 note 字段
    transaction.setNote(request.getNote());
    transactionMapper.updateById(transaction);
    return toDTO(transaction, account, category);
}
```

**坑点 3: LIKE 通配符注入导致异常查询结果**

现象: 用户搜索关键词 `%` 或 `_` 时，返回了全部记录而不是包含这些字符的记录。

根因: MySQL LIKE 语法中 `%` 和 `_` 是通配符。如果用户输入未被转义，`LIKE '%_%'` 会匹配所有至少含 1 个字符的记录而非字面量的 `_`。

解决方案: `TransactionServiceImpl.java:544-547` 的 `escapeLikeKeyword()` 方法:
```java
private String escapeLikeKeyword(String keyword) {
    if (keyword == null || keyword.isEmpty()) return null;
    return keyword.replace("\\", "\\\\")
                  .replace("%", "\\%")
                  .replace("_", "\\_");
}
```
配合 `XML: LIKE CONCAT('%', #{keyword}, '%') ESCAPE '\\'` 确保字面匹配。

### 6.1 常见坑点与解决方案

**坑点 1: Promise.allSettled 中单个接口失败导致图表空白**

现象: Dashboard 5 个图表中有 1 个报错（如超时），但所有 5 个都显示空白。

根因: 早期实现使用 `Promise.all()`。根据 JS Promise 规范，`Promise.all()` 是 fail-fast——任何一个 Promise reject 都会立即拒绝整个 Promise，其余 fulfilled 的结果被丢弃。

解决方案: 已改为 `Promise.allSettled()`（`DashboardPage.vue:337`）。`allSettled` 等所有 Promise 结束后返回每个的结果（无论 resolved 还是 rejected），这样 4 个成功的图表依然能正常渲染，仅失败的 1 个显示空态。

对比:
```javascript
// ❌ Promise.all — 1 个失败全部丢弃
const results = await Promise.all([...5个请求]);
// ✅ Promise.allSettled — 各自处理
const results = await Promise.allSettled([...5个请求]);
results.forEach(r => {
  if (r.status === 'fulfilled') renderChart(r.value);
  else console.warn('图表加载失败:', r.reason);
});
```

**坑点 2: ECharts 实例未销毁导致切换页面时内存泄漏**

现象: 多次进出 Dashboard 页面后，浏览器内存占用持续上升，最终页面卡顿。

根因: ECharts 实例持有 DOM 引用 + resize 事件监听器。页面切换（路由变化）时，Vue 销毁了 DOM 但 `echarts.dispose()` 未被调用——resize 监听器仍在内存中监听 window resize。

解决方案: `DashboardPage.vue:344-350` 的 `onUnmounted` 钩子:
```javascript
onUnmounted(() => {
  chartInstances.forEach(instance => {
    instance.off('click');           // 解绑事件
    instance.dispose();              // 销毁 ECharts 实例
  });
  chartInstances.length = 0;         // 清空引用
  window.removeEventListener('resize', handleResize);
});
```

### 11.2 死锁预防机制 (P1-1 Q-CR Loop1 修复)

**问题**: 线程1 转账 A→B，线程2 转账 B→A。原实现先锁 fromAccount 再锁 toAccount，形成循环等待 → 死锁。

**修复** (`TransactionServiceImpl.java:329-334`):
```java
Long firstId = Math.min(fromAccountId, toAccountId);    // 较小的 ID 先锁
Long secondId = Math.max(fromAccountId, toAccountId);   // 较大的 ID 后锁
Account firstAccount = selectByIdForUpdate(firstId);    // 先锁小 ID
Account secondAccount = selectByIdForUpdate(secondId);   // 再锁大 ID
// 根据 from/to 还原引用 (锁顺序与业务方向解耦)
fromAccount = (fromAccountId.equals(firstId)) ? firstAccount : secondAccount;
toAccount = (toAccountId.equals(firstId)) ? firstAccount : secondAccount;
```

**原理**: 所有转账线程都按 ID 升序加锁，破坏死锁的「循环等待」必要条件（操作系统四大必要条件之一）。

### 11.3 MySQL InnoDB 死锁检测机制

#### 11.3.1 InnoDB 如何检测死锁

MySQL InnoDB 引擎内置了**死锁检测器（Deadlock Detector）**，使用 **wait-for graph（等待图）** 算法。当一个事务等待另一个事务持有的行锁超过一定时间，检测器会:

1. 构建事务间的等待关系图（有向图: 节点=事务，边=等待关系）
2. 在图中搜索环（cycle）——存在环 = 死锁
3. 选择一个事务作为 **victim** 回滚（通常是 undo log 最少的事务）以打破环路
4. 回滚的 transaction 收到 `Error 1213: Deadlock found when trying to get lock; try restarting transaction`

```
       ┌─────────┐
       │   T1    │ ← 环路！死锁
       └─────────┘
```

#### 11.3.3 日志中的关键字段解读

| 字段 | 含义 | 本例值 |
|---|:---|:---|
| `(1) TRANSACTION` / `(2) TRANSACTION` | 参与死锁的两个事务 | T1: 转 3→5, T2: 转 5→3 |
| `HOLDS THE LOCK(S)` | 该事务当前持有的锁 | T1 持有 id=3, T2 持有 id=5 |
| `WAITING FOR THIS LOCK TO BE GRANTED` | 该事务正在等待的锁 | T1 等待 id=5, T2 等待 id=3 |
| `WE ROLL BACK TRANSACTION (2)` | InnoDB 选择了哪个事务回滚 | T2 被回滚（undo log 更少） |
| `RECORD LOCKS` | 行级别锁 | `index PRIMARY` = 主键索引上的行锁 |

#### 11.3.4 本项目如何主动预防（而非依赖 InnoDB 死锁检测）

依赖 InnoDB 死锁检测是被动的——死锁已经发生，只是被检测到后回滚了一个事务。被回滚的事务需要重试，增加延迟和复杂性。本项目的处理方式:

1. **主动预防 > 被动检测**: §11.2 的 ID 升序加锁策略在源头消除死锁条件（破坏循环等待），InnoDB 的死锁检测器**永远不会被触发**
2. **验证方式**: 并发转账测试（JMeter 10 线程 × 100 循环 = 1000 笔并发转账）→ 零死锁日志 `SHOW ENGINE INNODB STATUS` 中无 `LATEST DETECTED DEADLOCK`
3. **兜底**: 即使极端情况下仍发生死锁，Spring `@Transactional` 默认 `rollbackFor = RuntimeException` —— InnoDB 抛 `MySQLTransactionRollbackException(1213)` → Spring 回滚 → GlobalExceptionHandler 返回 500 给前端 → 前端提示"转账失败，请重试"

#### 11.3.5 如何开启和查看 InnoDB 死锁日志

```sql
-- 1. 开启 InnoDB 监控（输出死锁日志到 MySQL error log）
SET GLOBAL innodb_print_all_deadlocks = ON;

-- 2. 查看最新一次死锁详情
SHOW ENGINE INNODB STATUS\G
-- 搜索 "LATEST DETECTED DEADLOCK" 段落

-- 3. 查看 MySQL error log 中的死锁历史记录
-- Windows: %PROGRAMDATA%\MySQL\MySQL Server 8.4\Data\%COMPUTERNAME%.err
-- Linux: /var/log/mysql/error.log

-- 4. 手动模拟死锁用于测试（两个 MySQL 客户端同时执行）:
-- 客户端1: BEGIN; SELECT * FROM account WHERE id = 3 FOR UPDATE; -- 持有 id=3
-- 客户端2: BEGIN; SELECT * FROM account WHERE id = 5 FOR UPDATE; -- 持有 id=5
-- 客户端1: SELECT * FROM account WHERE id = 5 FOR UPDATE;         -- 等待 id=5
-- 客户端2: SELECT * FROM account WHERE id = 3 FOR UPDATE;         -- 等待 id=3 → 死锁！
-- 结果: 其中一个客户端收到 ERROR 1213 (40001): Deadlock found
```

### 11.4 常见坑点与解决方案

**坑点 1: @Transactional 在同类自调用时不生效**

现象: `TransactionServiceImpl` 中某方法调用同一类中另一个带 `@Transactional` 的方法，预期后者以新事务执行，但实际与原方法共用同一事务。

根因: Spring AOP 是通过**代理**实现 `@Transactional` 的。同类自调用（`this.method()`）绕过了 Spring 代理，注解不生效。

解决方案: 转账流程仅在 Controller → Service 层面使用 @Transactional，不在 ServiceImpl 内部自调用需要独立事务的方法。如果确实需要（如预算预警的 `REQUIRES_NEW`），必须通过 Spring Bean 注入调用:
```java
// ❌ 同类自调用 — @Transactional 失效
this.processUserBudgetAlerts();

// ✅ 通过 Spring 代理调用
budgetAlertProcessorService.processUserBudgetAlerts();
```
详见 §15 P2-2 的 `BudgetAlertProcessorService` 架构分析。

**坑点 2: "其他"分类的歧义匹配**

现象: 转账自动选择的"其他"支出分类有时会匹配到多条记录，导致选择错误。

根因: `TransactionServiceImpl.java:347-350` 中 `findOtherCategory()` 用 `LambdaQueryWrapper` 按 `name = "其他"` 查询。如果种子数据被修改或存在多条名为"其他"的分类，查询结果非唯一。

解决方案: 已增加 `eq(Category::getType, CategoryType.EXPENSE)` 条件过滤，确保只在支出分类中匹配"其他"。同时种子数据 `sql/01-init.sql` 保证 `name = '其他'` 分类在支出类（type=1）中只有 1 条。

```javascript
export async function loadEcharts() {

    return await echartsPromise;

}
```

### 13.1 常见坑点与解决方案

**坑点 1: 改密后 token 未失效导致旧 token 仍可用**

现象: 用户修改密码后，之前的 JWT token 仍然有效（因为 JWT 是无状态的），任何人拿到旧 token 仍可访问系统。

根因: JWT 的设计决定了"一旦签发，在过期前始终有效"。本项目未引入 token 黑名单（需要 Redis 存储），所以改密后旧 token 不会立即失效。

解决方案与权衡:
- 当前策略: 接受这个风险（课堂项目、攻击面小）。JWT 有效期 7 天，token 仅存在用户浏览器的 localStorage，攻击者难以获取
- 增强方案（可选 P2 加分项）: 在 `user` 表增加 `token_version` 字段，改密时 `token_version++`；JWT payload 中包含 `tokenVersion`；拦截器校验时对比版本号
- 妥协方案: 前端改密成功后自动 `logout()` 清除 token → 跳登录页，确保当前设备上的旧 token 被清除

**坑点 2: BCrypt.matches() 耗时过长导致登录响应慢**

现象: 登录接口有时需要 500ms+ 才能返回，用户体感卡顿。

根因: BCrypt(12) 的验证时间约 200-400ms（取决于 CPU）。虽然单次验证可接受，但这是同步阻塞操作——后端线程在此期间无法处理其他请求。

解决方案: 当前 BCrypt cost=12 是安全性与性能的平衡点:
- cost=10: ~50ms 验证，安全性中等
- cost=12: ~250ms 验证，安全性高（本项目选择）
- cost=14: ~1s 验证，安全性极高但用户体验差
搭配 `LoginRateLimiter` 限流（每秒 2 次），即使 BCrypt 耗时高，也无法被用于 DDoS。

### 15.3 常见坑点与解决方案

**坑点 1: @Scheduled 定时任务在集群环境中重复执行**

现象: 如果部署了多个 Spring Boot 实例（如负载均衡），每个实例的 `@Scheduled` 都会在凌晨 2:00 执行预警检查，同一个用户的预警被插入多次。

根因: `@Scheduled` 是进程级别的，不存在跨进程协调。当 N 个实例同时运行时，N 个定时任务同时执行。

当前状态与建议:
- **当前状态**: 本项目为单体架构，仅 1 个 Spring Boot 实例，不存在此问题
- **未来扩展**: 如果升级为多实例部署，建议用 `@SchedulerLock`（ShedLock）或 Quartz 分布式锁防止重复执行
- **数据库幂等保护**: `BudgetAlertProcessorServiceImpl.java:80` 已实现"先 DELETE 后 INSERT"的幂等逻辑——即使多个实例同时执行，每个用户仅保留最后一次的预警记录

**坑点 2: Spring AOP 自调用导致 @Transactional(REQUIRES_NEW) 失效 — 详细分析**

> 这是本项目中最经典的一个 Spring AOP 陷阱，已在 Q-CR Loop1 修复。

原代码（有问题）:
```java
// BudgetScheduler.java (修复前)
@Scheduled(cron = "0 0 2 * * ?")
public void checkBudgetAlerts() {
    for (Long userId : userIds) {
        this.processUserBudgetAlerts(userId);  // ❌ 自调用！
    }
}

@Transactional(propagation = Propagation.REQUIRES_NEW)  // 此注解不生效
private void processUserBudgetAlerts(Long userId) { ... }
```

修复后:
```java
// BudgetScheduler.java (修复后) — 将处理方法提取为独立 Bean
@Service
public class BudgetAlertProcessorService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)  // ✅ 注解生效
    public void processUserBudgetAlerts(Long userId, ...) { ... }
}

// BudgetScheduler.java
@Scheduled(cron = "0 0 2 * * ?")
public void checkBudgetAlerts() {
    for (Long userId : userIds) {
        budgetAlertProcessorService.processUserBudgetAlerts(userId, ...);  // ✅ 通过代理调用
    }
}
```

**为什么自调用不生效**: Spring 在启动时为 `BudgetScheduler` 创建一个 JDK/CGLIB 代理。外部调用 `scheduler.checkBudgetAlerts()` 经过代理 → 代理触发事务切面。但方法内部的 `this.processUserBudgetAlerts()` 是直接字节码调用（Java 方法分派），完全绕过了代理。

**修复关键**: 将 `@Transactional(REQUIRES_NEW)` 的 method 提取到独立 Bean（`BudgetAlertProcessorService`），通过 Spring 注入 `@Autowired budgetAlertProcessorService`，调用 `budgetAlertProcessorService.processUserBudgetAlerts()` 时经过代理 → AOP 拦截 → REQUIRES_NEW 生效。

### 17.3 常见坑点与解决方案

**坑点 1: 汇率硬编码导致币种无法动态扩展**

现象: 用户想添加新币种（如 CAD 加元），但前端下拉框只有 6 种货币可选。

根因: `ExchangeRateServiceImpl.java:34-41` 的 `RATES` Map 是硬编码的 `static final` 常量。新增币种需要修改 Java 源码并重新编译部署。

当前设计取舍:
- 保持硬编码 6 种主流货币（USD/EUR/JPY/GBP/HKD/KRW），覆盖 95% 以上使用场景
- 如需动态扩展，改进方向:
  1. 将汇率配置移到数据库 `exchange_rate` 表或 `application.yml`
  2. 对接外部汇率 API（如 exchangerate-api.com）实现实时汇率
  3. 前端下拉框改为从后端动态获取可选币种列表

**坑点 2: 反汇率计算的精度累积误差**

现象: USD 的正向汇率是 `1 CNY = 0.138 USD`，反汇率 `1 / 0.138 = 7.2463...` 经过多次四舍五入后与真实汇率偏差增大。

根因: `ExchangeRateServiceImpl.java:56-63` computeInverseRates() 使用 `BigDecimal.divide()` + `HALF_UP` 四舍五入到 4 位小数。多次换算（USD→CNY→JPY→CNY）会产生累积误差。

解决方案: 反汇率统一使用 `MathContext.DECIMAL64` (16 位有效数字) 计算，仅在最终展示时四舍五入到 4 位。同时前端展示标注"仅供参考/非实时汇率"。

### 18.2 用户管理

**列表**: `AdminController.java:44 listUsers()` → `AdminServiceImpl.java:67 listAllUserDTOs()` → `UserDTO.fromUser(exclude password)`

**删除** (硬删除 + 级联清理):

2. 删除顺序 (按依赖关系): transaction → budget_alert → budget → recurring_bill → account → user
3. @Transactional 保证原子性

**角色切换**: `AdminServiceImpl.java:126 toggleUserRole()` → 翻转 role: 0→1 / 1→0，防止切换自己 (ADMIN_CANNOT_MODIFY_SELF 6002)

### 18.3 常见坑点与解决方案

**坑点 1: 删除用户时级联删除顺序错误导致外键约束冲突**

现象: 管理员删除用户时报 `Cannot delete or update a parent row: a foreign key constraint fails`。

根因: MySQL 外键要求先删除子表再删除父表。如果删除顺序不对（如先删 user 后删 transaction），MySQL 拒绝执行。

解决方案: `AdminServiceImpl.java:86 deleteUser()` 严格按依赖链反向顺序删除:
```
transaction → budget_alert → budget → recurring_bill → account → user
```
从叶子表到根表，确保每次 DELETE 时被引用方已清除。`@Transactional` 保证 6 步要么全成功要么全回滚。

**坑点 2: AdminInterceptor 与 LoginInterceptor 的中间件顺序**

现象: 未登录用户直接访问 `/api/v1/admin/users`，收到 `code: 6004 无管理员权限` 而非 `code: 401 未登录`。

根因: Spring 拦截器执行顺序由 `WebMvcConfig.addInterceptors()` 中的注册顺序决定。如果 `AdminInterceptor` 先于 `LoginInterceptor` 执行，`role` 属性为空 → 判断 `role != 1` → 直接抛 6004，未给 `LoginInterceptor` 机会提示用户先登录。

解决方案: `WebMvcConfig.java:54-72` 已按正确顺序注册——先 `LoginInterceptor` 再 `AdminInterceptor`。且 `AdminInterceptor.preHandle()` 先检查 `request.getAttribute("userId") != null`，如果 `LoginInterceptor` 未通过就提前 return true（放行到 401）而非抛 6004。

#### 23.5.3 覆盖率报告解读

#### 23.5.4 常见覆盖率陷阱

**陷阱 1: 高覆盖率不等于好测试**

```java
// ❌ 这个测试达到了 100% 覆盖率但毫无价值
@Test
void testCoverageOnly() {
    service.doSomething();
    // 没有任何断言 (assertion)
}
```

解决方案: 配合 §23.2 的白盒测试法 (基本路径法 + 逻辑覆盖法 + 边界值法) 确保每个测试都有 `assertNotNull`/`assertEquals`/`assertThrows` 等断言。覆盖率高但无断言的测试是"假绿"。

**陷阱 2: Lombok @Data 生成的 getter/setter 被计入覆盖率统计**

Lombok 自动生成的 `getName()/setName()` 等 boilerplate 方法在 JaCoCo 报告中会显示为"未覆盖"。可通过 JaCoCo 排除配置过滤:

```xml
<configuration>
    <excludes>
        <!-- 排除 Lombok 生成的 boilerplate -->
        <exclude>**/entity/**</exclude>
        <exclude>**/dto/**</exclude>
    </excludes>
</configuration>
```

### 25.5 JVM 内存调优建议

#### 25.5.1 JVM 参数默认值与建议值

```bash
# 查看当前 JVM 参数
java -XX:+PrintFlagsFinal -version | grep -i "HeapSize\|MetaspaceSize"

# Spring Boot 默认 JVM 参数（由 Spring Boot Maven Plugin 注入）:
# -Xms256m  -Xmx512m  (相对保守)
```

**调整建议** (在启动命令中传递，或写入 `application.yml` 的 `JAVA_TOOL_OPTIONS` 环境变量):

```bash
# === 开发环境 (2GB RAM 开发机) ===
java -Xms512m -Xmx1024m -jar system/backend/target/*.jar

# === 生产环境 (4GB+ RAM 服务器) ===
java \
  -Xms2048m \                          # 初始堆 2GB
  -Xmx2048m \                          # 最大堆 2GB (与 Xms 一致，减少 GC 开销)
  -XX:MaxMetaspaceSize=256m \          # 元空间上限 (防止 Metaspace OOM)
  -XX:+UseG1GC \                       # G1 垃圾回收器 (Java 21 默认)
  -XX:MaxGCPauseMillis=200 \           # 目标: GC 暂停 ≤ 200ms
  -XX:+PrintGCDetails \                # 输出 GC 日志 (生产可改为 -Xlog:gc*)
  -XX:+HeapDumpOnOutOfMemoryError \   # OOM 时自动 dump 堆 (分析用)
  -XX:HeapDumpPath=logs/heapdump.hprof \
  -jar system/backend/target/*.jar
```

#### 25.5.2 常见 JVM 问题与排查

| 症状 | 可能的 JVM 问题 | 排查命令 | 修复 |
|---|:---|:---|:---|
| 后端运行数小时后变慢，最终 OOM | 堆内存泄漏 (内存无法回收) | `jmap -heap <pid>` / 分析 heapdump | 检查 @Cacheable 缓存是否无限增长；调整 `-Xmx` |
| 频繁 Full GC（GC overhead limit exceeded） | 老年代空间不足 | `jstat -gc <pid> 1000` 查看 GC 频率 | 增大堆 (-Xmx) 或调优 GC 策略 |
| 启动后不到 1 分钟就 OOM | 堆初始值过小 | `jcmd <pid> VM.flags` 查看参数 | `-Xms` 至少设为 `-Xmx` 的一半 |
| `java.lang.OutOfMemoryError: Metaspace` | 元空间不足 (类加载过多) | `jstat -gc <pid>` 查看 MU/MC | 增加 `-XX:MaxMetaspaceSize=256m` |
| CPU 100% 但内存正常 | 死循环 / 线程阻塞 (非 GC 问题) | `jstack <pid>` + `top -H -p <pid>` 找高 CPU 线程 | 检查代码中的 while(true) / 死锁 |

## §28 系统局限性与改进方向

> **诚实声明**: 以下列出本系统在当前 v3.0 设计范围内的已知局限性，以及对应的改进方案。所有局限均为架构层面的有意取舍，非代码 Bug。

### 28.1 局限性清单

| # | 局限 | 影响范围 | 当前取舍理由 |
|:---:|---|:---|:---|
| 1 | **无分布式支持** | 系统为单机单体架构，无法水平扩展 | 课堂项目目标为单体全栈实践，分布式拆分会引入 RPC/消息队列/分布式事务等额外复杂度 |
| 2 | **无消息队列** | 定时任务 (预算预警) 直接在 JVM 内 `@Scheduled` 执行，无法异步削峰 | 预警处理量 < 100 用户，同步执行耗时 < 1s，消息队列在本次规模下是过度工程 |
| 3 | **前端无 E2E 测试** | 仅后端有 175 个单元测试，前端页面交互无自动化测试覆盖 | E2E 测试需要 Playwright/Cypress + CI/CD 基础设施，超出课堂项目时间预算 |
| 4 | **汇率硬编码** | 仅支持 6 种货币，汇率需手动更新源码 | 实时汇率 API 需要外网访问 + API key 管理，课堂内网环境受限 |
| 5 | **无 CI/CD 流水线** | 每次部署需手动执行 `mvn package` + `pnpm build`，无自动构建和部署 | CI/CD 需要 GitHub Actions/GitLab CI 配置 + 部署服务器，当前手动流程满足 P8 部署需求 |

### 28.2 改进方案 (按优先级排序)

#### 28.2.1 P0 (技术面试加分 — 建议答辩前了解)

**局限 1: 无分布式支持 → 微服务化**

```
当前架构:
  [Browser] → [单体 SpringBoot App] → [MySQL]

改进架构:
  [Browser] → [API Gateway (Spring Cloud Gateway)]
    ├─→ [user-service]     (用户注册/登录/JWT)
    ├─→ [account-service]   (账户 CRUD + 余额计算)
    ├─→ [transaction-service] (交易 CRUD + 转账 + CSV)
    ├─→ [budget-service]    (预算管理 + 预警)
    ├─→ [statistics-service] (统计 + ECharts 数据)
    └─→ [notification-service] (周期账单提醒)
  各 service → 独立 MySQL schema / 数据库
  服务发现: Nacos / Consul
  配置中心: Nacos Config
  分布式事务: Seata (AT 模式 或 TCC 模式用于转账)
```

关键技术点:
- 转账从单体 `@Transactional` 变为 TCC 分布式事务 (Try-Confirm-Cancel)
- JWT 在 Gateway 层统一校验，内部服务间用 `X-User-Id` Header 传递身份
- API 文档从静态 Markdown 升级为 Swagger/OpenAPI 3.0 自动生成

**局限 2: 无消息队列 → 引入 RabbitMQ**

```
当前:
  BudgetScheduler → BudgetAlertProcessorService (同步)

改进:
  定时任务 → RabbitMQ Topic `budget.check` → Consumer 集群消费
  优势:
  1. 削峰: 1000 用户预警分批消费，不阻塞定时任务线程池
  2. 重试: 消费失败自动 NACK → 死信队列 → 人工处理
  3. 解耦: 未来可扩展"推送通知"消费者 (邮件/钉钉/微信)
```

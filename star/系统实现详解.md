## 第一部分: 系统启动全过程 (Runtime)

> 模拟从 `java -jar` 到系统就绪的全部过程。以时间线推进，展示 JVM 内部、Spring 内部、Tomcat 内部、浏览器内部真正发生的事。

### T=0ms: JVM 启动

```
操作系统 (Windows 11): 用户在终端执行 java -jar backend.jar
  │
  ├── CreateProcess() → 创建新 Windows 进程
  │   ├── 进程 ID: 被 OS 分配, 如 12345
  │   ├── 虚拟地址空间: 4GB (32位) 或 128TB (64位)
  │   └── 主线程创建: "main" 线程
  │
  ├── java.exe → JVM 动态库 (jvm.dll) 加载
  │   ├── 解析 JVM 参数: -Xms256m -Xmx512m (从 MAVEN_OPTS 或默认值)
  │   ├── 堆内存分配: 初始 256MB / 最大 512MB
  │   │   ├── Young Gen: ~85MB (Eden + S0 + S1)
  │   │   └── Old Gen: ~171MB
  │   ├── Metaspace: 默认无上限 (用本地内存)
  │   ├── 类加载器层次结构:
  │   │   ├── Bootstrap ClassLoader (加载 java.lang.*, rt.jar 等)
  │   │   ├── Platform ClassLoader (JDK 9+, 原 ExtClassLoader)
  │   │   └── App ClassLoader (加载 classpath 中的类)
  │   └── GC 初始化: G1GC (JDK 21 默认)
  │
  └── JVM 启动完成 → 调用 Application.main(String[] args)
```

**此刻 JVM 内存中的关键对象**:
- 还没有任何 Spring Bean
- 只有 JVM 基础类: String, Object, ClassLoader, Thread, System
- 类路径已确定: target/classes/ + ~/.m2/repository/**/*.jar
- main 线程: Thread[id=1, name="main", priority=5, daemon=false]

## 第五部分: Filter / Interceptor / 安全 Runtime 分析

### 5.1 Servlet Filter vs Spring Interceptor: 执行时机

```
HTTP Request 进入 Tomcat 后:
  │
  ├── [Servlet Filter Chain] ← 最早, Servlet 容器级别
  │   │
  │   ├── Filter 1: CorsFilter (来自 CorsConfig.corsFilter())
  │   │   位置: CorsConfig.java:73-88
  │   │   执行: CorsFilter.doFilter(request, response, chain)
  │   │   职责: 检查 Origin → 设置 CORS 响应头 → chain.doFilter() 继续
  │   │   为什么放 Filter: CORS 是 HTTP 协议层面的事, 需要在 Spring MVC 之前处理
  │   │
  │   ├── Filter 2: CharacterEncodingFilter (Spring Boot 内置)
  │   │   职责: request.setCharacterEncoding("UTF-8") → response.setCharacterEncoding("UTF-8")
  │   │
  │   └── Filter 3: (未使用 Spring Security → 无额外 Filter)
  │       │
  │       └── chain.doFilter() → 最后进入 DispatcherServlet
  │           │
  │           └── [Spring Interceptor Chain] ← 较晚, Spring MVC 级别
  │               │
  │               ├── Interceptor 1: LoginInterceptor
  │               │   执行: preHandle() → Controller → postHandle() → afterCompletion()
  │               │   为什么放 Interceptor: 需要知道目标 Handler (Controller+Method)
  │               │   为什么 Filter 不够: Filter 不知道请求将去哪个 Controller
  │               │
  │               └── Interceptor 2: AdminInterceptor
  │                   仅拦截: /api/v1/admin/**
  │                   依赖: LoginInterceptor 先执行 (userId + role 已注入 request attribute)
  │
  └── 关键区别表:
      | 维度 | Filter | Interceptor |
      |------|--------|-------------|
      | 容器 | Servlet (Tomcat) | Spring MVC |
      | IOC | 不可直接注入 Bean | 可 @Autowired |
      | 处理 | request/response | 可修改 Handler |
      | 排除 | web.xml 或 FilterRegistrationBean | WebMvcConfigurer.excludePathPatterns |
      | 适用 | CORS, 编码, 安全头 | 认证, 授权, 日志, 性能 |
```

### 5.2 JWT 为什么不在 Spring Security Filter 中

```
如果使用 Spring Security (本项目未使用):
  SecurityFilterChain 会插入到 Filter Chain 中:
    CorsFilter → SecurityFilterChain → DispatcherServlet

  本项目不使用 Spring Security → 没有 SecurityFilterChain
  原因:
    1. Spring Security 默认返回 HTTP 401/403 → 与 body-code-first 冲突
    2. 需要额外配置 AuthenticationEntryPoint 改为 HTTP 200 + body code
    3. 需要配置 SecurityContextHolder 的 ThreadLocal 策略
    4. 7 层 Security Filter 对于仅需 JWT+role 的场景过重

  替代方案 (本项目选择):
    LoginInterceptor + request.setAttribute("userId", ...) + request.setAttribute("role", ...)
    → Controller 通过 LoginInterceptor.getUserId(request) 获取
    → 不依赖 SecurityContextHolder → 更简单, 线程安全
```

### 5.3 SecurityContext 传播机制 (对比)

```
Spring Security (本项目未用):
  SecurityContextHolder.getContext() → ThreadLocal<SecurityContext>
    → SecurityContext.getAuthentication() → Authentication
    → Principal, Credentials, Authorities
  传播方式: 必须在 ThreadLocal 生命期内使用
  问题: 异步线程 (@Async) 无法访问父线程的 SecurityContext
  解决: MODE_INHERITABLETHREADLOCAL 或 DelegatingSecurityContextRunnable

本项目方式:
  LoginInterceptor → request.setAttribute("userId", userId)
    → Controller → userId → Service → userId
  传播方式: 通过方法参数显式传递
  优势: 不依赖 ThreadLocal → 异步调用不会丢失上下文
  劣势: 每个 Service 方法都需要 userId 参数 (但这是显式的好处)
```

### 5.4 ThreadLocal 机制详解

```
为什么 request.setAttribute 而不是 ThreadLocal:
  ├── Tomcat 线程模型:
  │   ├── 每个 HTTP 请求 → 分配 1 个 Worker 线程
  │   ├── 线程处理完请求 → 回到线程池 → 等待下一个请求
  │   └── 同一个线程可能先后处理不同用户的请求!
  │
  ├── 如果用 ThreadLocal 存储 userId:
  │   T1 处理请求 A (userId=1) → ThreadLocal.set(1)
  │   T1 处理完 → 回到池 → 但 ThreadLocal 没清理!
  │   T1 处理请求 B (userId=2) → ThreadLocal.set(2)
  │   问题: 如果请求 A 的清理代码在请求 B 设置后执行 → 读到 userId=2
  │
  └── 如果用 request.setAttribute:
      每个请求有独立的 HttpServletRequest 实例 → 天然隔离
      不需要手动清理 → 请求结束后 GC 回收
```

### 6.2 Bean 单例 vs 多例

```
本项目所有 Bean 默认单例 (Spring 默认):
  ✅ @Service → 单例 (无状态, 线程安全)
  ✅ @Controller → 单例 (无状态, 线程安全)
  ✅ @Component → 单例 (LoginInterceptor 持有 ObjectMapper, 线程安全)
  ✅ Mapper 代理 → 单例 (SqlSessionTemplate 线程安全)

为什么单例:
  1. 无状态 Bean = 天然线程安全 → 不需要每次 new
  2. 减少 GC 压力 (不需要频繁创建/销毁)
  3. Spring 的默认值 → 最小配置

什么情况下需要多例:
  有状态的 Bean (持有用户相关状态) → 本项目没有这种场景
  所有"用户特定"状态通过方法参数 userId 传递 → 不存储在 Bean 字段中

如果 Mapper 不是单例:
  每次调用都创建新的 MapperProxy → 每次创建 SqlSession → 极大的性能开销
  且 MyBatis 的 SqlSession 非线程安全 → 多例反而可能引入 bug
```

### 6.3 AOP 代理工作机制

```
CGLIB 代理工作原理 (以 UserServiceImpl 为例):

public class UserServiceImpl$$SpringCGLIB$$0 extends UserServiceImpl {
    private TransactionInterceptor transactionInterceptor;

    public LoginResponse register(UserLoginRequest request) {
        // 代理拦截
        return transactionInterceptor.invoke(() -> {
            // → super.register(request)  ← 调用原始方法
            // 在调用前后: 开启/提交事务
        });
    }
}

运行时:
  1. Controller 调用 userService.login(request)
     → 实际调用的是 UserServiceImpl$$SpringCGLIB$$0.login(request)
  2. 代理的 login() → TransactionInterceptor.invoke()
     → getTransaction() → 从 DataSource 获取连接 → BEGIN
  3. 调用 super.login(request) → UserServiceImpl 原始代码执行
  4. 返回后 → commit() 或 rollback() → 释放连接

为什么自调用 (this.method()) 不经过代理:
  this 引用指向的是原始对象 UserServiceImpl (非代理)
  → this.method() 是直接字节码调用 → 跳过代理
  → @Transactional 失效

为什么注入调用 (bean.method()) 经过代理:
  bean 引用指向的是 Spring 注入的 CGLIB 代理对象
  → bean.method() 经过代理的 invoke()
  → @Transactional 生效

验证方式:
  IntelliJ Debug → 查看 userService 变量 → 类型显示为 "UserServiceImpl$$SpringCGLIB$$0"
```

```
1. HttpServletRequest (Tomcat 创建):
   线程-3: 接收请求 → 创建 RequestFacade@a101
   → 整个请求处理期间, 同一个 request 对象贯穿全程

2. LoginInterceptor → request.setAttribute("userId", 1L):
   request.attributes Map:
     "userId" → Long@b001 { value=1 }
     "role" → Integer@b002 { value=0 }

3. Controller 读取:
   LoginInterceptor.getUserId(request)
   → request.getAttribute("userId") → Long@b001

4. Service 方法:
   通过方法参数传递:
   transactionService.create(userId, request) → userId = 1L

5. Mapper SQL:
   WHERE user_id = #{userId} → userId = 1L
   → PreparedStatement.setLong(1, 1L)

关键安全保证:
  从 request 中提取 userId → 通过参数链传递到 SQL
  userId 从 JWT token 解析 (不可伪造, HMAC 签名验证)
  不是从 HTTP 请求参数获取 (防止越权: userId=1 的用户不能传 userId=2)
```

### 9.3 事务上下文传播

```
@Transactional 事务传播链:

调用链: Controller → userService.login() → 返回
  → 此时 transactionService.create() → 独立事务

TransactionSynchronizationManager (ThreadLocal):
  ├── resources: Map<DataSource, ConnectionHolder>
  │   └── HikariDataSource → ConnectionHolder@f001
  │       └── connection: Connection@a003 (从 HikariCP 借出)
  │       └── transactionActive: true
  ├── currentTransactionName: "com.example.finance.service.impl.UserServiceImpl.login"
  ├── currentTransactionReadOnly: true
  └── actualTransactionActive: true

PlatformTransactionManager.commit():
  1. DataSourceTransactionManager.doCommit()
  2. connection.commit() → MySQL COMMIT
  3. TransactionSynchronizationManager.clear() → 清理 ThreadLocal
  4. HikariCP: connectionBag.requite(Connection@a003)

REQUIRES_NEW 传播:
  TransactionAspectSupport:
    → 暂停当前事务 (savepoint 或 suspend connection)
    → 获取新的 Connection (Connection@a004)
    → 新事务开始
    → 新事务 COMMIT
    → 恢复暂停的事务 (resume)
```

## 第十一部分: 工程设计思想

> 不要只解释"这是什么"。要解释"为什么这样设计"和"如果不这样设计会怎么样"。

### 11.1 为什么分层架构

**分层不是为了"看起来整齐"。分层是为了控制变化的影响范围。**

```
如果在 Controller 里写 SQL:
  1. 改了一个 SQL → 可能不小心改了业务逻辑
  2. 为前端加了一个字段 → 需要改 Controller (HTTP 层) → 但 SQL 层也跟着变了
  3. 换数据库 (MySQL → PostgreSQL) → 需要改每一个 Controller

分层后:
  Controller 只关心 HTTP → 换 HTTP 库 (REST → GraphQL) → 只改 Controller
  Service 只关心业务规则 → 改业务规则 → 只改 Service
  Mapper 只关心 SQL → 换数据库 → 只改 Mapper + Entity

本质: 每一层是对变化的"防火墙"。一层的修改不传播到另一层。
```

### 11.2 为什么前后端分离

```
前后端不分离 (JSP/Thymeleaf):
  改了一个 UI 按钮颜色 → 需要重新编译后端 → 部署整个 war 包
  前端需要一个新接口 → 等后端开发完成 → 联调 → 部署

前后端分离:
  改 UI → pnpm build → 替换 nginx 下的静态文件 → 秒级发布
  新接口 → 后端提供 API → 前端 mock 数据并行开发
  本质: 解耦发布周期。前端可以独立于后端发布。
```

### 11.3 为什么 DTO/VO 分离

```
如果直接返回 Entity:
  1. User 有 password 字段 → 即使加了 @JsonIgnore, 换序列化库可能不管用
  2. Entity 随数据库表变化 → 改了数据库就要改 API 接口 → 前后端耦合
  3. 无法跨 Entity 组合 → Transaction 只有 categoryId, 前端需要 categoryName

DTO 的本质:
  DTO = API 接口契约 → 与数据库表结构独立
  Entity = 数据库表映射 → 与 API 接口独立
  两者各自演化 → 互不影响
```

### 11.4 为什么 JWT 无状态

```
Session (有状态):
  用户登录 → 服务器创建 session → sessionId → cookie
  优点: 服务器可控 (可以随时踢人)
  缺点: 需要存储 session (内存/Redis) → 水平扩展需要 session 共享

JWT (无状态):
  用户登录 → 服务器签发 token → 客户端存储
  优点: 服务器不需要存储任何会话数据 → 天然水平扩展
  缺点: 服务器无法主动踢人 (token 过期前始终有效)

本项目选择 JWT:
  单体应用 → 不需要水平扩展 → 但 JWT 更简单
  如果未来升级微服务 → JWT 天然支持 → 不需要改认证机制
```

### 11.5 为什么 IOC 存在

```
没有 IOC:
  UserController 需要 UserService:
    UserController() { this.userService = new UserServiceImpl(); }
  问题: Controller 和 ServiceImpl 硬绑定 → 无法替换实现 → 无法 mock 测试

有 IOC:
  UserController(UserService userService) { this.userService = userService; }
  → Spring 注入任何实现了 UserService 的 Bean
  → 测试时注入 mock → 单元测试不需要启动整个应用

IOC 的本质:
  控制反转 = 对象不控制自己的依赖 → 依赖由外部 (Spring) 注入
  结果: 低耦合 → 高可测试性 → 高可替换性
```

### 11.6 为什么 Repository/Mapper 模式存在

```
如果 Service 直接写 JDBC:
  Service 方法中 try { connection = getConnection(); stmt = ... } finally { close }
  → 每个 Service 方法都要管理连接 → 大量模板代码

Mapper 模式:
  Service 调用 userMapper.selectById(id) → MyBatis 自动处理连接管理
  → Service 只关心业务逻辑 → 不关心 JDBC 细节

本质: 抽象了数据访问层 → 隔离了基础设施细节 (JDBC 连接管理, 事务, 连接池)
```

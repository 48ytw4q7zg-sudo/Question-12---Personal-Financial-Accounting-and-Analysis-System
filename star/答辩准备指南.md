## S000 设计模式使用清单

| 设计模式 | 位置 | 为什么用 / 解决了什么 |
|---|---|:---|
| **模板方法模式** | `GlobalExceptionHandler` extends `ResponseEntityExceptionHandler` | 覆盖 Spring 父类的 12 种异常处理方法，统一为 Result<T> 格式，避免每个 Controller 写 try-catch |
| **策略模式** | `ErrorCode` enum (28 个错误码) | 集中管理 6 个模块的错误码与消息，Service 层直接引用枚举常量而非硬编码字符串，码值统一可维护 |
| **单例模式** | `LoginRateLimiter` (static ConcurrentHashMap + static methods) | 全局唯一限流器，所有登录请求共享同一 RateLimiter 实例 |
| **工厂方法模式** | `Result.success() / Result.error()` 静态工厂 | 统一创建 Result 对象，隐藏构造细节，前端 axios 拦截器据此解析 |
| **代理模式** | Spring AOP `@Transactional` + `BudgetScheduler` | 声明式事务管理（JDK 动态代理），REQUIRES_NEW 独立事务隔离单用户预警失败 |
| **拦截器链模式** | `LoginInterceptor` -> `AdminInterceptor` (WebMvcConfig 注册) | 责任链处理 HTTP 请求：JWT 认证 -> 管理员鉴权 -> Controller，每个拦截器关注单一职责 |
| **DTO 模式** | `entity/dto/` 26 个 DTO 类 | 解耦 Entity 与 API 响应，防止 @JsonIgnore 字段泄漏、防止循环引用、数据库字段变更不直接暴露给前端 |
| **依赖注入** | `@RequiredArgsConstructor` (全项目) | Lombok + Spring IoC 构造器注入，消除 @Autowired 字段注入的紧耦合，便于单元测试 mock |
| **门面模式** | `EntityValidator.java` (3 个 Service 共享) | 提取 AccountServiceImpl/TransactionServiceImpl/RecurringBillServiceImpl 中重复的 validateAccount/validateCategory，一处修改三处生效 |
| **缓存/懒加载** | `echarts-lazy.js` (单例 Promise) | 首次加载 ECharts 核心后缓存，后续调用零延迟，且失败后清除缓存可重试 |
| **建造者/组合模式** | `LambdaQueryWrapper<User>().eq(...)` | MyBatis-Plus 的链式 API 构建 SQL 条件，类型安全 + IDE 补全 |
| **观察者/调度模式** | `@Scheduled` (BudgetScheduler) | Spring 定时任务，每日凌晨 2:00 自动检查预算预警，解耦定时触发与业务处理 |
| **路由守卫模式** | `router.beforeEach` (Vue Router) | 前置钩子集中鉴权：Token 过期预检 -> 需登录拦截 -> 管理员拦截 -> Chunk 加载失败容错，6 步一个入口 |

### Q3: 怎么处理异常？

**三层全局异常处理**:

1. **Service 层**: `throw new BusinessException(code, msg)` (定义在 `common/BusinessException.java`)
2. **GlobalExceptionHandler**: `@RestControllerAdvice` 继承 `ResponseEntityExceptionHandler`，覆盖 12 种异常处理，统一转为 `Result.error(code, msg)` (定义在 `common/GlobalExceptionHandler.java`)
3. **前端**: axios 响应拦截器统一处理 `code !== 200`，ElMessage.error 弹窗

**body-code-first 约定**: HTTP 状态码始终 200，业务状态码放在 JSON body 的 code 字段。前端拦截器检查 `res.data.code`，与 LoginInterceptor 保持一致。

### Q5: 登录/JWT 怎么实现的？为什么不区分"用户不存在"和"密码错误"？

**完整流程**:
1. 前端 `LoginPage.vue:142` -> `api/user.js:43` -> `POST /api/v1/user/login`
2. 后端 `UserServiceImpl.java:97 login()`:
   - Guava RateLimiter 限流 (`LoginRateLimiter.java:61` . 每秒 2 次)
   - `userMapper.selectOne(username)` 查询 -> 不存在或密码错误统一返回 1002
   - `BCRYPT_ENCODER.matches(password, hash)` 验证 (cost=12)
   - `JwtUtils.generateToken()` 签发 (HS256, 7天)
3. 前端 `stores/user.js:59 setUser()` -> `localStorage.setItem('token', token)`

**为什么不区分**: 防用户名枚举攻击。如果返回"用户名不存在"和"密码错误"两个不同错误码，攻击者可以枚举出系统中已注册的用户名，为下一步暴力破解提供信息。

**代码证据**: `UserServiceImpl.java:106` -- `if (user == null || !BCRYPT_ENCODER.matches(...))` -- 一行判断同时覆盖两种情况，统一抛出 `ErrorCode.PASSWORD_ERROR(1002, "用户名或密码错误")`。

### Q6: JWT token 里有什么？为什么能防篡改？

**Payload 包含**: `sub`(userId) + `username` + `role` + `exp`(7天) + `iat`(签发时间)

**防篡改原理**: JWT 使用 **HMAC-SHA256 (HS256)** 签名。`JwtUtils.java:79-89 generateToken()` -> JJWT 库用 256-bit 密钥对 header+payload 做 HMAC 签名。任何人修改 payload 后签名不再匹配 -> 后端 `parseTokenPayload()` 验签失败 -> 返回 null -> LoginInterceptor 返回 401。

**性能优化**: `parseTokenPayload()` (第 98-117 行) **一次 HMAC 验签**同时提取 userId + username + role，消除旧实现 `parseToken()` + `parseRole()` **两次验签**的 50% 冗余密码学运算。

| 环节 | 文件 | 行号 | 证据 |

| 密钥注入 | `config/JwtConfig.java` | 80-98 | `@Value("${jwt.secret}")` 从环境变量注入 |
| 密钥校验 | `config/JwtConfig.java` | 80-98 | 长度 < 32 字节拒绝启动 |
| Token 生成 | `util/JwtUtils.java` | 79-89 | `Jwts.builder().signWith(HS256)` |

### Q7: 收支记录怎么分页？怎么筛选？

**后端**: `TransactionMapper.xml:25-55` -- MyBatis XML 动态 SQL:
- `selectTransactionList`: JOIN account + category (填充名称), RowBounds 物理分页, 动态 ORDER BY
- `selectTransactionCount`: 独立 COUNT(*), 与列表查询**共用** `<include refid="transactionFilterConditions"/>` (DRY 原则)
- 筛选条件: accountId/categoryId/startTime/endTime/keyword (LIKE %keyword%)

### Q9: 转账怎么实现？怎么防止并发死锁？

**核心流程** (`TransactionServiceImpl.java:302-400 transfer()`):
1. 校验: 金额>0, fromAccount!=toAccount
2. **死锁预防**: 按账户 ID **升序**加悲观锁 (第 329-334 行)
3. 余额校验: `initialBalance + 收入 - 支出 >= amount`
4. 查询"其他"支出分类 -> 生成 UUID transferId
5. `@Transactional` 内: INSERT 转出(支出) + INSERT 转入(收入)
6. 返回 `TransferDTO(transferId, outRecord, inRecord)`

**死锁预防原理**: 线程1转账 A->B、线程2转账 B->A：
- **旧实现** (有死锁): 线程1锁A等B, 线程2锁B等A -> 循环等待 -> 死锁
- **新实现** (零死锁): 都先锁 min(A,B) 再锁 max(A,B) -> 同样顺序 -> 无循环等待
- 这是操作系统课本"破坏死锁必要条件"的经典解法

**转账保护**:
- transfer_id 非空的记录**禁止修改金额** (error 3006)
- transfer_id 非空的记录**禁止单独删除** (error 3007, 保持配对完整性)

### Q12: 转账死锁预防的完整原理是什么？代码怎么实现的？

**问题建模**: A 和 B 是两个账户，线程1从 A 转账到 B (锁A后锁B)，线程2从 B 转账到 A (锁B后锁A)。

**死锁条件分析** (操作系统四大必要条件):
1. 互斥: `SELECT ... FOR UPDATE` 获取行级排他锁 -- 满足
2. 持有并等待: 线程1持有A的锁等待B，线程2持有B的锁等待A -- 满足
3. 不可剥夺: InnoDB 行锁不可被其他事务强制释放 -- 满足
4. 循环等待: A->B 与 B->A 形成闭环 -- 满足

**修复策略**: 破坏第 4 条"循环等待"。无论 from/to 哪个 ID 更小，所有线程都先锁 `min(A,B)` 再锁 `max(A,B)`，全局统一的加锁顺序。

**代码实现** (`TransactionServiceImpl.java:328-335`):
```java
Long firstId = Math.min(fromAccountId, toAccountId);   // 较小ID先锁
Long secondId = Math.max(fromAccountId, toAccountId);  // 较大ID后锁
Account firstAccount = accountMapper.selectByIdForUpdate(firstId);
Account secondAccount = accountMapper.selectByIdForUpdate(secondId);
// 还原 from/to 引用（锁顺序与业务方向解耦）
Account fromAccount = (fromAccountId.equals(firstId)) ? firstAccount : secondAccount;
Account toAccount = (fromAccountId.equals(firstId)) ? firstAccount : secondAccount;
```

**为什么用悲观锁不用乐观锁**: 转账是低频但强一致性场景。乐观锁(version字段)在并发转账时会反复重试，用户体验差且浪费DB资源。悲观锁一次锁定、一次成功。

**为什么不用分布式锁**: 本项目是单实例 SpringBoot + 单 MySQL，无需引入 Redis/Redisson 等额外组件。行级 SELECT FOR UPDATE 已足够。

## S6 扩展加分项答辩

### Q20: 多币种怎么支持的？

**六种货币**: CNY/USD/EUR/JPY/GBP/HKD (定义在 `ExchangeRateServiceImpl.java:34-41`)

**汇率双表**:
- 正向汇率: 1 CNY -> X 外币 (如 `CNY->USD = 0.1379`)
- 反向汇率: **自动计算** `1 / forwardRate` (数学一致性, 无需手动维护)

**余额换算** (`AccountServiceImpl.java:264`): `cnyEquivalent = currentBalance * rateInverse`

**前端提示**: DashboardPage 检测非 CNY 账户 -> 显示"多币种账户存在，统计金额为各币种原始值，未做CNY换算"提示条。

### Q21: ECharts 怎么做按需加载的？

**`utils/echarts-lazy.js`**: 单例 Promise 缓存 + 按需 import。
- 首次调用 -> `import('echarts/core')` + `Promise.all([import('echarts/charts'), import('echarts/components'), import('echarts/renderers')])` -> `echarts.use([...])` 注册
- 后续调用 -> 直接返回缓存的 Promise
- 失败后清除缓存，下次可重试 (不会永久损坏)

**导入组件**: PieChart / BarChart / LineChart + Grid + Tooltip + Legend + Title + CanvasRenderer (仅 8 个，非全量 50+ 组件)

**体积对比**: 全量 ~1MB -> 按需 ~350KB (减小 65%)

## S7 高频对比问题

### Q22: MyBatis-Plus vs 原生 MyBatis? 为什么选 LambdaQueryWrapper?

**MyBatis-Plus 优势**:
1. `BaseMapper` 内置 CRUD (insert/selectById/updateById/deleteById), 减少重复代码
2. `LambdaQueryWrapper` 编译期类型安全, 字段引用 `User::getUsername` 而非字符串 `"username"`, IDE 重构字段名时自动同步
3. 分页插件 `PaginationInnerInterceptor` 一行配置即可

**为什么不全部用 XML**: 项目遵循 CLAUDE.md S二.四 规范--简单查询用 LambdaQueryWrapper，只有多表 JOIN / GROUP BY / 动态 ORDER BY 等复杂场景才写 XML。

**用法示例** (来自 `AccountServiceImpl.java:78-84`):
```java
accountMapper.selectList(
    new LambdaQueryWrapper<Account>()
        .eq(Account::getUserId, userId)       // 编译期校验字段存在
        .eq(Account::getStatus, 1)            // 活跃账户
        .orderByDesc(Account::getCreateTime)  // 倒序
);
```

### Q23: Vue 3 Composition API 比 Options API 好在哪？

1. **逻辑复用**: 可提取独立的 composable 函数 (如 `logger('DashboardPage')`)
2. **类型推导**: `<script setup>` + TypeScript 配合更好
3. **代码组织**: 相关逻辑自然聚合 (不像 Options API 按 data/methods/mounted 分散)
4. **Tree-shaking**: 未使用的 API 在构建时被 Vite 移除

**本项目全用 `<script setup>` 写法** (对齐 CLAUDE.md S三.二)。

### Q25: 事务管理怎么处理的？隔离级别是什么？

**Spring Boot 声明式事务**: `@Transactional` (由 MybatisPlusConfig @EnableTransactionManagement 启用)

**使用场景**:

| 场景 | 事务类型 | 隔离级别 | 代码位置 |
|---|:---:|---|---|
| 创建交易 (记一笔) | @Transactional | InnoDB 默认 REPEATABLE READ | `TransactionServiceImpl.java:176` |
| 更新交易 | @Transactional | REPEATABLE READ | `TransactionServiceImpl.java:219` |
| 删除交易 | @Transactional | REPEATABLE READ | `TransactionServiceImpl.java:279` |
| 转账 (双 INSERT) | @Transactional | REPEATABLE READ | `TransactionServiceImpl.java:310` |
| CSV 导入 | @Transactional | REPEATABLE READ | `TransactionServiceImpl.java:430` |
| 预算保存 (并发安全) | @Transactional | REPEATABLE READ | `BudgetServiceImpl.java:129` |
| 交易列表查询 | @Transactional(readOnly=true) | REPEATABLE READ | `TransactionServiceImpl.java:124` |
| 预算进度查询 | @Transactional(readOnly=true) | REPEATABLE READ | `BudgetServiceImpl.java:241` |
| 预警处理 (独立事务) | @Transactional(REQUIRES_NEW) | REPEATABLE READ | `BudgetAlertProcessorServiceImpl` |

**隔离级别选择理由**: MySQL InnoDB 默认 REPEATABLE READ 即可满足需求。转账场景下，事务内 SELECT 余额和 INSERT 共享同一快照，配合悲观锁 (`SELECT ... FOR UPDATE`) 可完全防止并发透支。无需升级到 SERIALIZABLE (性能损失大) 或降级到 READ COMMITTED (可能幻读)。

**传播行为**: 大部分场景使用默认 REQUIRED。只有 BudgetScheduler 中单用户预警处理使用 REQUIRES_NEW，确保 A 用户的预算预警失败不回滚 B 用户的预警记录。

**AOP 自调用问题**: BudgetScheduler 内部原先直接 `this.processUserBudgetAlerts()` 导致 REQUIRES_NEW 失效 (Spring AOP 代理自调用穿透)。修复方案: 提取为独立的 `BudgetAlertProcessorServiceImpl`，通过注入的 Spring 代理调用。

### Q26: 并发控制怎么做的？悲观锁 vs 乐观锁的选择？

**悲观锁 (`SELECT ... FOR UPDATE`)**:

适用场景:
- 转账: 两个账户余额必须同时锁定，`accountMapper.selectByIdForUpdate()` 获取行级排他锁
- 理由: 转账是强一致性操作，不允许乐观冲突重试

**乐观锁 (DuplicateKeyException 捕获)**:

适用场景:
- 预算保存: 先 insert，并发冲突时 catch DuplicateKeyException -> 回退 updateById
- 理由: 预算设置是低冲突场景 (一个用户一月同一分类只设置一次)

**无锁 (快照读)**:

适用场景:
- 收支列表查询: `@Transactional(readOnly=true)` + InnoDB MVCC 快照读
- 理由: 查询类操作，利用 MVCC 实现无锁一致性读取

#### 代码证据表

| 并发策略 | 文件 | 行号 | 场景 |
|---|:---|:---:|---|
| SELECT FOR UPDATE | `TransactionServiceImpl.java` | 329-335 | 转账悲观锁 |
| DuplicateKeyException | `BudgetServiceImpl.java` | 176-192 | 预算并发保存 |
| MVCC 快照读 | `TransactionServiceImpl.java` | 124 | 列表查询 readOnly=true |

### Q27: 缓存策略是什么？Cacheable 怎么用的？

**Spring Cache + @Cacheable**:

1. **分类列表缓存** (`CategoryServiceImpl.java:63`): 13 条种子数据几乎不变，标注 `@Cacheable("categories")`。首次查询后缓存到内存，后续请求直接返回缓存，无需查 DB。

2. **ECharts 懒加载缓存** (`echarts-lazy.js`): 单例 Promise 全局变量，首次 `import('echarts/core')` 后缓存，后续调用直接返回。失败时清除缓存允许重试。

**缓存策略考量**:
- 分类数据低频变更，适合长期缓存
- 不使用 Redis (教学项目，减少组件依赖)
- 不使用 Caffeine (内置 ConcurrentHashMap + @Cacheable 足够)

### Q28: 前端的错误处理和加载状态怎么设计的？

**Axios 拦截器统一处理** (`request.js`):
- `code === 200`: 正常返回 `res.data`
- `code === 401`: 清除 token + 跳 `/login` + `ElMessage.error('未登录')`
- 其他 code: `ElMessage.error(msg)` + reject

**页面级错误处理**:
- try-catch 仅在组件需要**业务关联反馈**时使用 (如"用户名已存在"在表单字段上标记红色)
- 通用错误由拦截器统一处理，组件不再判断 `if (res.code !== 200)`

**加载状态设计**:
- `el-table v-loading`: 数据加载时表格自动显示 loading 骨架
- `el-button :loading`: 提交按钮显示转圈，防重复点击
- 空状态组件 (`EmptyState.vue`): 无数据时显示友好空提示，非空白页

**Chunk 加载容错** (`router/index.js:177-186`):
- `router.onError`: 检测 "Failed to fetch dynamically imported module"
- 提示用户刷新浏览器重试

### Q29: 日志策略怎么设计的？

**后端日志** (SLF4J + Logback):

| 级别 | 场景 | 示例位置 |
|---|:---|---|

**前端日志** (`utils/logger.js`):
- `logger('ComponentName')` 创建带模块前缀的日志函数

- 生产环境: 通过 Vite `build` 的 minify 自动移除 console.log (保留 console.error)

## S10 失败场景演练

### 场景 1: 数据库宕机

**故障**: MySQL 服务崩溃或网络不可达。

**系统行为**:
1. 后端 `DataSourceConfig` 连接池 (HikariCP) 检测到连接丢失 -> `SQLException`
2. `GlobalExceptionHandler` 未直接捕获 (HikariCP 内部重试 3 次)
3. 最终抛出 `DataAccessException` -> 前端收到 `Result.error(500, "服务器内部错误")`

**恢复**: MySQL 重新启动 -> docker-compose healthcheck 通过 -> backend 健康检查恢复 -> curl /api/v1/health 返回 200

**改进建议** (生产环境):
- HikariCP `connectionTimeout: 30000`
- Circuit Breaker (Resilience4j) 快速失败
- 前端显示 "服务暂不可用，请稍后重试"

### 场景 2: JWT Token 过期

**故障**: 用户 token 已过期 (7天后) 但仍在使用系统。

**系统行为 (双重防护)**:
1. **前端预检**: `router.beforeEach` -> `isTokenExpired(token)` -> 过期 -> 清 localStorage + 跳 /login
2. **后端拦截**: 即使前端未拦截，`LoginInterceptor` -> `parseTokenPayload()` 验签失败 -> 返回 401
3. **axios 拦截器**: `code === 401` -> 清 token + 跳 /login + `ElMessage.error('未登录')`

**用户体验**: 无缝跳转到登录页，黄色警告 "登录已过期，请重新登录"

**代码位置**:
- 前端预检: `router/index.js:126-128`
- 后端拦截: `interceptor/LoginInterceptor.java:67-71`
- axios 拦截器: `api/request.js:88-101`

### 场景 3: 并发转账死锁 (已预防)

**故障**: 用户 A 从账户 1 转账到账户 2，同时用户 B 从账户 2 转账到账户 1。

**旧代码行为**: 死锁 -> InnoDB 死锁检测器 (innodb_lock_wait_timeout=50s) -> 回滚一个事务 -> 用户看到错误

**新代码行为**: 按账户 ID 升序加锁 -> `Math.min(1,2)`=1 先锁, `Math.max(1,2)`=2 后锁 -> 两个线程加锁顺序均为 1->2 -> 无循环等待 -> 零死锁

**验证方法**: `CrossModuleIntegrationTest` 中模拟两个线程并发转账

### 场景 4: CSV 格式错误

**故障**: 用户上传格式错误的 CSV 文件 (缺列、金额非数字、日期格式错、分类ID不存在)。

**系统行为 (逐行容错)**:
1. 首行 (表头) 自动跳过
2. 列数不足 -> failRow: "列数不足，至少需要4列"
3. 分类ID不存在 -> failRow: "分类ID X 不存在"
4. 类型非法 -> failRow: "交易类型必须为1(收入)或2(支出)"
5. 金额 <=0 -> failRow: "金额必须大于0"
6. 日期格式错 -> failRow: "日期格式错误 (需 YYYY-MM-DD HH:mm:ss)"
7. 超 1000 条 -> failRow: "单次导入不能超过 1000 条"
8. 成功行正常插入，失败行不影响成功行

**返回**: `ImportResultDTO { successCount: 80, failCount: 20, failRows: [{row: 5, reason: "..."}, ...] }`

**前端**: `ImportPage.vue` 显示导入结果面板，成功数绿色，失败行红色列表展示

### 场景 5: 内存溢出 (OOM) 预防

**故障风险点**:
1. 收支列表查询: 无分页限制 -> 可能查询全表 -> 大内存占用
2. CSV 导入: 超大文件 -> 内存爆炸
3. 登录限流: 攻击者大量虚构用户名 -> ConcurrentHashMap 无限增长

**防护措施**:
1. **pageSize 上限**: `Math.min(pageSize, 100)` -> 最多一次查 100 条 (`TransactionServiceImpl.java:129`)
2. **文件大小上限**: 5MB (`TransactionServiceImpl.java:435`)
3. **导入条数上限**: 1000 条 (`TransactionServiceImpl.java:515`)
4. **限流器 TTL 过期**: 10 分钟无活动的限流器自动移除 (`LoginRateLimiter.java:30`)
5. **限流器节流清理**: 采样触发，size>=100 或距上次清理超 60s (`LoginRateLimiter.java:94-110`)

### 12.5 高频追加提问应对

**Q: 项目中最困难的部分是什么？**
> "最困难的是转账死锁问题的发现和修复。旧代码按 fromAccount -> toAccount 顺序加锁，双向并发转账会形成循环等待。我通过分析操作系统课本中的死锁四大必要条件，确认了问题是'循环等待'，然后用按账户 ID 升序加锁的策略破坏了这个条件。"

**Q: 如果让你重新设计，会做什么不同？**
> "我会引入 Redis 做分布式缓存和分布式锁，替代内存缓存和数据库悲观锁。另外会在前端加 E2E 测试 (Playwright/Cypress)，目前只有后端单元测试。"

**Q: 系统能支撑多少用户？**
> "当前是单实例 SpringBoot + 单 MySQL，预估支撑 500-1000 并发用户。如需扩展，可以做 MySQL 主从读写分离、Nginx 反向代理多后端实例、Redis 缓存热点数据。"

**Q: 怎么保证财务数据的准确性？**
> "三点：第一，金额全系统用 DECIMAL(12,2)，禁用 FLOAT/DOUBLE 防精度丢失；第二，转账用 @Transactional + SELECT FOR UPDATE 防并发透支；第三，转账记录禁止单独删除/修改金额，保证配对完整性。"

> - `/loop --max-loops N`：限制最大循环次数后自动退出

## 第二步：健康观测（场景 2）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

若请求 `--health` 或每轮开始前自动执行：

运行健康检查任务（独立于 Worker/Verifier）：

- **编译健康**：`cd system/backend && mvn clean compile`（后端）& `cd system/frontend && pnpm build`（前端）
- **测试健康**：运行关键单元测试和集成测试，统计通过率、flaky 测试数
- **API 健康**：调用 `http://localhost:8080/actuator/health` 或核心业务接口（登录、账户列表、记账），记录响应码和延迟
- **数据库健康**：连接 MySQL 执行 `SELECT 1`，检查连接池状态和迁移版本
- **运行时健康**：内存使用（<4GB）、进程数、磁盘空间、孤儿工作树
- **CI 健康**：若配置 CI 流水线，检查最近运行状态（可选）

生成健康报告并保存到 `.claude/state/health-bus.json`，同时打印表格摘要。

**若任何关键项失败**（编译失败、核心 API 5xx、数据库不可达），自动触发 **自愈流程**（见第五步）。

## 第三步：递归审查修复循环（场景 1）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

当执行 `--auto` 且没有 `--health` 或 `--commit` 时，进入以下循环：

### 3.1 任务选择
从任务队列（`.claude/state/task-queue.json`）中选取最高价值任务。若无任务且未收敛，则根据健康债务生成默认修复任务（如“修复最老的 regression”）。

### 3.2 Worker 执行（原子事务）
- 创建全新 sandbox：`git worktree add ../sandbox-<timestamp> main`（或使用临时目录）
- Worker 在 sandbox 中生成补丁（仅允许修改与任务相关的文件）
- 补丁大小受阶段限制（`worker.yaml` 中的 `max_diff_files` 和 `max_diff_lines`）
- 执行最小验证（单元测试 + 静态检查），通过后返回补丁信息

### 3.3 递归审查循环（核心）
补丁进入审查循环，最多迭代 `max_review_iterations`（默认 5 次）：

**每次迭代执行以下步骤：**

1. **Verifier 独立审查**（全新进程，不信任 Worker）：
   - 在另一干净 worktree 中应用补丁
   - 运行完整验证流水线：静态 → 单元 → 集成 → 语义（业务规则） → 回归（与历史基线对比） → 风格
   - 生成审查报告：`.claude/state/review-<timestamp>.json`，包含 `passed`, `issues[]`, `fix_suggestion`

2. **判断**：若 `passed == true`，跳出循环；否则继续。

3. **自动修复**：
   - 从审查报告中提取 `issues` 作为新任务输入
   - 启动修复 Worker（同一沙箱）生成增量补丁
   - 修复完成后回到步骤 1（迭代计数器 +1）

4. **熔断机制**（从 `recursive-guard.yaml` 加载）：
   - 同一问题连续失败 3 次 → 标记 `BLOCKED`，生成升级包，**不再继续修**
   - 同一文件修改超 5 次 → 冻结该模块，跳过任务
   - 基准性能波动 >5% → 强制回滚，增加验证深度
   - 达到最大迭代次数 → 退出循环，进入人工升级流程

### 3.4 多环境共识与确定性重放
审查通过后，补丁进入：
- **多环境共识**：在本地、Docker、干净安装环境重放关键验证（`environment-consensus.yaml`）
- **确定性重放**：回放历史 API 会话、用户工作流、回归场景（`replay.yaml`）

### 3.5 更新基线与状态
所有验证通过后：
- 将补丁合并到主分支（`git merge --ff-only`）
- 更新稳定基线快照（`.claude/state/stable-baseline.json`）
- 更新热力图、回归图谱（`.claude/state/heatmap.json`）
- 减少 `objective_distance`

### 3.6 规范提交（触发场景 3）
合并后自动执行第四步（规范提交）。

## 第四步：规范提交（场景 3）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

**每次合并后**（或手动执行 `/loop --commit "message"`），自动生成标准 commit message。

### 4.1 自动检测提交信息
- `type`：从改动内容自动推断（`feat`/`fix`/`refactor`/`docs`/`test`/`chore`/`perf`）
- `scope`：从改动的文件路径自动推断（`auth`/`account`/`transaction`/`dashboard`/`budget`/`recurring`/`api`/`db`/`build`/`ci`）
- `subject`：从任务描述或手动输入提取（≤50 字符）

### 4.2 强制校验
从 `git-governance.yaml` 加载规则：
- ✅ **必须**包含 `Author: qxw` 和 `Author-ID: 2501060122`
- ✅ **必须**包含验证结果（单元测试、语义审查、重放验证状态）
- ✅ **必须**包含审查各阶段状态（static/semantic/regression/style）
- ✅ **必须**包含健康各维度状态（build/api/database）
- ✅ **必须**包含补丁风险评估（`risk: low|medium|high`，`review-iterations`）
- ❌ **禁止**出现 `temp`, `wip`, `fix again`, `try fix`, `final final` 等无效提交信息

若缺少任何必要字段，**拒绝 commit**，提示补全并暂停循环。

### 4.3 生成与提交
调用 `/git-committer` 提交并 push。若提交失败（网络、冲突），自动重试最多 3 次，仍失败则生成 `state/escalation-packet.md` 并暂停。

## 第五步：失败分类与自愈
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

**失败分类器**（`failure-classifier.yaml`）：
- `compile` → 立即修复，重试上限 3 次
- `unit_test` → 标准修复，重试上限 3 次
- `integration` → 深度修复，重试上限 2 次，达到后 escalate
- `semantic` → 高风险修复，重试上限 2 次，自动回滚
- `replay` → 强制回滚，重试上限 2 次
- `flaky` → 仅重跑（不计入失败），最多 3 次
- `architecture` → 直接 `BLOCKED`，人工介入

**自愈流程**：
1. 健康观测失败 → 分类失败类型
2. 从 `recovery-matrix.yaml` 查找对应修复动作序列
3. 自动创建修复任务，进入递归审查闭环
4. 修复完成后重新健康观测，若仍失败则 escalate

## 第六步：收敛检测与自动停机
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

每轮结束后检查收敛条件（`convergence.yaml`）：
- 连续 10 轮无新增检查点
- 零 regression，健康度全绿
- `objective_distance < 0.02`
- 所有核心模块已冻结（热力图显示稳定）
- 最近 10 轮审查迭代次数均 ≤ 2
- 无 `BLOCKED` 任务

若满足，则：
- 拒绝所有代码改动
- 生成最终 release 包（`RELEASE.md`, `CHANGELOG.md`）
- 输出终止证明并停机

```yaml
max_review_iterations: 5
```

### 3.2 `git-governance.yaml`
```yaml
author: "qxw"
author_id: "2501060122"
types: ["feat", "fix", "docs", "refactor", "test", "chore", "perf"]

require_validation_results: true
reject_missing_fields: true
forbidden_messages: ["temp", "wip", "fix again", "try fix", "final final", "test commit"]
auto_detect_scope: true
auto_detect_type: true
```

### 3.3 `failure-classifier.yaml`
```yaml
failure_classes:
  compile: { strategy: "immediate_fix", max_retries: 3, auto_escalate: false }
  unit_test: { strategy: "standard_fix", max_retries: 3, auto_escalate: false }
  integration: { strategy: "deep_fix", max_retries: 2, auto_escalate: true }

  replay: { strategy: "force_rollback", max_retries: 2, auto_escalate: true }
  flaky: { strategy: "retry_only", max_retries: 3, auto_escalate: false }
  architecture: { strategy: "blocked", max_retries: 0, auto_escalate: true }
```

```yaml
    - action: "rollback_last_patch"

    - action: "recheck_api_spec"
    - action: "check_database_connection"
```

## 六、常见问题与调试

### Q1: `/loop` 命令不显示或报错 `command not found`
- 确认文件路径：`.claude/commands/loop.md` 必须位于项目根目录下
- 重启 Claude Code：输入 `/exit` 退出，重新运行 `claude`

### Q2: 健康检查中 API 测试失败
- 确保后端服务已启动（`mvn spring-boot:run`）且监听 8080 端口
- 检查 `application.yml` 中的端口配置与健康检查端点一致
- 若未启动服务，可在健康检查脚本中自动启动（建议改进）

### Q3: 递归审查循环卡在某个迭代不退出
- 检查 `recursive-guard.yaml` 中的 `max_review_iterations` 是否太小（默认 5 足够）
- 查看 `.claude/state/review-*.json` 中的 `issues`，分析为何 Verifier 始终不通过
- 若为 flaky 测试，可调整 `failure-classifier.yaml` 将 `flaky` 策略改为 `retry_only` 并增加重试次数

### Q4: 提交信息缺少作者字段
- 确认 `git-governance.yaml` 中 `author` 和 `author_id` 配置正确
- 检查 loop.md 第四步中是否强制插入了这些字段（已在模板中强制）

### Q5: 长时间运行后系统变得很慢
- 检查 `.claude/state/micro-snapshots/` 目录大小，定期清理旧快照
- 调整 `project-health.yaml` 中的 `runtime.memory_limit_mb` 和 `max_worktrees`
- 设置 Windows 任务计划每周重启一次 Claude session（通过 `--continue false`）

## 七、最终验证：三个场景的全流程演示

### 场景 1：自动优化 → 审查 → 修复 → 再审查 → 通过
1. 故意在 `TransactionService.java` 中引入一个语义错误（例如将 `type.equals("transfer")` 改为 `type.equals("expense")`）
2. 运行 `/loop --auto`
3. 系统会：
   - Worker 生成补丁
   - Verifier 检测到语义错误（transfer 被计入支出）
   - 自动生成修复任务
   - 修复 Worker 改正错误
   - 再次审查通过
   - 提交 commit（含详细审查记录）
4. 最终您会看到类似输出：
   ```
   Review iteration 1: FAILED (semantic violation)
   Auto-fix applied...
   Review iteration 2: PASSED
   ```

### 场景 2：健康观测
运行 `/loop --health`，输出表格应包含：
- Build: ✅ PASS
- Tests: 98% pass (2 flaky)
- API: ✅ 200 OK (p95=142ms)
- Database: ✅ connected
- Runtime: memory 2.1GB/4GB
- Overall: HEALTHY

### 场景 3：规范提交
运行 `/loop --commit "fix(auth): correct token expiration"`，生成的 commit message 应包含：
```
fix(auth): correct token expiration

Author: qxw
Author-ID: 2501060122

Validation:
- unit-tests: passed
- semantic-review: passed
- deterministic-replay: passed

...
```
